# v0.4 Learning Guide — Functions, `BL`, `RET`, and `x30`

This guide is beginner-first. It assumes v0.3 stack instructions may still feel difficult, so it explains function calls slowly.

v0.4 adds this ability:

```text
The program can call a function and return to where it came from.
```

The two most important instructions are:

```text
BL  = branch with link, also known as call
RET = return from function
```

The most important register is:

```text
x30 = link register = return address register
```

---

## 1. The basic problem: how does a function return?

Consider this assembly:

```asm
movz x0, #2
bl   add_three
hlt  #0

add_three:
    add x0, x0, #3
    ret
```

We want execution to do this:

```text
1. set x0 = 2
2. jump to add_three
3. add 3 to x0
4. return back to the instruction after BL
5. execute HLT
```

The key question is:

```text
When add_three executes RET, how does it know where to return?
```

Answer:

```text
BL saves the return address in x30.
RET jumps to the address stored in x30.
```

---

## 2. What `BL` does

`BL` means “branch with link.”

Assembly:

```asm
bl add_three
```

C-like pseudocode:

```c
x30 = pc + 4;        // remember where to return
pc = add_three;      // jump to function
```

Why `pc + 4`?

Because ARM64 instructions are 4 bytes. The instruction after the current one is at:

```text
current pc + 4
```

So `BL` stores the address of the next instruction in `x30`.

---

## 3. What `RET` does

Assembly:

```asm
ret
```

C-like pseudocode:

```c
pc = x30;
```

That is it.

`RET` does not magically know about function names. It simply jumps to the address stored in `x30`.

---

## 4. Slow walkthrough: simple function call

Assembly:

```asm
movz x0, #2
bl   add_three
hlt  #0

add_three:
    add x0, x0, #3
    ret
```

Address table:

```text
Address   Instruction
-------   ----------------
0x1000    movz x0, #2
0x1004    bl add_three
0x1008    hlt #0
0x100c    add x0, x0, #3
0x1010    ret
```

Step table:

```text
Step   pc       Instruction        x0   x30      What happens
----   ------   ----------------   --   ------   ---------------------------------
1      0x1000   movz x0,#2         2    0        x0 = 2; pc = 0x1004
2      0x1004   bl add_three       2    0x1008   x30 = 0x1008; pc = 0x100c
3      0x100c   add x0,x0,#3       5    0x1008   x0 = 5; pc = 0x1010
4      0x1010   ret                5    0x1008   pc = x30 = 0x1008
5      0x1008   hlt #0             5    0x1008   stop
```

Final result:

```text
x0 = 5
```

The most important line is step 2:

```text
BL saves 0x1008 in x30 before jumping.
```

Then step 4 uses that saved address:

```text
RET sets pc = x30.
```

---

## 5. `RET Xn`: return through a custom register

Normal `ret` means:

```asm
ret x30
```

ARM64 can also return through another register:

```asm
ret x0
```

C-like pseudocode:

```c
pc = x0;
```

Our emulator supports this for v0.4, but most normal function examples use plain `ret` / `ret x30`.

---

## 6. Why nested calls are harder

A nested call means one function calls another function.

Example:

```asm
main:
    bl outer
    hlt #0

outer:
    bl inner
    ret

inner:
    ret
```

This looks simple, but it has a problem.

### What goes wrong?

When `main` calls `outer`:

```text
BL outer stores main's return address in x30
```

Then when `outer` calls `inner`:

```text
BL inner stores outer's return address in x30
```

That second `BL` overwrites the old `x30`.

The original return address back to `main` is lost.

---

## 7. Slow walkthrough: broken nested call

Address table:

```text
Address   Instruction
-------   ----------------
0x1000    bl outer
0x1004    hlt #0
0x1008    bl inner
0x100c    ret
0x1010    ret
```

Trace:

```text
Step   pc       Instruction   x30 after   Meaning
----   ------   -----------   ---------   -----------------------------------
1      0x1000   bl outer      0x1004      main return address saved
2      0x1008   bl inner      0x100c      x30 overwritten with outer return
3      0x1010   ret           0x100c      returns to outer's ret
4      0x100c   ret           0x100c      returns to itself forever
```

The program gets stuck because `x30` no longer contains `0x1004`.

This is why nested calls must save `x30` somewhere safe.

Where can we save it?

```text
On the stack.
```

---

## 8. Saving `x30` with the stack

The common pattern is:

```asm
stp x29, x30, [sp, #-16]!
...
ldp x29, x30, [sp], #16
ret
```

From v0.3:

```asm
stp x29, x30, [sp, #-16]!
```

means:

```c
sp = sp - 16;
memory[sp] = x29;
memory[sp + 8] = x30;
```

And:

```asm
ldp x29, x30, [sp], #16
```

means:

```c
x29 = memory[sp];
x30 = memory[sp + 8];
sp = sp + 16;
```

So a function can save the caller's `x30`, call another function, then restore the old `x30` before returning.

---

## 9. Correct nested call pattern

Assembly:

```asm
movz x0, #0
bl   outer
hlt  #0

outer:
    stp x29, x30, [sp, #-16]!
    add x0, x0, #1
    bl  inner
    add x0, x0, #1
    ldp x29, x30, [sp], #16
    ret

inner:
    add x0, x0, #1
    ret
```

C-like pseudocode:

```c
x0 = 0;
outer();
halt();

void outer() {
    save(x29, x30);
    x0 = x0 + 1;
    inner();
    x0 = x0 + 1;
    restore(x29, x30);
    return;
}

void inner() {
    x0 = x0 + 1;
    return;
}
```

Final result:

```text
x0 = 3
```

---

## 10. Slow stack table for nested call

Assume before `outer` starts:

```text
sp  = 0x100000
x30 = 0x1008   // return to main's hlt
```

When `outer` executes:

```asm
stp x29, x30, [sp, #-16]!
```

Stack becomes:

```text
sp = 0x0ffff0

Address      Stored value
---------    --------------------------
0x0ffff0     old x29
0x0ffff8     old x30 = 0x1008
```

Then `outer` calls `inner`:

```asm
bl inner
```

This overwrites `x30`:

```text
x30 = address after bl inner
```

That is okay because the old `x30` is safe in memory at `0x0ffff8`.

Before `outer` returns, it executes:

```asm
ldp x29, x30, [sp], #16
```

This restores:

```text
x29 = old x29
x30 = 0x1008
sp  = 0x100000
```

Now `ret` can correctly jump back to main's `hlt`.

---

## 11. Sequential calls are easier than nested calls

Sequential calls:

```asm
movz x0, #0
bl inc
bl inc
bl inc
hlt #0

inc:
    add x0, x0, #1
    ret
```

Each `BL` happens after the previous function already returned.

So `x30` can be overwritten each time safely.

Final result:

```text
x0 = 3
```

Nested calls are harder because a function calls another function **before** it has returned.

---

## 12. Invalid returns

`RET` sets `pc` to whatever is in the return register.

If `x30` contains a bad address, the emulator should fail safely.

Bad examples:

```text
x30 = 0x1       // not 4-byte aligned
x30 = 0x100000  // outside valid instruction-fetch area
x30 = 0         // below loaded program area in this project policy
```

The emulator validates return targets so it does not continue from nonsense addresses.

---

## 13. How v0.4 fits into emulator code

Simplified `BL` execution:

```c
target = pc + offset;
validate_branch_target(target);
x30 = pc + 4;
pc = target;
```

Important atomicity idea:

```text
If the BL target is invalid, do not change pc and do not change x30.
```

Simplified `RET` execution:

```c
target = read_register(rn);  // usually x30
validate_return_target(target);
pc = target;
```

## 14. Common mistakes

### Mistake 1: thinking `RET` knows the function name

It does not.

`RET` only knows a register value.

```asm
ret
```

means:

```c
pc = x30;
```

### Mistake 2: forgetting `BL` overwrites `x30`

Every `BL` writes a new return address into `x30`.

That is fine for simple calls, but dangerous for nested calls.

### Mistake 3: thinking `x29` is required by the emulator

The emulator does not require `x29`.

`x29` is a convention for frame pointer style code.

We save `x29` with `x30` because real ARM64 function prologues commonly do that.

### Mistake 4: thinking stack frames are a new magic feature

They are not magic. They are just `STP`/`LDP` from v0.3 used to save and restore registers.

```asm
add_one:
    add x0, x0, #1

```

Question:

```text
Final x0?
```

Answer:

```text

```

### Exercise 2

At address `0x1004`, this instruction runs:

```asm
bl func
```

Question:

```text
What value goes into x30?
```

Answer:

```text
x30 = 0x1008
```

### Exercise 3

Why does this nested pattern break?

```asm
outer:
    bl inner
    ret
inner:
    ret
```

Answer:

```text
BL inner overwrites x30, so outer loses its return address.
```

## 16. One-sentence summary

v0.4 teaches the emulator function calls: `BL` stores the return address in `x30` and jumps to a function, `RET` jumps back using `x30`, and nested calls must save/restore `x30` on the stack so the original return address is not lost.
