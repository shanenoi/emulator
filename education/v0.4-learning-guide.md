# v0.4 Learning Guide — Functions and Returns

v0.4 adds the smallest useful model of ARM64 function calls.

Before v0.4, the emulator could run straight-line code, loops, branches, and stack memory operations. v0.4 teaches the CPU how to call a helper routine and come back to the caller.

## The new mental model

A function call is just controlled `pc` movement plus one saved return address.

```text
caller instruction
    |
    | BL target
    v
function body
    |
    | RET
    v
instruction after BL
```

The important register is `x30`, also called the link register or `LR`.

## `BL`: branch with link

`BL <label>` means:

```text
x30 = pc + 4
pc = target label address
```

The `pc + 4` value is the address of the instruction immediately after the call. That is where the function should return.

Example:

```asm
movz x0, #2
bl add_three
hlt #0

add_three:
    add x0, x0, #3
    ret
```

If the `BL` is at `0x1004`, then `x30` becomes `0x1008`, because `0x1008` is the instruction after the call.

## `RET`: return from a function

`RET` means:

```text
pc = x30
```

`RET Xn` means:

```text
pc = xn
```

The default form uses `x30`:

```asm
ret
```

The explicit form is also supported:

```asm
ret x30
ret x0
```

A return target must be aligned and inside emulator memory. This emulator also rejects return targets below the raw-binary load address, so `ret xzr` fails cleanly instead of jumping to address zero.

## Step-by-step simple call

Program:

```asm
movz x0, #2
bl add_three
hlt #0

add_three:
    add x0, x0, #3
    ret
```

Execution:

```text
0x1000: movz x0, #2      x0 = 2, pc = 0x1004
0x1004: bl add_three     x30 = 0x1008, pc = add_three
0x100c: add x0, x0, #3   x0 = 5, pc = 0x1010
0x1010: ret              pc = x30 = 0x1008
0x1008: hlt              stop
```

Final result:

```text
x0 = 5
x30 = 0x1008
pc = 0x1008
```

## Why nested calls need stack saves

A common beginner trap is this:

```asm
first:
    bl second
    ret
```

When `first` was called, `x30` pointed back to the original caller. But when `first` calls `second`, the new `BL` overwrites `x30`.

So after `second` returns, `first` no longer knows how to return to its caller.

The fix is to save `x30` before making another call and restore it before returning.

```asm
first:
    stp x29, x30, [sp, #-16]!
    bl second
    ldp x29, x30, [sp], #16
    ret
```

This uses the v0.3 stack behavior:

```text
STP saves x29 and x30 on the stack.
LDP restores x29 and x30 from the stack.
```

## What v0.4 intentionally does not do

v0.4 does not implement a full calling convention. It does not enforce which registers are caller-saved or callee-saved. It does not know symbolic function names. It does not do stack unwinding.

It only implements the CPU mechanics needed for hand-written examples:

```text
BL writes x30 and jumps.
RET jumps through x30 or another register.
Stack examples can save/restore x30 manually.
```

## Trace mode

Trace mode is unchanged from v0.2:

```sh
./emulator trace examples/v0_4/simple_call.bin
```

It prints each executed `pc`. That lets you see execution jump into a function and then return to the caller.

## v0.4 examples

```text
examples/v0_4/simple_call.s
examples/v0_4/sequential_calls.s
examples/v0_4/ret_x30.s
examples/v0_4/ret_custom_register.s
examples/v0_4/nested_calls.s
examples/v0_4/call_with_stack_frame.s
examples/v0_4/invalid_return.s
examples/v0_4/unsaved_nested_call.s
```

The `unsaved_nested_call.s` example is intentionally wrong. It demonstrates why a function that makes another call must preserve `x30` if it still needs to return to its own caller.

The `sequential_calls.s` example calls the same `inc` function three times in a row. Each `BL` overwrites `x30` with that call's own return address, but this is safe because each call returns before the next one starts.
