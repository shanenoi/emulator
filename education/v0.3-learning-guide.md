# v0.3 Learning Guide — Memory and Stack

v0.3 extends the emulator from “register-only programs” into programs that can move data between registers and memory.

Earlier versions could do this:

```asm
movz x0, #42
add  x1, x0, #1
hlt  #0
```

That only changes CPU registers. v0.3 can now do this:

```asm
movz x0, #42
str  x0, [sp, #-8]!
ldr  x1, [sp], #8
hlt  #0
```

That stores a value into memory, loads it back, and restores the stack pointer.

## 1. The mental model

The emulator still has the same main pieces:

```text
+------------------------------+
|          Emulator            |
|                              |
|  +--------+    +----------+  |
|  |  CPU   | -> |  Memory  |  |
|  +--------+    +----------+  |
|                              |
| registers       bytes        |
| pc              code/data    |
| sp              stack area   |
+------------------------------+
```

v0.1 and v0.2 mostly changed registers and `pc`.

v0.3 adds instructions that calculate an address and read or write bytes at that address.

## 2. What is an effective address?

A memory instruction usually has a base register and an offset:

```asm
ldr x0, [x1, #16]
```

This means:

```text
effective_address = x1 + 16
x0 = memory_read64(effective_address)
```

The emulator calls this calculated memory address the effective address.

## 3. `SP` as a base register

The stack pointer starts at the top of memory:

```text
sp = 0x100000
```

The highest valid byte address is:

```text
0x0fffff
```

So reading 8 bytes from `[sp]` fails, because address `0x100000` is already just past the end of memory.

To push one 64-bit value onto the stack, we first move `sp` down:

```asm
str x0, [sp, #-8]!
```

This means:

```text
sp = sp - 8
memory_write64(sp, x0)
```

After that, the value is stored at:

```text
0x0ffff8
```

## 4. Pre-index addressing

Pre-index means “change the base first, then access memory.”

Example:

```asm
str x0, [sp, #-8]!
```

Step by step:

```text
old sp = 0x100000
new sp = old sp - 8 = 0x0ffff8
store x0 at 0x0ffff8
```

The `!` tells ARM64 to write the updated address back into the base register.

## 5. Post-index addressing

Post-index means “access memory first, then change the base.”

Example:

```asm
ldr x1, [sp], #8
```

Step by step:

```text
load x1 from current sp
sp = sp + 8
```

This is useful for popping a value from the stack.

## 6. Push and pop example

Program:

```asm
movz x0, #42
str  x0, [sp, #-8]!
ldr  x1, [sp], #8
hlt  #0
```

Initial state:

```text
sp = 0x100000
x0 = 0
x1 = 0
```

After `movz x0, #42`:

```text
x0 = 42
```

After `str x0, [sp, #-8]!`:

```text
sp = 0x0ffff8
memory[0x0ffff8..0x0fffff] = 42 in little-endian bytes
```

After `ldr x1, [sp], #8`:

```text
x1 = 42
sp = 0x100000
```

The final stack pointer is restored.

## 7. Little-endian memory

A 64-bit value is stored as 8 bytes. This emulator uses little-endian order.

For value `42`:

```text
hex value: 0x000000000000002a
bytes:     2a 00 00 00 00 00 00 00
```

You can see this with the dump command:

```sh
./emulator dump examples/v0_3/memory_store_load.bin 0xffff8 8
```

Expected memory dump line:

```text
0x00000000000ffff8: 2a 00 00 00 00 00 00 00
```

## 8. 32-bit `W` loads and stores

ARM64 has 64-bit `X` registers and 32-bit `W` views of those registers.

```asm
str w0, [sp, #-4]!
ldr w1, [sp], #4
```

`STR Wt` writes only 4 bytes.

`LDR Wt` reads 4 bytes and zero-extends the value into the full 64-bit `X` register.

So if memory contains:

```text
ff ff ff ff
```

then:

```asm
ldr w0, [sp]
```

produces:

```text
x0 = 0x00000000ffffffff
```

not:

```text
x0 = 0xffffffffffffffff
```

## 9. `LDUR` and `STUR`

`LDR` / `STR` unsigned-offset forms use scaled offsets.

For example:

```asm
ldr x0, [x1, #16]
```

The encoded immediate is scaled by 8 for a 64-bit load.

`LDUR` / `STUR` use signed byte offsets instead:

```asm
ldur x0, [x1, #-8]
stur x2, [x3, #15]
```

These are useful when the offset is negative or not naturally scaled.

## 10. Pair operations: `STP` and `LDP`

Pair operations store or load two registers at once.

```asm
stp x0, x1, [sp, #-16]!
ldp x2, x3, [sp], #16
```

The store does:

```text
sp = sp - 16
memory[sp]     = x0
memory[sp + 8] = x1
```

The load does:

```text
x2 = memory[sp]
x3 = memory[sp + 8]
sp = sp + 16
```

This is the first step toward understanding real function stack frames, but v0.3 does not implement `BL`, `RET`, or real calling conventions yet.

## 11. Register 31: `SP` vs `XZR`

ARM64 register field `31` can mean different things depending on context.

For arithmetic and normal register reads/writes in this project:

```text
31 means XZR/WZR, the zero register
```

For load/store base registers:

```text
31 means SP, the stack pointer
```

So:

```asm
str x0, [sp, #-8]!
```

uses `sp` as the base address.

But:

```asm
str xzr, [sp, #-8]!
```

stores zero, because the source register is the zero register.

## 12. Failure behavior

v0.3 is careful about failed memory operations.

If a load fails:

```text
- destination register is not changed
- base register write-back is not applied
- emulator returns an error
```

If a store fails:

```text
- memory is not partially changed
- base register write-back is not applied
- emulator returns an error
```

If a pair operation fails:

```text
- neither register/memory half is changed
- write-back is not applied
```

This makes failures easier to test and reason about.

## 13. What v0.3 still does not do

v0.3 does not include:

```text
- BL / RET function calls
- real stack frames
- ELF or Mach-O loading
- syscalls
- debugger REPL
- virtual memory
- byte/halfword loads and stores
- sign-extending loads
- atomic/exclusive memory instructions
```

Those belong to later versions.
