# v0.5 Learning Guide — Debugger REPL

v0.5 adds an interactive debugger to the emulator.

The debugger does not change the CPU model. It gives you a way to pause, inspect, and resume the same fake ARM64 machine that `run` already executes.

## Mental model

Before v0.5, the normal runner did this:

```text
load program
run until HLT / error / instruction limit
print final state
```

The debugger changes that to:

```text
load program
wait for user command
maybe execute one instruction
maybe continue until a breakpoint
maybe print registers or memory
repeat until quit
```

So the debugger wraps the existing emulator core:

```text
Debugger
├── Emulator
│   ├── CPU
│   └── Memory
└── Breakpoints
```

## Starting the debugger

```sh
./emulator debug examples/v0_1/add.bin
```

The program is loaded at `0x1000`, but it does not execute immediately.

You will see a prompt:

```text
emu>
```

## Important commands

```text
help                         show commands
run | r                      reset and run from the start
step | s                     execute exactly one instruction
continue | c                 run until breakpoint, HLT, error, or limit
regs                         print CPU registers
mem | x <address> <length>   dump memory
break | b <address>          set breakpoint
delete <id-or-address>       delete breakpoint
breakpoints                  list breakpoints
trace on                     enable pc trace
trace off                    disable pc trace
quit | q                     leave debugger
```

## Run vs continue vs step

### `step`

`step` executes exactly one instruction at the current `pc`.

If `pc = 0x1000`, then:

```text
emu> step
pc=0x0000000000001004
```

For the v0.1 add example, that first instruction is:

```asm
movz x0, #2
```

After one step:

```text
x0 = 2
pc = 0x1004
```

### `run`

`run` resets the CPU and memory back to the initially loaded program state, then starts execution.

This means:

```text
step
step
run
```

will not continue from the two previous steps. It starts from a clean loaded state.

Breakpoints are preserved across `run`.

### `continue`

`continue` resumes from the current state. It does not reset.

It stops when one of these happens:

```text
- a breakpoint is hit
- HLT executes
- an emulator error happens
- instruction limit is reached
```

## Breakpoints

A breakpoint is an instruction address where the debugger should stop before executing that instruction.

Example:

```text
emu> break 0x1008
breakpoint 1 set at 0x0000000000001008
emu> run
breakpoint hit at 0x0000000000001008
```

At this point, `pc` still equals `0x1008`.

That is important:

```text
breakpoints stop before execution
```

For the v0.1 add example:

```asm
0x1000: movz x0, #2
0x1004: movz x1, #3
0x1008: add  x2, x0, #3
0x100c: hlt  #0
```

If we break at `0x1008`, then `x0` and `x1` are already set, but `x2` has not been computed yet.

Expected state at the breakpoint:

```text
x0 = 2
x1 = 3
x2 = 0
pc = 0x1008
```

Then:

```text
emu> continue
halted
```

The debugger skips the current breakpoint once so it does not stop forever at the same address.

## Register inspection

Use:

```text
emu> regs
```

This prints the same stable register format as the normal runner:

```text
x0  = 0x0000000000000002
x1  = 0x0000000000000003
x2  = 0x0000000000000005
...
sp  = 0x0000000000100000
pc  = 0x000000000000100c
nzcv = 0000
instructions = 0x0000000000000004
```

This is useful after `step`, at a breakpoint, or after `run`.

## Memory inspection

Use:

```text
emu> mem 0xffff8 8
```

or the alias:

```text
emu> x 0xffff8 8
```

The output matches the normal CLI `dump` style:

```text
memory dump address=0x00000000000ffff8 length=0x0000000000000008
0x00000000000ffff8: 2a 00 00 00 00 00 00 00
```

This is especially useful for v0.3 stack examples.

## Trace mode

Inside the debugger:

```text
emu> trace on
emu> run
```

Each executed instruction prints its `pc` before execution:

```text
trace pc=0x0000000000001000
trace pc=0x0000000000001004
trace pc=0x0000000000001008
```

Disable it with:

```text
emu> trace off
```

## Scriptable mode

The REPL reads from stdin, so it can be scripted:

```sh
printf 'break 0x1008\nrun\nregs\ncontinue\nquit\n' | ./emulator debug examples/v0_1/add.bin
```

This is important because the future v0.5 automated tests can drive the debugger without a human typing.

## Error behavior

Bad commands should not crash the process.

Example:

```text
emu> wat
error: unknown command: wat
emu> regs
...
emu> quit
```

Invalid arguments also keep the REPL alive:

```text
emu> mem
error: usage: mem <address> <length>
```

Runtime errors are reported, but the debugger remains usable until `quit`.

## What v0.5 does not do

v0.5 is intentionally not a full symbolic debugger.

Out of scope:

```text
- source-level debugging
- symbols and labels
- DWARF
- ELF / Mach-O loading
- watchpoints
- reverse execution
- conditional breakpoints
- terminal UI
- disassembly view
```

## Good learning workflow

For a simple program:

```sh
./emulator debug examples/v0_1/add.bin
```

Then:

```text
regs
step
regs
step
regs
break 0x1008
run
regs
continue
quit
```

This teaches the most important debugger idea:

```text
The debugger is just controlled execution of the same fetch/decode/execute loop.
```
