# v0.7 Lesson — Toy Syscalls

v0.7 adds the emulator's first tiny host-provided runtime. Older examples ended with `HLT` and only changed registers or memory. The new runtime lets a raw ARM64 program ask the emulator to perform a small service with `SVC #0`.

This is intentionally **not** Linux compatibility. It is a deterministic teaching ABI that borrows familiar AArch64 Linux syscall numbers where that makes examples easier to understand.

## ABI

- Trap instruction: `svc #0`
- Syscall number: `x8`
- Arguments: `x0` through `x5`
- Return value: `x0`

Implemented syscalls:

| Number | Name | Arguments | Result |
| --- | --- | --- | --- |
| `64` | `write` | `x0 = fd`, `x1 = guest buffer address`, `x2 = byte length` | byte count in `x0`, or a negative fake errno |
| `93` | `exit` | `x0 = status` | stops execution; CLI returns low 8 bits |

Supported write file descriptors:

- `1`: stdout
- `2`: stderr

Unsupported write fds return fake `-EBADF` in `x0`. Unknown syscall numbers return fake `-ENOSYS` in `x0`. Invalid guest write buffers are emulator runtime errors because they indicate a bad emulated memory access rather than a normal syscall-level failure.

## Hello example

The `examples/v0_7/hello.s` program writes a message and then exits:

```asm
movz x0, #1      // stdout
movz x1, #0x1020 // message address in the raw image
movz x2, #13     // message length
movz x8, #64     // write
svc  #0
movz x0, #0      // exit status
movz x8, #93     // exit
svc  #0
```

Because the project still loads a raw `.text` image at `0x1000`, these examples use fixed addresses for embedded bytes. That keeps v0.7 focused on syscall behavior before v0.8 introduces richer program loading.

## Why this matters

A fake syscall boundary is the first step toward standalone programs. It gives tests and examples an observable behavior beyond final register dumps while keeping the emulator small and predictable.
