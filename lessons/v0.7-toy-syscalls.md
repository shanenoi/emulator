# v0.7 Lesson — SVC, Write, Exit, and Toy Syscalls

This guide is beginner-first. It explains how guest programs can ask the emulator for a few tiny outside-world services.

In v0.6, you learned to read instructions clearly in trace output.

v0.7 adds one new ability:

```text
Guest programs can print bytes and return an exit status.
```

Before v0.7, guest programs could compute values, use memory, call functions, and stop with `HLT`.

After v0.7, guest programs can also execute `SVC #0` and ask the emulator to:

```text
write bytes to stdout
write bytes to stderr
exit with a status code
```

This makes examples feel more like small standalone programs, while still keeping the emulator simple.

---

## 1. Why we need toy syscalls

Older examples were useful, but their visible behavior was mostly final register dumps.

For example, a program could compute:

```text
x0 = 5
```

Then stop with `HLT`.

That is good for learning CPU instructions, but real programs usually do visible things:

```text
print a message
return an exit status
report an error
```

v0.7 adds a small bridge between guest code and the host program running the emulator.

That bridge is called a **toy syscall ABI**.

Do not worry if that phrase sounds formal. This lesson builds it from a concrete hello-world example.

---

## 2. Start with hello: the visible result

Run the main v0.7 example:

```sh
make examples/v0_7/hello.bin
./emulator run examples/v0_7/hello.bin
```

It prints:

```text
hello, v0.7!
```

That is new.

The guest program is no longer only leaving values in registers. It is asking the emulator to write bytes to your terminal.

---

## 3. The hello assembly

The important part of the example is:

```asm
movz x0, #1              // fd = stdout
movz x1, #0x1020         // guest memory address of message
movz x2, #13             // number of bytes to write
movz x8, #64             // syscall number = write
svc  #0
```

Read this as:

```text
Set up registers.
Execute SVC #0.
Let the emulator perform the requested service.
```

Before `SVC #0`, the registers form a little request:

| Register | Value | Meaning |
|---|---:|---|
| `x8` | `64` | choose `write` |
| `x0` | `1` | fd 1 = stdout |
| `x1` | `0x1020` | guest address where message bytes start |
| `x2` | `13` | number of bytes to copy |

Then `SVC #0` means:

```text
Emulator, please use these registers to perform one toy syscall.
```

---

## 4. Memory diagram for hello

The emulator loads the raw `.bin` at address `0x1000`.

The hello example stores message bytes after the instructions.

Conceptually:

```text
Guest memory

0x1000: instruction: movz x0, #1
0x1004: instruction: movz x1, #0x1020
0x1008: instruction: movz x2, #13
0x100c: instruction: movz x8, #64
0x1010: instruction: svc #0
0x1014: instruction: movz x0, #0
0x1018: instruction: movz x8, #93
0x101c: instruction: svc #0
0x1020: byte 'h'
0x1021: byte 'e'
0x1022: byte 'l'
0x1023: byte 'l'
0x1024: byte 'o'
...
```

So this register pair:

```text
x1 = 0x1020
x2 = 13
```

means:

```text
copy 13 bytes starting at guest memory address 0x1020
```

It does **not** mean:

```text
print the number 0x1020
```

`x1` is an address.

`x2` is a length.

Use the v0.3 mental model:

```text
guest memory is a big byte array
registers can hold addresses into that byte array
```

## 5. What `SVC #0` means

`SVC` stands for **supervisor call**.

In a real operating system, an instruction like `svc` can trap from user code into kernel code.

In this teaching emulator, there is no real kernel.

So v0.7 defines a small fake meaning:

```text
SVC #0 = ask the emulator to handle one toy syscall
```

The immediate value matters:

```text
SVC #0     supported
SVC #1     rejected by this emulator
SVC #123   rejected by this emulator
```

Why only `SVC #0`?

Because v0.7 keeps the rule simple:

```text
The instruction says "do a syscall".
The registers say which syscall and which arguments.
```

In source and trace output you may see lowercase `svc #0` or uppercase `SVC #0`. They mean the same instruction; this lesson uses uppercase when talking about the instruction name.

---

## 6. What an ABI is

**ABI** means **Application Binary Interface**.

A formal definition:

```text
An ABI is an agreement about how machine-code programs communicate at binary/runtime boundaries.
```

For this lesson, use the simpler version:

```text
The ABI is the rulebook shared by the guest program and the emulator.
```

The v0.7 rulebook says:

| Register | Meaning |
|---|---|
| `x8` | syscall number |
| `x0` | argument 1, and later the return value |
| `x1` | argument 2 |
| `x2` | argument 3 |
| `x3`–`x5` | reserved for future syscall arguments |

So before `SVC #0`, the guest fills registers like a form.

After `SVC #0`, the emulator writes the result into `x0`.

---

## 7. The two syscalls in v0.7

v0.7 implements exactly two toy syscalls:

| Number in `x8` | Name | Purpose |
|---:|---|---|
| `64` | `write` | copy bytes from guest memory to stdout/stderr |
| `93` | `exit` | stop the guest program and return a status code |

Short form:

```text
write = 64
exit = 93
```

These numbers intentionally resemble Linux AArch64 syscall numbers, but this project is **not** trying to be Linux-compatible yet.

The toy rule is:

```text
x8 = 64  means fake write
x8 = 93  means fake exit
anything else returns fake -ENOSYS in x0
```

---

## 8. What `fd` means here

`fd` means **file descriptor**.

In real Unix-like systems, file descriptors are small numbers used as handles for open files or streams.

In this toy emulator, v0.7 supports only two output descriptors:

| fd | Meaning |
|---:|---|
| `1` | stdout |
| `2` | stderr |

So:

```text
write(1, address, length)  -> write bytes to stdout
write(2, address, length)  -> write bytes to stderr
```

Any other fd is unsupported:

```text
write(9, address, length) -> return -EBADF in x0
```

`-EBADF` means:

```text
bad file descriptor
```

The emulator did not crash. The guest simply asked for a write target this toy runtime does not support.

---

## 9. Slow walkthrough: what happens during `write`

The fake write syscall follows this flow:

```text
1. Guest executes SVC #0.
2. Emulator checks that the SVC immediate is 0.
3. Emulator reads x8.
4. If x8 is 64, this is write.
5. Emulator reads:
       x0 = fd
       x1 = guest buffer address
       x2 = length
6. Emulator checks whether the guest memory range is valid.
7. Emulator copies bytes from guest memory to host stdout/stderr.
8. Emulator writes the result into x0.
9. Emulator moves pc to the next instruction.
```

Small pseudocode:

```c
if (x8 == 64) {
    fd = x0;
    address = x1;
    length = x2;

    if (fd == 1) stream = stdout;
    else if (fd == 2) stream = stderr;
    else {
        x0 = -EBADF;
        pc += 4;
        return;
    }

    if (!valid_guest_range(address, length)) {
        runtime_error();
    }

    copy guest_memory[address..address+length) to stream;
    x0 = length;
    pc += 4;
}
```

The important result rule:

```text
successful write: x0 = number of bytes written
bad fd:           x0 = -EBADF
bad host stream:  x0 = -EIO
bad guest memory: runtime error
```

## 10. Exiting the guest program

The hello example prints first, then exits:

```asm
movz x0, #0              // exit status
movz x8, #93             // syscall number = exit
svc  #0
```

Before `SVC #0`:

| Register | Value | Meaning |
|---|---:|---|
| `x8` | `93` | choose `exit` |
| `x0` | `0` | exit status |

The fake exit syscall means:

```text
stop guest execution
remember the guest exit status
make the CLI return that status
```

Try the non-zero exit example:

```sh
make examples/v0_7/exit_status.bin
./emulator run examples/v0_7/exit_status.bin
echo $?
```

Expected status:

```text
7
```

The CLI returns only the low 8 bits, matching normal command-line exit-status behavior.

---

## 11. Guest vs host

There are two programs involved:

```text
Host program:
    ./emulator
    written in C
    runs on your real computer

Guest program:
    examples/v0_7/hello.bin
    raw ARM64 machine code
    runs inside fake emulator memory
```

The guest cannot directly call your real operating system.

Instead, the guest executes `SVC #0`.

A picture:

```text
Guest ARM64 code                  Host C emulator
----------------                  ----------------
set x8 = 64        ----SVC #0---> read x8
set x0 = 1                       read x0/x1/x2
set x1 = address                 copy bytes from guest memory
set x2 = length                  write to host stdout
SVC #0             <------------ write result into x0
continue guest code
```

This is why v0.7 is a toy runtime, not a real operating system.

The emulator provides only the tiny services it chooses to support.

## 12. Negative fake errno values

Some syscalls fail without crashing the emulator.

For those, the result goes into `x0` as a negative fake errno value.

| Value | Meaning | Example |
|---|---|---|
| `-EBADF` | bad file descriptor | guest calls `write` with fd `9` |
| `-ENOSYS` | syscall not implemented | guest puts unknown syscall number in `x8` |
| `-EIO` | host I/O failure | host stream write/flush fails |

Why return the error in `x0`?

Because the ABI says:

```text
x0 is both argument 1 before the syscall
and the return value after the syscall
```

So after a bad fd:

```text
before SVC:
    x8 = 64
    x0 = 9
    x1 = valid address
    x2 = length

after SVC:
    x0 = -EBADF
```

This is a syscall-level failure, not necessarily an emulator-level failure.

---

## 13. Runtime errors vs syscall errors

v0.7 makes an important distinction.

### Recoverable syscall errors

These write a negative value into `x0` and execution continues:

```text
unsupported fd      -> x0 = -EBADF
unknown syscall     -> x0 = -ENOSYS
host output failure -> x0 = -EIO
```

The guest program can inspect `x0` and decide what to do next.

### Emulator runtime errors

These stop execution with an error message:

```text
SVC immediate is not 0
write buffer points outside guest memory
write buffer range overflows
instruction limit reached
```

Why is invalid guest memory a runtime error?

Because this is not just “the guest asked for unsupported service.”

It means the emulator was asked to read bytes outside the fake memory array.

The emulator protects itself and reports an execution error.

## 14. Debugging a syscall

Debugger support is important because `SVC #0` is a boundary between guest code and emulator runtime code.

Try:

```sh
make examples/v0_7/hello.bin
./emulator debug examples/v0_7/hello.bin
```

Inside the debugger:

```text
b 0x1010
c
regs
s
regs
q
```

What this means:

```text
b 0x1010   set a breakpoint before the first SVC #0
c          continue until that breakpoint
regs       inspect x0/x1/x2/x8 before the syscall
s          step exactly one instruction, executing SVC #0 once
regs       inspect x0 after the syscall
```

Before `SVC #0`, you should see the syscall arguments prepared:

```text
x0 = 1
x1 = 0x1020
x2 = 13
x8 = 64
```

After stepping over `SVC #0`, the message has been written, and `x0` contains the result:

```text
x0 = 13
```

That means the fake write syscall wrote 13 bytes.

### Debugger error case

If `SVC #0` tries to write from invalid guest memory, stepping or continuing into that instruction reports a runtime error instead of silently reading outside memory.

That gives you a safe place to inspect what arguments were wrong.

## 15. Trace output with `SVC #0`

Trace mode shows the syscall instruction before it executes.

Try:

```sh
./emulator trace examples/v0_7/hello.bin
```

You should see lines like:

```text
trace pc=0x0000000000001010 0x0000000000001010: 0xd4000001  svc #0x0
```

The trace line appears before the syscall side effect.

So the order is:

```text
1. print trace line for SVC #0
2. execute fake syscall
3. write message to stdout or stderr
4. continue or exit
```

This is the same rule from v0.6:

```text
trace shows the instruction about to run
```

## 16. Why v0.7 still uses fixed message addresses

The hello example contains this line:

```asm
movz x1, #0x1020
```

That fixed address may look strange at first.

The reason is that this project still loads raw `.bin` files at a fixed address:

```text
load address = 0x1000
```

The example places message bytes after the instructions.

Because the raw `.bin` has no ELF metadata, no labels at runtime, and no relocation support, the example uses a known address for the message.

This is temporary and intentional.

v0.7 focuses on:

```text
Can a guest ask the emulator to write bytes and exit?
```

It does not yet focus on:

```text
Can the emulator load a full executable format with symbols and relocations?
```

That belongs to a later version.

## 17. What changed inside the emulator

Before v0.7, instruction execution mostly went directly through the CPU stepper.

With v0.7, `SVC` needs special handling because it asks the emulator runtime to do something.

Conceptually:

```c
opcode = fetch(pc);
instruction = decode(opcode);

if (instruction is SVC) {
    emulator_handle_syscall();
} else {
    cpu_execute_instruction();
}
```

Inside syscall handling:

```c
if (svc_immediate != 0) {
    error("unsupported svc immediate");
}

switch (x8) {
case 64:
    handle_write();
    break;
case 93:
    handle_exit();
    break;
default:
    x0 = -ENOSYS;
    pc += 4;
    break;
}
```

This separation matters:

```text
CPU instruction logic knows how to decode SVC.
Emulator runtime logic knows what fake services SVC provides.
```

That keeps `SVC #0` from becoming a normal arithmetic instruction.

It is a boundary.

## 18. Edge cases to understand

v0.7 is small, but edge cases teach important emulator-design thinking.

### Zero-length write

```text
write(1, valid_or_end_address, 0)
```

Writes no bytes and succeeds with:

```text
x0 = 0
```

No guest memory bytes need to be read.

### NUL bytes

`write` copies bytes, not C strings.

So this memory:

```text
41 00 42
```

means three bytes:

```text
'A', NUL, 'B'
```

The length controls how many bytes are written.

There is no automatic stop at `0x00`.

### Bad fd

```text
x0 = 9
x8 = 64
SVC #0
```

returns:

```text
x0 = -EBADF
```

Execution continues.

### Unknown syscall number

```text
x8 = 12345
SVC #0
```

returns:

```text
x0 = -ENOSYS
```

Execution continues.

### Invalid guest buffer

```text
x1 = address outside emulator memory
x2 = non-zero length
x8 = 64
SVC #0
```

This is a runtime error because the emulator refuses to read outside its fake memory.

### Non-zero `SVC` immediate

```asm
svc #1
```

This is rejected because v0.7 only defines `SVC #0`.

## 19. What v0.7 intentionally does not do

v0.7 does **not** add:

```text
real Linux syscall compatibility
real kernel mode
open/read/close syscalls
filesystems
mmap/brk
ELF loading
relocations
symbol lookup
processes
signals
threads
MMU/page tables
```

It is a toy runtime for learning the syscall boundary.

The important goal is:

```text
guest registers + guest memory + SVC #0
    become
observable host output or guest exit status
```

## 21. Common mistakes

### Mistake 1: thinking `SVC #0` itself means print

`SVC #0` only means:

```text
ask the emulator to handle one toy syscall
```

The registers decide which syscall happens.

For printing, `x8` must contain `64` for `write`.

### Mistake 2: confusing a guest address with a host pointer

When `x1 = 0x1020`, that is an address inside emulator guest memory.

It is not a C pointer into your real process.

The emulator must validate the guest range before copying bytes.

### Mistake 3: thinking `-EBADF` means the emulator crashed

`-EBADF` is a recoverable syscall result.

It means the guest asked for an unsupported fd.

Execution can continue.

### Mistake 4: forgetting that `x0` changes meaning

Before `write`, `x0` is argument 1, the fd.

After `write`, `x0` is the return value.

### Mistake 5: thinking `HLT` and `exit` are the same

`HLT` stops execution as an emulator halt instruction.

`exit` is a guest syscall request that carries a status code in `x0` and maps to the CLI exit status.

## 22. Exercises

### Exercise 1

Before `SVC #0`, the registers are:

```text
x8 = 64
x0 = 1
x1 = 0x1020
x2 = 13
```

What syscall is this?

Answer:

```text
write stdout, using 13 bytes starting at guest memory address 0x1020
```

### Exercise 2

After a syscall, `x0 = -EBADF`.

Did the emulator crash?

Answer:

```text
No. The guest asked for an unsupported file descriptor.
The syscall returned an error value in x0 and execution can continue.
```

### Exercise 3

Why does `write` need both address and length?

Answer:

```text
The address says where bytes start in guest memory.
The length says how many bytes to copy.
```

### Exercise 4

Why does v0.7 use `SVC #0` instead of `HLT` for hello-world output?

Answer:

```text
HLT only stops execution.
SVC lets the guest ask the emulator for a service, such as writing bytes to stdout.
```

### Exercise 5

What is the difference between `-ENOSYS` and invalid guest memory?

Answer:

```text
-ENOSYS is a recoverable syscall result for an unknown syscall number.
Invalid guest memory is an emulator runtime error because the emulator refuses to read outside its memory array.
```

## 23. One-sentence summary

v0.7 teaches the emulator a tiny syscall boundary: guest code prepares registers, executes `SVC #0`, and the emulator uses `write = 64` or `exit = 93` to produce host-visible output or a guest exit status while keeping unsupported services explicit and safe.
