# v1.5 Lesson — Toy Kernel Boot and Cooperative Tasks

## 1. The learning ladder so far

The emulator has grown one learning layer at a time.

v0.1 taught the smallest CPU loop: fetch an instruction, decode it, execute it,
and stop on `HLT`.

v0.2 made `pc` interesting by adding branches, loops, comparisons, and flags.
Programs could now choose where execution goes next.

v0.3 added memory and the stack. Registers were no longer the only place values
could live, and `sp` became useful.

v0.4 added function calls. `BL`, `RET`, and `x30` introduced the idea that a
program can temporarily leave one piece of code and later come back.

v0.5 added the debugger REPL so the CPU, memory, breakpoints, and traces could
be inspected while the emulator runs.

v0.6 made traces and disassembly readable enough to learn from.

v0.7 added toy syscalls. `SVC #0` became a controlled doorway from guest code to
host-provided services such as fake `write` and fake `exit`.

v0.8 added ELF64 loading, so programs could have segments, entry points, and
zero-filled memory instead of only flat raw binaries.

v0.9 proved that tiny freestanding C programs can target this emulator when the
compiler profile is intentionally limited.

v1.0 made the project stable with release checks, deterministic fixtures, and
clear documentation.

v1.1 added a small Mach-O teaching loader.

v1.2 replaced the old "one big byte array" view with page mappings and
permissions.

v1.3 added memory-mapped devices. Guest code could talk to UART, timer, random,
and later exception-controller registers by using normal loads and stores.

v1.4 added exceptions, traps, and a deterministic interrupt skeleton. Faults and
`BRK` could become controlled transfers instead of always being fatal host-side
errors.

v1.5 uses those pieces to introduce the first **toy-kernel profile**.

A toy kernel is not a real operating system. In this project, it means a small,
deterministic execution mode where the emulator can boot a kernel-shaped program,
optionally expose boot metadata, and run a fixed set of cooperative tasks.

## 2. Why v1.5 comes after exceptions

A scheduler is mostly controlled control flow.

When a task yields, execution must leave that task, save enough state, choose
another task, restore that task's state, and resume somewhere else. That looks a
lot like exception handling from v1.4:

```text
normal guest code runs
something special happens
emulator saves context
control moves to another place
later execution resumes
```

v1.4 introduced the vocabulary: trap, saved context, vector, return, and
interrupt. v1.5 applies that vocabulary to a smaller and more concrete teaching
problem: cooperative task switching.

The word **cooperative** matters. A task only switches away when it executes a
documented toy-kernel trap. The emulator does not use host wall-clock time and
does not interrupt arbitrary instructions to preempt tasks.

## 3. The old mental model: one program owns the CPU

Before v1.5, the simplest run looked like this:

```text
load program
pc = entry
sp = stack top
run instructions until HLT, fake exit, error, or instruction limit
```

There was only one active register set. `x0` meant the current program's `x0`.
`sp` meant the current program's stack. `pc` meant the current instruction.

That model is still the default. v1.5 does not change raw, ELF64, Mach-O,
syscall, exception-vector, or debugger behavior unless the toy-kernel profile is
explicitly enabled.

## 4. The new mental model: a kernel profile with tasks

With the v1.5 toy-kernel profile enabled, the emulator owns a small task table:

```text
task 0: saved pc, saved sp, saved registers, state
task 1: saved pc, saved sp, saved registers, state
task 2: saved pc, saved sp, saved registers, state
...
```

At any moment, at most one task is `RUNNING`. Other runnable tasks are `READY`.
Exited tasks are `EXITED` and are skipped by the scheduler.

The first development pass supports host-created tasks through the emulator API
and CLI. This is enough to build deterministic examples and future tests before
adding a full guest-side task creation ABI.

## 5. What v1.5 intentionally does not emulate

v1.5 is still a teaching profile, not a production kernel environment.

It intentionally does not implement:

- real ARM exception levels such as `EL0` and `EL1`,
- real per-process address spaces,
- real page tables,
- preemptive multitasking,
- SMP or host threads,
- atomics, locks, or memory-ordering behavior,
- POSIX process APIs,
- dynamic linking,
- filesystems,
- networking,
- device-tree parsing,
- real GIC behavior,
- arbitrary third-party kernel booting.

The goal is smaller: make the scheduler idea visible and testable.

## 6. Enabling toy-kernel mode

The toy-kernel profile is opt-in.

From C code, enable it after loading a program:

```c
bool emulator_enable_toy_kernel(Emulator *emu,
                                bool with_boot_info,
                                char *error,
                                size_t error_size);
```

From the CLI, use:

```sh
./emulator run kernel.bin --kernel
```

A boot-info block can also be requested:

```sh
./emulator run kernel.bin --kernel --kernel-boot-info
```

This keeps old lessons stable. A v0.7 fake-syscall program still uses `SVC #0`.
A v1.4 exception fixture still uses `--exception-vector`. A v1.5 toy-kernel run
must ask for the new profile.

## 7. The boot contract

When toy-kernel mode is enabled, the emulator validates the initial entry point
before running guest code.

The entry point must be:

- 4-byte aligned,
- executable according to the v1.2 memory mapping model,
- ordinary guest RAM, not a v1.3/v1.4 device range.

The initial kernel register contract is intentionally simple:

```text
pc       loaded program entry
sp       loaded program stack top
x0       boot-info pointer, or 0
x1       boot-info size, or 0
x2-x30   zero
flags    N=0 Z=0 C=0 V=0
```

If boot-info is not requested, `x0 == 0` and `x1 == 0`.

If boot-info is requested, `x0` points at a readable/writable guest page holding
a small metadata structure, and `x1` holds that structure's size.

## 8. Boot-info block

The boot-info block exists so a tiny kernel can discover the teaching platform
without hard-coding every address in its source.

The current v1.5 boot-info fields are:

```text
magic
version
size
memory_base
memory_size
kernel_stack_base
kernel_stack_size
task_stack_base
task_stack_size
task_count
uart_base
timer_base
random_base
exception_controller_base
```

The magic value is `0x454d554b45524e31`, which is the teaching marker
`EMUKERN1` in a compact numeric form.

The version is `1`.

The device addresses match the earlier lessons:

```text
UART                  0x09000000
Timer                 0x09010000
Random                0x09020000
Exception controller  0x09030000
```

## 9. Adding cooperative tasks

v1.5 lets the host add tasks before execution starts:

```c
bool emulator_toy_kernel_add_task(Emulator *emu,
                                  uint64_t entry,
                                  char *error,
                                  size_t error_size);
```

The CLI mirrors this with repeatable task entries:

```sh
./emulator trace kernel.bin --kernel --kernel-task 0x1040 --kernel-task 0x1080
```

Each task gets:

- a validated executable entry point,
- a fixed-size 16 KiB stack,
- saved CPU state,
- a state field,
- an exit code field,
- a yield counter,
- a wake tick for sleeping tasks,
- fault metadata for task-local failures.

The current maximum task count is `8`.

Task stacks are ordinary mapped guest RAM. The scheduler keeps them separate by
allocating them downward below the main kernel stack with a guard-sized gap
between stack regions.

## 10. Kernel boot handoff

A v1.5 run has two phases:

```text
phase 1: kernel entry code runs
phase 2: scheduler runs configured tasks
```

The emulator no longer silently jumps straight into the first task. The loaded
program's entry point runs first as kernel boot code. That boot code can inspect
`x0`/`x1`, initialize memory, write a banner, configure devices, or panic.

When the kernel is ready to start tasks, it executes:

```asm
brk #0x154
```

That trap is the explicit scheduler handoff. If no tasks were configured, the
handoff completes successfully and halts. If tasks exist, task 0 is selected by
the round-robin scheduler.

This makes the learning story much clearer:

```text
loader -> kernel entry -> explicit scheduler start -> task contexts
```

## 11. Task states

The toy-kernel state machine is small:

```text
EMPTY     slot is unused
READY     task can be scheduled
RUNNING   task currently owns the CPU registers
BLOCKED   task is sleeping until a future toy-kernel timer tick
EXITED    task finished through the task-exit trap
FAULTED   task hit an unhandled task-local exception
```

The scheduler chooses only `READY` tasks. It skips `EXITED`, `FAULTED`, and
`BLOCKED` tasks.

A blocked task becomes ready again when the toy-kernel timer tick counter reaches
its saved `wake_tick`. A faulted task stays faulted so the debugger and `info`
command can explain what happened.

## 12. Saved context

A context switch saves enough state to leave one task and resume another.

The v1.5 implementation saves:

```text
x0-x30
sp
pc
flags
task state
exit code
yield count
wake tick
fault cause
fault address
```

Saving all general registers is simpler than only saving the AArch64
callee-saved subset. It also makes early tests easier because learners can fill
registers with patterns and check that no task receives another task's values by
accident.

Later lessons can explain real calling conventions and reduce the saved set if
the project wants to teach ABI-specific context switching.

## 13. Toy-kernel traps

v1.5 uses `BRK #imm` as the teaching trap instruction for toy-kernel services.

The current trap numbers are:

```text
BRK #0x150   yield
BRK #0x151   task_exit(x0)
BRK #0x152   panic(x0)
BRK #0x153   console_write(x0=buffer, x1=length)
BRK #0x154   start configured tasks from kernel boot code
BRK #0x155   sleep current task for x0 toy-kernel timer ticks
```

Boot traps are recognized while kernel entry code is running. Task traps are
recognized after scheduling has started.

Outside toy-kernel mode, `BRK` keeps the v1.4 behavior: it is routed through the
exception machinery when a vector is configured, or reported as an unhandled
exception when no vector is configured.

This separation is important. v1.5 does not steal older `BRK` behavior.

## 14. Yield

A yield means:

```text
save current task registers
save resume pc = instruction after BRK
mark task READY
choose next READY task by round-robin order
restore next task registers
continue execution
```

If there is only one ready task, yielding resumes that same task at the next
instruction.

This makes a single-task kernel deterministic and prevents a yield from becoming
an accidental halt.

## 15. Task exit

A task exits with:

```text
x0 = status
BRK #0x151
```

The scheduler saves the status, marks the task `EXITED`, and schedules the next
ready task.

If all tasks are exited, the emulator halts successfully. If at least one task
faulted and no runnable tasks remain, the emulator completes with the stable
v1.5 task-fault status code `71`.

That gives future tests two clear completion rules:

```text
all tasks exited == toy-kernel run completed successfully
all runnable tasks gone and a task faulted == toy-kernel run completed with task-fault status
```

## 16. Panic

A kernel panic uses:

```text
x0 = panic_code
BRK #0x152
```

The emulator records the panic code and stops with a runtime error:

```text
toy kernel panic: code=0x...
```

The CLI maps this to stable status `70`.

This is intentionally fatal. A panic means the kernel profile decided it cannot
continue safely.

## 17. Console output

The toy-kernel console trap uses:

```text
x0 = buffer address
x1 = byte length
BRK #0x153
```

The emulator validates the buffer through the v1.2 readable-memory path, then
writes the bytes to the configured stdout stream.

This is a convenience teaching path. A toy kernel can also write directly to the
v1.3 UART device at `0x09000000`.

## 18. Sleeping and blocked tasks

A task can sleep with:

```text
x0 = number of toy-kernel timer ticks to sleep
BRK #0x155
```

The scheduler saves the task context, marks it `BLOCKED`, and stores:

```text
wake_tick = current_timer_tick + max(x0, 1)
```

The task is not runnable again until the toy-kernel timer reaches that wake tick.

Because this emulator's timer is instruction-count based, not wall-clock based,
a sleeping task needs some other task to keep executing instructions so ticks can
advance. If every task blocks, the emulator reports a deterministic toy-kernel
deadlock instead of pretending time can pass by itself.

## 19. Timer scheduling policy

v1.4 introduced deterministic timer interrupts through:

```sh
--timer-interrupt <interval>
```

v1.5 gives those ticks a toy-kernel meaning when all of the following are true:

```text
toy-kernel mode is enabled
scheduler has started
no exception vector is configured
interrupts are enabled
a timer interrupt is pending
```

In that case the emulator consumes the pending interrupt, increments the
`toy_kernel.timer_ticks` counter, wakes sleeping tasks whose `wake_tick` has
arrived, and requests a scheduler switch at the deterministic instruction
boundary.

If an exception vector is configured, the v1.4 behavior wins: the timer interrupt
enters the exception vector instead of becoming a scheduler tick. This preserves
the exception lesson and lets future kernels write their own timer handler.

## 20. Per-task fault policy

If a running task triggers an unhandled exception and no exception vector is
configured, v1.5 treats that as a task-local fault:

```text
save current task context
mark task FAULTED
record fault cause
record fault address
schedule next READY task
```

This lets one broken task fail without necessarily ending the whole toy-kernel
run. If no ready tasks remain, the emulator completes with task-fault status
`71`.

Faults during kernel boot, scheduler code, or exception-handler execution remain
fatal emulator errors. v1.5 only isolates faults from a running task context.

## 21. Round-robin scheduling

The scheduler uses fixed deterministic round-robin order.

For two tasks, if both keep yielding, the order is:

```text
task 0 -> task 1 -> task 0 -> task 1 -> ...
```

For three tasks, the order is:

```text
task 0 -> task 1 -> task 2 -> task 0 -> ...
```

Exited, faulted, and blocked tasks are skipped.

There is no host time involved. The same input program produces the same task
order every run.

## 22. Interaction with v1.4 exceptions

Toy-kernel traps are implemented as a special v1.5 policy on top of the existing
`BRK` decoder.

The important compatibility rule is:

```text
only recognized toy-kernel BRK immediates are scheduler services
all other BRK behavior remains v1.4 behavior
```

That means exception-vector examples can continue to use `BRK` for trap lessons,
while toy-kernel examples can use documented `BRK #0x150` through `BRK #0x155`
for scheduler services.

Timer behavior has the same compatibility rule. With an exception vector,
`--timer-interrupt` enters the vector. Without a vector, started toy-kernel tasks
use the tick for deterministic scheduling.

`ERET` keeps its v1.4 meaning: it is only valid while the emulator is inside an
active simplified exception. In normal toy-kernel task code, `ERET` outside an
active exception is treated like an invalid task instruction. The faulting task
is marked `FAULTED`, other ready tasks continue, and the final CLI status is
`71` if any task faulted.

## 23. Interaction with v0.7 syscalls

`SVC #0` keeps its older fake-syscall meaning outside the toy-kernel profile.

v1.5 does not redefine the fake `write` and fake `exit` ABI. Tiny v0.7 and v0.9
programs should continue to work as before.

Inside toy-kernel mode, the recommended teaching ABI is `BRK`, not `SVC #0`, so
the syscall profile and the toy-kernel profile stay easy to distinguish.

## 24. Debugger and info visibility

The `info` command now prints toy-kernel metadata when the profile is enabled:

```text
toy_kernel_enabled: yes
toy_kernel_boot_info: yes/no
toy_kernel_entry: ...
toy_kernel_task_count: ...
toy_kernel_timer_ticks: ...
task[0]: state=ready entry=... pc=... sp=... wake=... fault=...
```

The debugger adds:

```text
kernel
```

That command prints the live toy-kernel task table, current task index, timer
counters, panic state, wake ticks, and fault metadata.

Trace mode prints scheduler events such as:

```text
trace kernel-start-tasks count=...
trace kernel-task-switch index=0 pc=... sp=...
trace kernel-task-fault index=... cause=...
trace kernel-complete tasks=... status=...
```

This is intentionally readable rather than architecture-accurate. The goal is to
help learners see when control moved from one task context to another.

## 25. Slow walkthrough: one task exits

Imagine a kernel image shaped like this:

```text
0x1000: brk #0x154        ; kernel starts scheduler
0x1004: hlt               ; would run only if scheduler returned here
0x1008: movz x0, #7       ; task entry
0x100c: brk #0x151        ; task_exit(7)
```

Run it as task 0:

```sh
./emulator trace kernel.bin --kernel --kernel-task 0x1008
```

The flow is:

```text
1. loader maps the raw program at 0x1000
2. toy-kernel mode validates entry 0x1000
3. task 0 is added with its own stack
4. CPU starts at the kernel entry, not at task 0
5. kernel executes BRK #0x154
6. scheduler switches to task 0
7. CPU fetches MOVZ X0, #7
8. CPU fetches BRK #0x151
9. toy-kernel trap handler reads x0 as the exit status
10. task 0 becomes EXITED
11. no ready tasks remain
12. emulator halts successfully
```

No exception vector is required for this path because `BRK #0x154` and
`BRK #0x151` are recognized v1.5 toy-kernel services.

## 26. Slow walkthrough: two tasks yield

Suppose task A does:

```text
write marker A
yield
write marker A again
task_exit
```

and task B does:

```text
write marker B
yield
write marker B again
task_exit
```

Round-robin scheduling should produce:

```text
A B A B
```

The important detail is that each task resumes after its own yield instruction.
The scheduler saves `pc + 4` as the resume address, not the address of the `BRK`
instruction itself.

If it resumed at the `BRK`, the task would yield again forever.

## 27. What changed inside the emulator

v1.5 adds these public teaching pieces:

- `EmuToyKernel`, the toy-kernel profile state,
- `EmuToyTask`, the saved task context,
- `EmuToyKernelBootInfo`, the optional guest-visible metadata block,
- task state names from `EMPTY` through `FAULTED`,
- `emulator_enable_toy_kernel()`,
- `emulator_toy_kernel_add_task()`,
- `emulator_get_toy_kernel()`,
- CLI flags `--kernel`, `--kernel-boot-info`, and repeatable `--kernel-task`,
- toy-kernel trap numbers for yield, task exit, panic, console write, scheduler start, and sleep,
- trace, `info`, and debugger visibility for task scheduling.

The implementation is deliberately host-managed for now. Future tests can pin
this behavior before the project adds guest-side task creation helpers.

## 28. What tests should cover next

The v1.5 test plan should eventually cover:

- default behavior remains unchanged when `--kernel` is absent,
- invalid kernel entry addresses are rejected,
- boot registers are deterministic,
- kernel code runs before tasks and starts scheduling with `BRK #0x154`,
- boot-info fields are readable and correct,
- task entries must be aligned and executable,
- task stacks do not overlap,
- single task exit completes successfully,
- one task yielding resumes itself,
- two and three task round-robin order,
- exited, blocked, and faulted tasks are skipped,
- timer ticks wake sleeping tasks and can request deterministic scheduling,
- task-local faults mark only the faulting task as `FAULTED`,
- `x0-x30`, `sp`, `pc`, and flags survive context switches,
- panic trap returns a stable diagnostic and CLI status,
- console trap validates readable buffers,
- unknown `BRK` immediates keep v1.4 exception behavior,
- old `SVC #0` fake syscalls still pass.

## 29. Common mistakes

The most common mistakes are:

- expecting toy-kernel traps to work without `--kernel`,
- forgetting the kernel must call `BRK #0x154` before tasks run,
- using `SVC #0` for toy-kernel services instead of the documented `BRK`
  immediates,
- forgetting that task entry addresses are guest addresses, not host pointers,
- adding a task whose entry is not inside an executable mapping,
- expecting host-wall-clock preemption,
- expecting a sleeping task to wake when every task is blocked and no
  instructions can execute,
- expecting `HLT` to mean task exit,
- assuming this is an architectural ARM kernel ABI.

## 31. Exercises

1. Write a task that yields once and then exits.
2. Write two tasks that print different UART bytes and yield between prints.
3. Modify one task to exit early and confirm the second task continues.
4. Trigger `BRK #0x152` with a panic code in `x0` and read the error.
5. Inspect the task table with `info` before running the program.
6. Compare `BRK #0x150` with an unknown `BRK #0x999` in kernel mode.
7. Explain why saving `pc + 4` matters for yield and task exit.
8. Add a sleeping task and explain why another runnable task is needed for the
   instruction-count timer to advance.
9. Fault one task and verify that another task can still exit.

## 32. One-sentence summary

v1.5 turns the emulator's existing loader, memory, device, and exception lessons
into an opt-in toy-kernel profile where fixed cooperative tasks can start from a
visible kernel boot handoff, yield, sleep, exit, panic, fault independently, and
preserve CPU context deterministically.
