# v0.1 Lesson: Understanding the Emulator

This guide is for someone who is new to the project and still building intuition for CPU emulation, ARM64 assembly, registers, memory, and how the example program runs.

It combines the earlier beginner notes with the more detailed explanation of the flow from assembly text to machine-code instructions.

The short version:

```text
This project is a tiny fake ARM64 computer.

It loads raw instruction bytes into fake memory,
uses pc to fetch one 32-bit instruction at a time,
decodes the instruction bits,
updates fake CPU registers/memory,
and stops when it reaches HLT.
```

---

## 1. What this project is

`emulator` is a tiny ARM64 emulator written in C, with small demo programs written in ARM64 assembly.

An emulator is a program that pretends to be another machine.

In this project:

```text
Your real computer
    runs
./emulator
    which pretends to be
a tiny ARM64 CPU + memory system
    which runs
raw ARM64 machine code
```

v0.1 is intentionally small. It does not boot Linux, iOS, or any real operating system. It is an instruction sandbox.

The v0.1 goal is:

```text
assembly example
    -> assembled into raw ARM64 instruction bytes
    -> loaded into fake memory
    -> fetched by pc
    -> decoded by the emulator
    -> executed by updating fake CPU state
```

---

## 2. The mental model

Think of the emulator as a fake computer with two main parts:

```text
+------------------------------+
|          Emulator            |
|                              |
|  +--------+    +----------+  |
|  |  CPU   | -> |  Memory  |  |
|  +--------+    +----------+  |
|      |               |       |
|      v               v       |
|  registers       program     |
|  pc              bytes       |
|  sp                          |
+------------------------------+
```

The CPU repeatedly does this:

```text
1. Fetch instruction from memory at pc.
2. Decode the instruction bits.
3. Execute the instruction.
4. Update registers/memory.
5. Move pc to the next instruction.
6. Repeat until HLT or error.
```

In simplified C-like pseudocode:

```c
while (!cpu->halted) {
    uint32_t instruction = memory_read32(memory, cpu->pc);
    execute_instruction(cpu, memory, instruction);
}
```

That loop is the heart of the emulator.

---

## 3. Build time vs run time

There are two separate worlds.

```text
World 1: Build time
ASM text  ->  object file  ->  raw machine-code .bin

World 2: Run time
.bin bytes  ->  emulator memory  ->  fetch/decode/execute loop
```

This distinction is important:

```text
Assembly is for humans.
Machine code bytes are for the emulator.
```

The emulator does not read the text `movz x0, #2` directly. It reads the bytes that the assembler produced from that text.

---

## 4. The v0.1 example program

The main example is:

```asm
movz x0, #2
movz x1, #3
add  x2, x0, #3
hlt  #0
```

In plain English:

```text
Put 2 into x0.
Put 3 into x1.
Add 3 to x0 and store the result in x2.
Stop.
```

Expected final result:

```text
x0 = 2
x1 = 3
x2 = 5
```

---

## 5. What is assembly?

Assembly is a human-readable form of CPU instructions.

Example:

```asm
movz x0, #2
```

Meaning:

```text
Put the number 2 into register x0.
```

Another example:

```asm
add x2, x0, #3
```

Meaning:

```text
x2 = x0 + 3
```

So if:

```text
x0 = 2
```

then after:

```asm
add x2, x0, #3
```

we get:

```text
x2 = 5
```

---

## 6. ASM is text, not executable bytes

The file:

```text
examples/v0_1/add.s
```

contains human-readable ARM64 assembly.

This line:

```asm
movz x0, #2
```

is still just text. The emulator does not execute the letters `m`, `o`, `v`, `z`.

Before the emulator can run the program, another tool must translate the assembly into machine code. That tool is the assembler.

---

## 7. The assembler converts ASM into machine code

When you run:

```sh
make examples
```

the Makefile assembles the `.s` file into an object file, roughly like this:

```sh
clang --target=aarch64-none-elf -c examples/v0_1/add.s -o examples/v0_1/add.o
```

That means:

```text
Take add.s.
Assemble it as ARM64 code.
Produce add.o.
```

The `.o` file is an object file. It contains machine code plus metadata.

v0.1 of our emulator is intentionally simple. It does not understand object files yet.

So the Makefile extracts only the raw instruction bytes:

```sh
llvm-objcopy -O binary -j .text examples/v0_1/add.o examples/v0_1/add.bin
```

Now we have:

```text
examples/v0_1/add.bin
```

That `.bin` file contains raw ARM64 instruction bytes.

---

## 8. Each ARM64 instruction is 4 bytes

ARM64 instructions are normally fixed-width:

```text
1 instruction = 32 bits = 4 bytes
```

Our example has four instructions:

```asm
movz x0, #2
movz x1, #3
add  x2, x0, #3
hlt  #0
```

So the raw binary is:

```text
4 instructions * 4 bytes = 16 bytes
```

Conceptually, `add.bin` looks like:

```text
[ instruction 1 ][ instruction 2 ][ instruction 3 ][ instruction 4 ]
     4 bytes          4 bytes          4 bytes          4 bytes
```

The emulator reads one 4-byte group at a time.

---

## 9. Seeing the raw bytes yourself

After building examples, run:

```sh
xxd examples/v0_1/add.bin
```

You may see output like:

```text
00000000: 4000 80d2 6100 80d2 420c 0091 0000 40d4
```

Those bytes are the actual program.

They correspond to:

```asm
movz x0, #2
movz x1, #3
add  x2, x0, #3
hlt  #0
```

The emulator reads them as four 32-bit instructions:

```text
0xd2800040
0xd2800061
0x91000c42
0xd4400000
```

Because ARM64 is little-endian, each 32-bit instruction appears byte-reversed in the hex dump.

Example:

```text
bytes in file:     40 00 80 d2
32-bit value:      0xd2800040
assembly meaning:  movz x0, #2
```

This is the bridge:

```text
movz x0, #2
    -> assembler
40 00 80 d2
    -> emulator reads little-endian
0xd2800040
    -> emulator decodes as MOVZ x0, #2
    -> emulator sets x0 = 2
```

---

## 10. What is memory?

Memory is just a big array of bytes.

In v0.1, the emulator uses:

```text
1 MiB memory
= 1,048,576 bytes
= addresses 0x00000000 to 0x000FFFFF
```

The project loads the program at:

```text
0x1000
```

So memory looks like this:

```text
0x00000000  +------------------+
            | unused / zero    |
0x00001000  +------------------+
            | program code     |
            | instruction 1    |
            | instruction 2    |
            | instruction 3    |
            | instruction 4    |
            +------------------+
            | unused / zero    |
            |                  |
0x00100000  +------------------+
```

The stack pointer starts at the top of memory:

```text
sp = 0x100000
```

v0.1 does not really use stack instructions yet. Stack becomes important in later versions.

---

## 11. What is a CPU register?

A register is a tiny storage slot inside the CPU.

In ARM64, important general registers are:

```text
x0
x1
x2
...
x30
```

Each `x` register is 64 bits wide.

You can think of them as variables owned by the fake CPU:

```text
x0 = 2
x1 = 3
x2 = 5
```

The emulator stores them in C conceptually like this:

```c
uint64_t x[31];
```

So:

```text
x0  -> x[0]
x1  -> x[1]
x2  -> x[2]
...
x30 -> x[30]
```

---

## 12. Special registers: pc and sp

Besides `x0-x30`, the CPU state also has:

```text
pc = program counter
sp = stack pointer
```

### pc

`pc` tells the CPU where the next instruction is.

Example:

```text
pc = 0x1000
```

means:

```text
Read the next instruction from memory address 0x1000.
```

ARM64 instructions are normally 4 bytes, so after one normal instruction:

```text
pc = pc + 4
```

Example:

```text
Before instruction: pc = 0x1000
After instruction:  pc = 0x1004
```

### sp

`sp` is the stack pointer.

In v0.1 it is initialized, but not heavily used yet.

Later, when memory/stack instructions are implemented, `sp` will point to the current top of the stack.

---

## 13. Fetch, decode, execute

Every instruction goes through three steps:

```text
FETCH:
Read 4 bytes from memory at pc.

DECODE:
Look at the bits and figure out which instruction it is.

EXECUTE:
Perform the instruction's effect.
```

In simplified C:

```c
while (!cpu->halted) {
    uint32_t instruction = fetch(memory, cpu->pc);
    execute(cpu, memory, instruction);
}
```

---

## 14. What is instruction decoding?

Instruction decoding means taking raw bits and figuring out what instruction they represent.

The CPU does not see:

```asm
movz x0, #2
```

It sees a 32-bit value:

```text
0xd2800040
```

The emulator uses masks and shifts to inspect those bits.

Conceptually:

```c
if ((instruction & SOME_MASK) == MOVZ_PATTERN) {
    execute_movz(...);
}
```

A mask selects important bits.

Example:

```c
instruction & 0xFF000000
```

means:

```text
Only keep the top group of bits.
Ignore the rest.
```

A shift moves bits into a convenient position.

Example:

```c
rd = instruction & 0x1F;
```

This extracts the destination register field when it lives in the lowest 5 bits.

Why 5 bits? Because 5 bits can represent numbers from 0 to 31, which is enough to identify ARM64 register numbers.

## 15. Step-by-step example run

Example program:

```asm
movz x0, #2
movz x1, #3
add  x2, x0, #3
hlt  #0
```

Initial CPU state:

```text
pc = 0x1000
sp = 0x100000

x0 = 0
x1 = 0
x2 = 0
halted = false
```

### Step 1

`pc` is:

```text
0x1000
```

Memory at `0x1000` contains the 4 bytes for:

```asm
movz x0, #2
```

The emulator fetches those 4 bytes and decodes:

```text
This is MOVZ.
Destination register is x0.
Immediate value is 2.
```

Then it executes:

```text
x0 = 2
pc = pc + 4
```

Now:

```text
pc = 0x1004
x0 = 2
x1 = 0
x2 = 0
```

### Step 2

`pc` is:

```text
0x1004
```

Memory at `0x1004` contains the 4 bytes for:

```asm
movz x1, #3
```

The emulator decodes:

```text
This is MOVZ.
Destination register is x1.
Immediate value is 3.
```

Executes:

```text
x1 = 3
pc = pc + 4
```

Now:

```text
pc = 0x1008
x0 = 2
x1 = 3
x2 = 0
```

### Step 3

`pc` is:

```text
0x1008
```

Memory at `0x1008` contains the 4 bytes for:

```asm
add x2, x0, #3
```

The emulator decodes:

```text
This is ADD immediate.
Destination register is x2.
Source register is x0.
Immediate value is 3.
```

Executes:

```text
x2 = x0 + 3
x2 = 2 + 3
x2 = 5
pc = pc + 4
```

Now:

```text
pc = 0x100c
x0 = 2
x1 = 3
x2 = 5
```

### Step 4

`pc` is:

```text
0x100c
```

Memory at `0x100c` contains the 4 bytes for:

```asm
hlt #0
```

The emulator decodes:

```text
This is HLT.
Stop execution.
```

Executes:

```text
halted = true
```

The run loop ends.

Final result:

```text
x0 = 2
x1 = 3
x2 = 5
```

## 16. What v0.1 currently supports

v0.1 supports only a small subset of ARM64:

```text
NOP
HLT
MOVZ
ADD immediate
SUB immediate
```

It also supports:

```text
- 1 MiB flat memory
- raw binary loading at 0x1000
- pc initialized to 0x1000
- sp initialized to 0x100000
- final register dump
- instruction limit to prevent runaway execution
- error handling for unsupported instructions and bad fetches
```

## 17. What each v0.1 instruction does

### NOP

```asm
nop
```

Meaning:

```text
Do nothing.
Move to the next instruction.
```

Effect:

```text
pc = pc + 4
```

### HLT

```asm
hlt #0
```

Meaning:

```text
Stop execution.
```

In our emulator:

```text
cpu->halted = true
```

### MOVZ

```asm
movz x0, #2
```

Meaning:

```text
x0 = 2
```

`MOVZ` stands for “move wide with zero.” It clears the destination register first, then places the immediate value.

### ADD immediate

```asm
add x2, x0, #3
```

Meaning:

```text
x2 = x0 + 3
```

### SUB immediate

```asm
sub x2, x0, #1
```

Meaning:

```text
x2 = x0 - 1
```


This is the core of the emulator:


## 20. Important C concepts used in the emulator

### Structs

A struct groups related data.

Conceptually:

```c
typedef struct {
    uint64_t x[31];
    uint64_t sp;
    uint64_t pc;
    bool halted;
} Cpu;
```

This means the CPU state is one object.

### Pointers

Functions receive pointers so they can modify the real object.

Example:

```c
void cpu_init(Cpu *cpu);
```

This function modifies the actual CPU struct. Without a pointer, it would modify only a copy.

### Fixed-width integers

The emulator uses precise integer sizes:

```text
uint64_t = unsigned 64-bit integer
uint32_t = unsigned 32-bit integer
uint8_t  = unsigned 8-bit integer
```

They matter because emulator code must be precise:

```text
ARM64 register = 64 bits
ARM64 instruction = 32 bits
memory byte = 8 bits
```

---

## 21. Why little-endian matters

Memory stores bytes.

A 32-bit instruction is 4 bytes.

In little-endian order, the least significant byte comes first.

Example:

```text
32-bit value: 0x12345678
```

Stored in memory as:

```text
address + 0: 0x78
address + 1: 0x56
address + 2: 0x34
address + 3: 0x12
```

So `memory_read32` must combine bytes correctly.

Conceptually:

```c
value = byte0
      | byte1 << 8
      | byte2 << 16
      | byte3 << 24;
```

If endian handling is wrong, the emulator decodes nonsense instructions.

## 22. What the v0.1 test plan protects

The test plan is not only checking “does add work?”

It protects the emulator from subtle bugs.

### CPU initialization

Checks that:

```text
- all registers start at 0
- pc starts at 0x1000
- sp starts at 0x100000
- halted starts false
```

### Memory safety

Checks that:

```text
- memory starts zeroed
- reads/writes work
- out-of-bounds access fails
- boundary-crossing access fails
```

### Instruction behavior

Checks that:

```text
- MOVZ writes correct values
- ADD computes correctly
- SUB computes correctly
- NOP changes nothing except pc
- HLT stops execution
```

### Error behavior

Checks that the emulator handles:

```text
- bad opcodes
- misaligned pc
- pc outside memory
- infinite loops / instruction limit
- missing input file
- empty input file
```

For an emulator, error behavior is as important as success behavior.

## 23. How to debug this project

Debugging is easier if you follow the emulator flow.

When instruction output is wrong, ask:

```text
1. Did memory load the correct bytes?
2. Did fetch read the correct 32-bit instruction?
3. Did decode match the correct instruction type?
4. Did decode extract the correct register fields?
5. Did execute write to the correct register?
6. Did pc advance correctly?
```

Useful debug print:

```c
printf("pc=0x%llx instruction=0x%08x\n", cpu->pc, instruction);
```

If the program crashes, use LLDB:

```sh
lldb ./emulator
```

Then:

```text
run run examples/v0_1/add.bin
bt
```

Useful LLDB commands:

```text
break main
run run examples/v0_1/add.bin
next
step
print cpu
print cpu.pc
print instruction
continue
bt
```

Read `cpu.c` in this order:

```text
1. CPU initialization
2. Register read/write helpers

4. Execute one instruction
5. Run loop
```

```text
run loop

```

## 25. Common beginner confusion

### “Why not just call C functions directly?”

Because we are not running C logic directly. We are simulating how a CPU executes machine code.

This assembly:

```asm
add x2, x0, #3
```

does not become a C function call like:

```c
add(x2, x0, 3);
```

Instead, it becomes 4 bytes of machine code. The emulator reads those bytes and manually performs the effect.

### “Why do we need raw `.bin` files?”

Because v0.1 has no real executable loader yet.

A normal program file like ELF or Mach-O contains metadata:

```text
- headers
- sections
- symbols
- segment info
- debug info
- relocations
```

v0.1 skips all of that. It only wants raw instruction bytes.

### “Why does pc increase by 4?”

Because ARM64 instructions are normally 4 bytes.

So instruction addresses look like:

```text
0x1000
0x1004
0x1008
0x100c
```

Each instruction occupies 4 bytes.

### “Why is x31 special?”

ARM64 uses register number 31 specially.

Depending on the instruction context, it can mean:

```text
xzr = zero register
sp  = stack pointer
```

In v0.1, for supported instructions, it mainly behaves like the zero register.

The zero register means:

```text
Reading it gives 0.
Writing to it does nothing.
```

## 26. How future versions build on v0.1

### v0.2 — Branches and loops

Adds instructions that change `pc`.

Instead of always:

```text
pc = pc + 4
```

a branch can do:

```text
pc = some_other_address
```

This enables loops.

### v0.3 — Memory and stack

Adds instructions that read/write memory:

```asm
str x0, [sp, #-8]!
ldr x1, [sp], #8
```

This enables local variables and stack behavior.

### v0.4 — Functions

Adds:

```text
BL
RET
```

`BL` calls a function by saving the return address in `x30`, then changing `pc`.

`RET` returns by setting `pc` back to `x30`.

### v0.5 — Debugger REPL

Adds interactive commands:

```text
step
regs
mem
break
continue
```

### v0.7 — Fake syscalls

Allows a program to ask the emulator for simple services:

```text
print this string
exit with this code
```

This is how we eventually get `hello world`.

### v0.8 — ELF loader

Loads real ARM64 ELF files instead of raw `.bin` files.

### v1.1 — Mach-O loader

Adds Apple-focused executable loading. Mach-O is the executable format used by macOS and iOS.

## 27. What to learn next

The best learning order is:

```text
1. CPU registers: x0-x30, pc, sp
2. How pc moves through instructions
3. Basic ARM64 assembly: movz, add, sub, hlt
4. Binary/hex instruction representation
5. Instruction decoding with masks and shifts
6. Memory reads/writes and endian behavior
7. Debugging C with printf + lldb/gdb
8. Writing tests for emulator behavior
```

Do not worry too much about CPU flags yet. Flags become more useful in v0.2 when comparisons and branches arrive.
