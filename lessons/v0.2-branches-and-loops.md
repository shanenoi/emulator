# v0.2 Lesson — Branches, Loops, and Flags

This guide is written as a slow continuation of `lessons/v0.1-instruction-sandbox.md`.

In v0.1, the emulator ran programs in a straight line:

```text
0x1000 -> 0x1004 -> 0x1008 -> 0x100c
```

Every normal ARM64 instruction is 4 bytes, so the emulator usually does:

```c
pc = pc + 4;
```

v0.2 adds one big new idea: **control flow**.

Control flow means the program can choose where execution goes next. A loop is just control flow that jumps backward.

---

## 1. The most important mental model

The `pc` is the address of the next instruction.

```text
pc = 0x1000 means:
"fetch 4 bytes from memory address 0x1000 and execute that instruction"
```

In v0.1, the `pc` mostly moved forward:

```text
before instruction: pc = 0x1000
after instruction:  pc = 0x1004
```

In v0.2, some instructions can set `pc` to a different address:

```text
normal instruction:       pc = pc + 4
branch taken:             pc = target_address
branch not taken:         pc = pc + 4
```

This is the whole foundation of branches and loops.

---

## 2. What is a label?

Assembly lets humans write names for addresses:

```asm
loop:
    sub  x0, x0, #1
    cbnz x0, loop
```

The label `loop` is not stored as the word `loop` in the final `.bin` file.

During assembly, the assembler turns labels into addresses or offsets.

Think of it like this:

```text
Assembly text:
loop:
    sub  x0, x0, #1
    cbnz x0, loop

After assembly, conceptually:
0x1000: sub  x0, x0, #1
0x1004: cbnz x0, 0x1000
```

The CPU/emulator never sees the name `loop`. It only sees encoded instruction bytes that mean “branch backward to that address.”

---

## 3. Labels become offsets

ARM64 branches usually encode a relative offset, not a full address.

That means the branch says something like:

```text
from my current pc, jump forward 8 bytes
```

or:

```text
from my current pc, jump backward 4 bytes
```

Example:

```asm
    b target
    nop
target:
    hlt #0
```

Address table:

```text
Address   Instruction
-------   ----------------
0x1000    b target
0x1004    nop
0x1008    hlt #0
```

The branch is at `0x1000` and the target is `0x1008`.

So the encoded offset is:

```text
target - current_pc = 0x1008 - 0x1000 = 8 bytes
```

The source code says `target`, but the machine code says “jump +8 bytes.”

Backward branches work the same way:

```asm
loop:
    sub  x0, x0, #1
    cbnz x0, loop
```

Address table:

```text
Address   Instruction
-------   ----------------
0x1000    sub  x0, x0, #1
0x1004    cbnz x0, loop
```

The `cbnz` is at `0x1004`. The target `loop` is at `0x1000`.

```text
target - current_pc = 0x1000 - 0x1004 = -4 bytes
```

So this branch jumps backward by 4 bytes.

---

## 4. Unconditional branch: `B`

`B` means “branch always.”

Assembly:

```asm
b target
```

C-like pseudocode:

```c
pc = target;
```

Example:

```asm
movz x0, #1
b    done
movz x0, #99
done:
hlt  #0
```

Address table:

```text
Address   Instruction       Effect
-------   ---------------   -------------------------
0x1000    movz x0, #1       x0 = 1
0x1004    b done            pc = 0x100c
0x1008    movz x0, #99      skipped
0x100c    hlt #0            stop
```

Final result:

```text
x0 = 1
```

`movz x0, #99` never executes.

---

## 5. Conditional branch idea

A conditional branch has two possible outcomes:

```text
condition true:   pc = target
condition false:  pc = pc + 4
```

This means the emulator asks a question before deciding where to go.

Examples:

```asm
b.eq target    // branch if equal
b.ne target    // branch if not equal
b.lt target    // branch if less than
b.ge target    // branch if greater than or equal
```

But how does the CPU know whether something is equal or less than?

It uses **flags**.

---

## 6. What are flags?

Flags are tiny yes/no values stored in the CPU.

v0.2 uses these flags:

```text
Z = zero flag
N = negative flag
C = carry flag
V = overflow flag
```

For now, focus on `Z` and `N` first.

### `Z` flag

`Z` means the last comparison result was zero.

```text
5 - 5 = 0  -> Z = 1
5 - 3 = 2  -> Z = 0
```

So:

```asm
cmp x0, #5
b.eq equal
```

means:

```c
if (x0 == 5) goto equal;
```

### `N` flag

`N` means the result looked negative in signed arithmetic.

```text
3 - 5 = -2 -> N = 1
7 - 5 = 2  -> N = 0
```

This helps with signed comparisons like `b.lt` and `b.ge`.

### `C` and `V`

`C` and `V` are harder:

```text
C = useful for unsigned comparisons
V = useful for signed overflow detection
```

You do not need to master them immediately. For v0.2, it is enough to know that `b.lt`, `b.le`, `b.gt`, and `b.ge` use a combination of flags, not just one flag.

---

## 7. `CMP`: compare without storing the result

`CMP` is one of the confusing parts.

Assembly:

```asm
cmp x0, #5
```

C-like pseudocode:

```c
temporary = x0 - 5;
update_flags_from(temporary);
// do not store temporary anywhere
```

Important: `CMP` does **not** change `x0`.

Example:

```asm
movz x0, #5
cmp  x0, #5
hlt  #0
```

Step table:

```text
Step   Instruction   x0   What changes?
----   -----------   --   ----------------------------
1      movz x0,#5    5    x0 becomes 5
2      cmp x0,#5     5    flags change, x0 stays 5
3      hlt           5    stop
```

After `cmp x0, #5`:

```text
5 - 5 = 0
Z = 1
```

That means `b.eq` would be taken.

---

## 8. `CBZ` and `CBNZ`

`CBZ` means:

```text
Compare register with zero, then branch if zero.
```

C-like pseudocode:

```c
if (x0 == 0) {
    pc = target;
} else {
    pc = pc + 4;
}
```

`CBNZ` means:

```text
Compare register with zero, then branch if not zero.
```

C-like pseudocode:

```c
if (x0 != 0) {
    pc = target;
} else {
    pc = pc + 4;
}
```

`CBZ` / `CBNZ` are useful for simple loops because they do not require a separate `CMP` instruction.

## 9. Slow walkthrough: countdown loop with `CBNZ`

Assembly:

```asm
movz x0, #3

loop:
    sub  x0, x0, #1
    cbnz x0, loop

hlt #0
```

C-like pseudocode:

```c
x0 = 3;
do {
    x0 = x0 - 1;
} while (x0 != 0);
halt();
```

Address table:

```text
Address   Instruction
-------   ----------------
0x1000    movz x0, #3
0x1004    sub  x0, x0, #1
0x1008    cbnz x0, loop
0x100c    hlt #0
```

Trace table:

```text
Step   pc       Instruction        x0 before   x0 after   Next pc
----   ------   ----------------   ---------   --------   -------
1      0x1000   movz x0, #3        0           3          0x1004
2      0x1004   sub x0, x0, #1     3           2          0x1008
3      0x1008   cbnz x0, loop      2           2          0x1004  taken
4      0x1004   sub x0, x0, #1     2           1          0x1008
5      0x1008   cbnz x0, loop      1           1          0x1004  taken
6      0x1004   sub x0, x0, #1     1           0          0x1008
7      0x1008   cbnz x0, loop      0           0          0x100c  not taken
8      0x100c   hlt #0             0           0          stop
```

Final result:

```text
x0 = 0
instructions executed = 8
```

Notice why the instruction count is 8:

```text
movz executes once
sub executes three times
cbnz executes three times
hlt executes once

1 + 3 + 3 + 1 = 8
```

This is why loop instruction counts can feel surprising.

## 10. Slow walkthrough: `CMP` + `B.LT` loop

Assembly:

```asm
movz x0, #0
movz x1, #3

loop:
    add  x0, x0, #1
    cmp  x0, x1
    b.lt loop

hlt #0
```

C-like pseudocode:

```c
x0 = 0;
x1 = 3;
do {
    x0 = x0 + 1;
} while (x0 < x1);
halt();
```

Address table:

```text
Address   Instruction
-------   ----------------
0x1000    movz x0, #0
0x1004    movz x1, #3
0x1008    add  x0, x0, #1
0x100c    cmp  x0, x1
0x1010    b.lt loop
0x1014    hlt #0
```

Step table:

```text
Step   pc       Instruction      x0   x1   Flags meaning       Next pc
----   ------   --------------   --   --   -----------------   -------
1      0x1000   movz x0,#0       0    0    -                   0x1004
2      0x1004   movz x1,#3       0    3    -                   0x1008
3      0x1008   add x0,x0,#1     1    3    -                   0x100c
4      0x100c   cmp x0,x1        1    3    1 < 3               0x1010
5      0x1010   b.lt loop        1    3    true                0x1008
6      0x1008   add x0,x0,#1     2    3    -                   0x100c
7      0x100c   cmp x0,x1        2    3    2 < 3               0x1010
8      0x1010   b.lt loop        2    3    true                0x1008
9      0x1008   add x0,x0,#1     3    3    -                   0x100c
10     0x100c   cmp x0,x1        3    3    equal, Z = 1        0x1010
11     0x1010   b.lt loop        3    3    false               0x1014
12     0x1014   hlt #0           3    3    -                   stop
```

Final result:

```text
x0 = 3
x1 = 3
instructions executed = 12
```

## 11. How v0.2 fits into the emulator code

At a high level, `cpu_step()` still does the same v0.1 loop:

```c
opcode = fetch(pc);
instruction = decode(opcode);
execute(instruction);
```

The new part is that execution sometimes changes `pc` differently.

For normal instructions:

```c
cpu->pc += 4;
```

For an unconditional branch:

```c
cpu->pc = target;
```

For `CBNZ`:

```c
if (read_register(rn) != 0) {
    cpu->pc = target;
} else {
    cpu->pc += 4;
}
```

For `CMP`:

```c
result = left - right;
set_flags(result);
cpu->pc += 4;
```

For `B.cond`:

```c
if (condition_passed(flags, condition)) {
    cpu->pc = target;
} else {
    cpu->pc += 4;
}
```

## 12. Common mistakes

### Mistake 1: thinking labels exist at runtime

They do not. Labels are assembly-time names.

Runtime uses addresses and offsets.

### Mistake 2: thinking `CMP` stores a result

It does not.

```asm
cmp x0, #5
```

means:

```text
change flags only
```

### Mistake 3: forgetting branch instructions execute too

In this loop:

```asm
sub  x0, x0, #1
cbnz x0, loop
```

Both `sub` and `cbnz` count as executed instructions each time.

### Mistake 4: thinking not-taken branch means “do nothing”

A not-taken branch still executes. It just moves to the next instruction:

```text
pc = pc + 4
```

## 13. Exercises

### Exercise 1

```asm
movz x0, #2
loop:
    sub  x0, x0, #1
    cbnz x0, loop
hlt #0
```

Questions:

```text
Final x0?
How many times does sub execute?
How many times does cbnz execute?
```

Answer:

```text
x0 = 0
sub executes 2 times
cbnz executes 2 times
```

### Exercise 2

```asm
movz x0, #3
cmp  x0, #3
b.eq equal
movz x1, #99
equal:
hlt #0
```

Question:

```text
Does movz x1, #99 execute?
```

Answer:

```text
No. cmp sets Z = 1, so b.eq is taken.
```

---

## 14. One-sentence summary

v0.2 teaches the emulator how to change `pc`: branches jump to labels that became offsets, loops are backward branches, `CMP` changes flags, and conditional branches use those flags to decide whether to jump or keep moving forward.
