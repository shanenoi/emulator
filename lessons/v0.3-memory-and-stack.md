# v0.3 Lesson — Memory and Stack

This guide is beginner-first. It assumes you understand the v0.1 idea that the emulator has CPU registers and a big memory array, but it does **not** assume you are comfortable with stack or load/store instructions yet.

v0.3 adds a huge new ability:

```text
The program can read and write data memory.
```

v0.1/v0.2 mostly changed registers and `pc`.
v0.3 lets instructions move values between registers and memory.

---

## 1. Memory is numbered bytes

Think of memory as a giant table:

```text
Address      Byte
---------    ----
0x00000000   00
0x00000001   00
0x00000002   00
...
0x00001000   program byte
...
0x000fffff   00
```

In this emulator:

```text
memory size = 1 MiB
valid addresses = 0x00000000 through 0x000fffff
```

The stack pointer starts at the top:

```text
sp = 0x00100000
```

`0x00100000` is just past the last valid byte. That is okay because the stack grows downward before storing.

---

## 2. Registers vs memory

Registers are inside the CPU:

```text
x0 = 42
x1 = 0
sp = 0x100000
```

Memory is outside the CPU:

```text
memory[0xffff8] = 00
memory[0xffff9] = 00
...
```

v0.3 instructions move values between these two worlds.

```text
STR = store register into memory
LDR = load memory into register
```

A helpful direction rule:

```text
STR: register -> memory
LDR: memory   -> register
```

---

## 3. First simple example: store then load

Assembly:

```asm
movz x0, #42
str  x0, [sp, #-8]!
ldr  x1, [sp], #8
hlt  #0
```

High-level meaning:

```c
x0 = 42;
sp = sp - 8;
memory[sp] = x0;
x1 = memory[sp];
sp = sp + 8;
halt();
```

Final result:

```text
x0 = 42
x1 = 42
sp = 0x100000
```

The stack pointer returns to its starting value because we pushed once and popped once.

---

## 4. What do brackets mean?

In ARM64 assembly, brackets mean “use this as a memory address.”

```asm
str x0, [x2]
```

means:

```c
memory[x2] = x0;
```

If:

```text
x2 = 0x2000
x0 = 42
```

then:

```asm
str x0, [x2]
```

stores 42 at memory address `0x2000`.

---

## 5. Base + offset addressing

Assembly:

```asm
str x0, [x2, #16]
```

C-like pseudocode:

```c
address = x2 + 16;
memory[address] = x0;
```

Important: this form does **not** change `x2`.

Example:

```text
before:
x2 = 0x2000
x0 = 42

instruction:
str x0, [x2, #16]

after:
memory[0x2010] = 42
x2 = 0x2000   // unchanged
```

---

## 6. Why the stack grows downward

The stack starts near the top of memory:

```text
sp = 0x100000
```

When we push something, we usually subtract from `sp` first:

```text
sp = sp - 8
```

Then we store at the new address.

Why downward? That is a common CPU/OS convention. It lets stack memory grow from high addresses toward lower addresses while code/data can live lower in memory.

In our emulator, remember this rule:

```text
push: sp goes down
pop:  sp goes up
```

---

## 7. Pre-index form: `[sp, #-8]!`

This is the syntax that often causes confusion:

```asm
str x0, [sp, #-8]!
```

The `!` means “write the changed address back into the base register.”

For pre-index:

```text
change base first, then access memory
```

C-like pseudocode:

```c
sp = sp - 8;
memory[sp] = x0;
```

Slow table:

```text
Initial:
sp = 0x100000
x0 = 42
```

After:

```text
sp = 0x0ffff8
memory[0x0ffff8] = 42
```

Memory picture:

```text
Address      Content
---------    ----------------
0x0ffff8     42 stored here
0x100000     old top of stack
```

---

## 8. Post-index form: `[sp], #8`

Assembly:

```asm
ldr x1, [sp], #8
```

For post-index:

```text
access memory first, then change base
```

C-like pseudocode:

```c
x1 = memory[sp];
sp = sp + 8;
```

Slow table:

```text
Before:
sp = 0x0ffff8
memory[0x0ffff8] = 42
x1 = 0
```

After:

```text
x1 = 42
sp = 0x100000
```

So this behaves like a pop.

---

## 9. Push/pop walkthrough

Assembly:

```asm
movz x0, #42
str  x0, [sp, #-8]!
ldr  x1, [sp], #8
hlt  #0
```

Step table:

```text
Step   Instruction          sp before   sp after    x0   x1   Memory effect
----   ------------------   ---------   --------   --   --   --------------------------
1      movz x0,#42          0x100000    0x100000   42   0    none
2      str x0,[sp,#-8]!     0x100000    0x0ffff8   42   0    memory[0x0ffff8] = 42
3      ldr x1,[sp],#8       0x0ffff8    0x100000   42   42   read memory[0x0ffff8]
4      hlt #0               0x100000    0x100000   42   42   stop
```

This is the main v0.3 story.

---

## 10. Little-endian memory

A 64-bit value uses 8 bytes.

Example value:

```text
0x1122334455667788
```

In little-endian memory, the lowest byte comes first:

```text
Address      Byte
---------    ----
0x2000       88
0x2001       77
0x2002       66
0x2003       55
0x2004       44
0x2005       33
0x2006       22
0x2007       11
```

This feels reversed when you first see it.

Helpful rule:

```text
The full number is written big-looking: 0x1122334455667788
Memory stores the smallest byte first: 88 77 66 55 44 33 22 11
```

When you run:

```sh
./emulator dump examples/v0_3/memory_store_load.bin 0xffff8 8
```

you may see bytes in little-endian order.

---

## 11. 32-bit `W` registers vs 64-bit `X` registers

ARM64 has 64-bit registers like `x0`, `x1`, etc.

The lower 32 bits of each are named `w0`, `w1`, etc.

```text
x0 = full 64-bit register
w0 = lower 32 bits of x0
```

For v0.3:

```text
str x0 = store 8 bytes
str w0 = store 4 bytes
ldr x0 = load 8 bytes
ldr w0 = load 4 bytes and zero-extend into x0
```

Zero-extension means the upper 32 bits become zero.

Example:

```text
memory has 4-byte value: 0xffffffff
ldr w0, [address]
```

After:

```text
x0 = 0x00000000ffffffff
```

Not:

```text
x0 = 0xffffffffffffffff
```

---

## 12. `STP` and `LDP`: pair instructions

`STP` means “store pair.”

```asm
stp x29, x30, [sp, #-16]!
```

C-like pseudocode:

```c
sp = sp - 16;
memory[sp]     = x29;
memory[sp + 8] = x30;
```

`LDP` means “load pair.”

```asm
ldp x29, x30, [sp], #16
```

C-like pseudocode:

```c
x29 = memory[sp];
x30 = memory[sp + 8];
sp = sp + 16;
```

Why 16 bytes?

```text
x29 is 8 bytes
x30 is 8 bytes
8 + 8 = 16 bytes
```

---

## 13. Why `x29` and `x30` are often paired

Later, in v0.4, functions use `x30` as the return address register.

Many function examples save:

```asm
stp x29, x30, [sp, #-16]!
```

because:

```text
x29 = frame pointer conventionally
x30 = return address / link register
```

For v0.3, you do not need to fully understand function calls yet. Just understand that `STP` stores two registers and `LDP` restores them.

---

## 14. Full `STP` / `LDP` walkthrough

Assembly:

```asm
movz x29, #100
movz x30, #200
stp  x29, x30, [sp, #-16]!
movz x29, #0
movz x30, #0
ldp  x29, x30, [sp], #16
hlt  #0
```

Step table:

```text
Step   Instruction                 sp         x29   x30   What happens
----   -------------------------   --------   ---   ---   ------------------------------
1      movz x29,#100               0x100000   100   0     set x29
2      movz x30,#200               0x100000   100   200   set x30
3      stp x29,x30,[sp,#-16]!      0x0ffff0   100   200   store both on stack
4      movz x29,#0                 0x0ffff0   0     200   overwrite x29
5      movz x30,#0                 0x0ffff0   0     0     overwrite x30
6      ldp x29,x30,[sp],#16        0x100000   100   200   restore both
7      hlt                         0x100000   100   200   stop
```

Memory after `stp`:

```text
Address      Stored value
---------    ------------
0x0ffff0     old x29 = 100
0x0ffff8     old x30 = 200
```

## 15. SP vs XZR confusion

Register number 31 is special in ARM64.

Depending on context, it can mean:

```text
SP  = stack pointer
XZR = zero register
```

In v0.3 memory instructions:

```text
as a memory base: register 31 means SP
as a data register: register 31 means XZR/WZR
```

Example:

```asm
str x0, [sp, #-8]!
```

Here `sp` is the base address register.

But:

```asm
ldr xzr, [sp]
```

means:

```text
read memory, but discard the result
```

because `xzr` is the zero register.

---

## 16. Failure behavior

v0.3 must avoid partial mutation on failed memory operations.

Example:

```asm
str x0, [sp, #-8]!
```

If subtracting 8 from `sp` would create an invalid address, the emulator should fail without changing `sp`.

Why?

Because half-updated CPU state makes debugging and tests unreliable.

Safe behavior:

```text
check address first
if valid: update sp and store
if invalid: do not change sp, do not store
```

## 17. How v0.3 fits into emulator code

v0.3 makes `cpu_step()` need mutable memory.

Earlier versions could mostly read instruction bytes and update registers.

Now instructions can write to memory:

```c
memory_write64(memory, address, value, error, error_size);
```

Simplified `STR` execution:

```c
address = calculate_address(base, offset, mode);
value = read_register(rt);
memory_write64(memory, address, value);
update_base_if_needed();
pc += 4;
```

Simplified `LDR` execution:

```c
address = calculate_address(base, offset, mode);
value = memory_read64(memory, address);
write_register(rt, value);
update_base_if_needed();
pc += 4;
```

## 18. Common mistakes

### Mistake 1: mixing up `LDR` and `STR`

```text
LDR = memory -> register
STR = register -> memory
```

### Mistake 2: missing the `!`

```asm
str x0, [sp, #-8]
```

uses `sp - 8` but does not change `sp`.

```asm
str x0, [sp, #-8]!
```

changes `sp` to `sp - 8`.

### Mistake 3: confusing pre-index and post-index

```text
pre-index:  change base, then access memory
post-index: access memory, then change base
```

### Mistake 4: expecting memory bytes to match the written number left-to-right

Little-endian reverses byte order in memory dumps.

## 20. One-sentence summary

v0.3 teaches the emulator to move data between registers and memory: `STR` stores register values into memory, `LDR` loads memory into registers, stack push/pop changes `sp`, and `STP`/`LDP` store or load two 64-bit registers at once.
