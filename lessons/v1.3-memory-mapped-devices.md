# v1.3 Lesson — Memory-Mapped Devices

This guide is beginner-first. It continues the learning chain from
`lessons/v0.1-instruction-sandbox.md` through
`lessons/v1.2-virtual-memory.md`.

The short version:

```text
v1.3 starts teaching memory-mapped hardware devices.

Before v1.3, a data address either meant RAM or a memory fault.
After v1.3, a data address can mean:

RAM page        -> normal byte storage
MMIO device     -> a register on a tiny teaching device
invalid address -> deterministic fault
```

The CPU loop still looks familiar:

```text
fetch instruction at pc
  -> decode instruction
  -> execute instruction
  -> update registers, RAM, or device state
  -> stop on HLT, fake exit, breakpoint, or error
```

The new question is what happens when an instruction such as `STRB` or `LDR`
uses an address outside ordinary RAM:

```text
CPU data access -> bus route -> RAM mapping or device register -> result/fault
```

v1.3 does **not** turn the emulator into a full board emulator. There are no
interrupts, DMA engines, caches, memory barriers, PCI devices, device trees,
real terminal input, host wall-clock timers, or real entropy. The goal is a
small deterministic teaching platform that shows why some addresses behave like
hardware registers instead of normal memory.

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
v1.0: the whole learning path becomes a stable release surface
v1.1: the loader can load and inspect a tiny Mach-O executable profile
v1.2: loaded programs now get mapped pages and memory permissions
v1.3: data accesses can reach a tiny memory-mapped device bus
```

v1.3 is closest in spirit to v0.3, v0.7, and v1.2:

```text
v0.3 taught: load/store instructions calculate data addresses.
v0.7 taught: guest code can ask the emulator to print through SVC #0.
v1.2 taught: not every address is mapped, writable, readable, or executable.
v1.3 teaches: some data addresses are not RAM at all; they are device registers.
```

The important boundary is this:

```text
Instruction fetch still comes from executable RAM mappings.
Device ranges are only for data reads and writes.
```

That means a program may write a byte to the UART device address, but it may not
jump to the UART address and execute it as code.

## 2. The old mental model: RAM-only data access

In v1.2, the emulator asks mapping questions before memory access succeeds:

```text
Is the address inside the 1 MiB RAM byte array?
Is that RAM page mapped?
Does the mapping allow read, write, or execute?
```

For example:

```text
LDR W0, [X1]  -> needs readable RAM at the address in X1
STR W0, [X1]  -> needs writable RAM at the address in X1
fetch at pc   -> needs executable RAM at pc
```

That model is useful, but it is still incomplete for many low-level systems
ideas. Real machines often reserve some physical or virtual address ranges for
hardware. Reading or writing those addresses talks to a device instead of to
normal memory.

The classic example is a serial console:

```text
write byte 'A' to UART data register -> terminal displays A
```

No string buffer needs to exist at the UART address. The address is a control
surface, not storage.

---

## 3. The new mental model: a tiny bus

v1.3 adds a small bus between CPU data access and the final target.

Think of the bus as a routing table:

```text
address in RAM mapping range?     -> use RAM path
address in device range?          -> use device path
address in neither?               -> deterministic fault
address crosses a boundary?       -> deterministic fault before state changes
```

A bus route is atomic in this teaching model:

```text
Either the whole access is valid for one target,
or the whole access fails before RAM/device state changes.
```

That rule keeps edge cases understandable. A single multi-byte access should not
silently split across two devices, split across RAM and a device, or partly write
before failing.

---

## 4. The v1.3 platform map

The initial teaching platform uses fixed addresses:

```text
0x0000000000001000 - code / raw program area
0x0000000000010000 - common data area
0x0000000000080000 - stack area near top of 1 MiB RAM
0x0000000009000000 - UART device
0x0000000009010000 - timer device
0x0000000009020000 - random device
```

The device ranges are outside the 1 MiB RAM byte array on purpose. That makes the
lesson visible:

```text
0x0000000000001000 is RAM-backed program memory.
0x0000000009000000 is not RAM; it is a device register range.
```

Each device range is one teaching page in size:

```text
UART:   0x0000000009000000 - 0x0000000009001000
Timer:  0x0000000009010000 - 0x0000000009011000
Random: 0x0000000009020000 - 0x0000000009021000
```

The end address is exclusive, just like RAM mappings:

```text
0x09000000 is inside UART
0x09000fff is inside UART
0x09001000 is not inside UART
```

---

## 5. Device ranges are not RAM mappings

A RAM mapping describes bytes in the guest memory array:

```text
r-x 0x0000000000001000-0x0000000000002000 name=raw:program
rw- 0x00000000000f0000-0x0000000000100000 name=stack
```

A device range describes a register window:

```text
rw- 0x0000000009000000-0x0000000009001000 name=uart
rw- 0x0000000009010000-0x0000000009011000 name=timer
rw- 0x0000000009020000-0x0000000009021000 name=random
```

The permission letters still use the familiar `rwx` style, but their meaning is
slightly different for devices:

```text
r: the device has at least one readable register
w: the device has at least one writable register
x: devices are not executable in v1.3
```

Device-wide `rw-` does not mean every offset is both readable and writable. The
specific register still decides whether a particular access is valid.

For example:

```text
UART + 0x00 is writable as a byte.
UART + 0x04 is readable as a word.
Writing UART + 0x04 is still a register-policy fault.
```

## 6. UART: printing without SVC

The UART is the easiest device to see.

UART registers:

```text
UART + 0x00  DATA    write byte -> append one byte to emulator stdout
UART + 0x04  STATUS  read word  -> bit 0 means writable
```

To print `A`, guest code writes one byte to the data register:

```asm
movz x0, #0x0900, lsl #16   // x0 = 0x09000000
movz w1, #'A'
strb w1, [x0]
hlt #0
```

The important part is `strb`:

```text
STRB uses a data write path.
The bus sees 0x09000000.
The bus routes the write to UART DATA.
The UART appends one byte to stdout.
```

This is different from v0.7 fake syscalls:

```text
v0.7 SVC write:
  put fd/address/length in registers
  execute SVC #0
  emulator syscall handler copies bytes from RAM to host stdout

v1.3 UART write:
  put UART address in a register
  write a byte directly to the UART DATA register
  device handler appends that byte to host stdout
```

Both paths are intentionally supported. MMIO UART is an additional output path,
not a replacement for the toy syscall ABI.

## 7. Timer: deterministic ticks, not wall-clock time

The timer device is deterministic. It does not read the host machine's real
clock.

Timer registers:

```text
TIMER + 0x00  TICKS_LO  read word  -> low 32 bits of deterministic counter
TIMER + 0x04  TICKS_HI  read word  -> high 32 bits of deterministic counter
TIMER + 0x08  RESET     write word -> reset counter to 0
```

The current teaching behavior is:

```text
read TICKS_LO -> returns current low bits, then advances the counter
read TICKS_HI -> returns current high bits, then advances the counter
write RESET   -> sets the counter back to 0
```

This keeps tests and lessons repeatable. A learner can run the same program twice
and observe the same sequence.

Why not use the host clock?

```text
Real time makes tests flaky.
Real time makes beginner examples harder to reason about.
Real time suggests more hardware behavior than this version is trying to teach.
```

---

## 8. Random device: deterministic pseudo-random values

The random device is also deterministic.

Random registers:

```text
RANDOM + 0x00  VALUE  read word  -> next deterministic pseudo-random value
RANDOM + 0x04  SEED   write word -> set deterministic seed
```

The current teaching behavior is:

```text
read VALUE -> advance the pseudo-random generator and return the next word
write SEED -> replace the generator state
```

This is not cryptography and not host entropy. It is a predictable teaching
sequence. The point is to demonstrate that a device read can return a value that
is not stored in a RAM byte at that address.

---

## 9. Access width matters

RAM load/store instructions already have widths:

```text
STRB / LDRB style access -> 1 byte
STRH / LDRH style access -> 2 bytes
STR  W / LDR W          -> 4 bytes
STR  X / LDR X          -> 8 bytes
```

Device registers are stricter than ordinary RAM:

```text
UART DATA accepts byte writes.
UART STATUS accepts word reads.
Timer registers accept word reads/writes at documented offsets.
Random registers accept word reads/writes at documented offsets.
```

Unsupported widths fault deterministically.

Examples:

```text
write 1 byte to UART DATA    -> valid
write 4 bytes to UART DATA   -> unsupported-width or invalid-register fault
read 4 bytes from UART STATUS -> valid
read 1 byte from UART STATUS -> invalid-register fault
read 8 bytes from TIMER      -> unsupported-width fault
```

This models an important systems idea: hardware registers often have a required
access size.

---

## 10. Offset matters

A device range is page-sized for readability, but only a few offsets are real
registers.

For UART:

```text
0x09000000 -> DATA
0x09000004 -> STATUS
```

Most other offsets are invalid:

```text
0x09000001 -> not a documented register
0x09000008 -> not a documented register
0x09000ffc -> inside UART range, but not a documented UART register
```

Being inside the device page is not enough. The access must match a documented
register and width.

---

## 11. Boundary and overflow edge cases

The bus checks address arithmetic before performing the access.

Important edge cases:

```text
address + width overflows uint64_t     -> fault
access starts in device and exits it   -> fault
access starts in one device and hits another -> fault
access starts in RAM and reaches outside RAM -> RAM bounds fault
access starts in unmapped space        -> unmapped/bounds-style fault
```

The rule is intentionally conservative:

```text
No split accesses.
No partial device writes.
No partially updated RAM.
No hidden wraparound.
```

That is especially important for write-back addressing modes. If an MMIO access
faults, the CPU should not quietly pretend the device write happened.

## 12. What `info`, `maps`, and `map` show

`emulator info <program>` now prints both RAM mappings and device ranges.

A simplified view looks like this:

```text
mappings: 2
  [0] r-x 0x0000000000001000-0x0000000000002000 size=0x0000000000001000 name=raw:program
  [1] rw- 0x00000000000f0000-0x0000000000100000 size=0x0000000000010000 name=stack
      guard --- 0x00000000000ef000-0x00000000000f0000 size=0x0000000000001000 name=stack-guard
devices: 3
  [0] rw- 0x0000000009000000-0x0000000009001000 size=0x0000000000001000 name=uart
  [1] rw- 0x0000000009010000-0x0000000009011000 size=0x0000000000001000 name=timer
  [2] rw- 0x0000000009020000-0x0000000009021000 size=0x0000000000001000 name=random
```

The debugger command `maps` uses the same map printer.

The debugger command `map <address>` can identify either a RAM mapping or a
device range:

```text
map 0x1000      -> raw program mapping
map 0x9000000   -> uart device
map 0x9010004   -> timer device
map 0x9020000   -> random device
```

The debugger `mem` command still inspects ordinary readable RAM. It is not the
same as performing a CPU load from a device register. That distinction keeps the
teaching model explicit:

```text
memory inspection reads RAM bytes
load instruction can trigger device behavior
```

## 13. How v1.3 fits with v1.4

The next planned milestone is a toy kernel mode. A kernel needs a way to print
without depending on the fake syscall ABI from v0.7.

The UART gives that future kernel a hardware-style output path:

```text
toy kernel code -> STRB to UART DATA -> visible output
```

That is why v1.3 matters before v1.4. It creates the first tiny platform device
that a future toy kernel can use.

---

## 14. Summary

v1.3 adds one major idea:

```text
A data address can name RAM or a device register.
```

The emulator now has:

```text
RAM mappings with read/write/execute permissions
fixed device ranges outside RAM
a UART DATA register for visible output
a deterministic timer device
a deterministic random device
info/debugger map output that shows both RAM and devices
clear faults for invalid offsets, invalid widths, boundary crossings, and overflows
```

The project is still small by design. The point is not to emulate real hardware
perfectly. The point is to make the first hardware-device concept concrete:

```text
Some addresses store bytes.
Some addresses control devices.
The bus decides which is which.
```
