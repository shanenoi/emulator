## 1. The learning ladder so far

The emulator has grown in small layers.

v0.1 started with the smallest possible instruction sandbox: fetch one
instruction, decode it, change registers, and stop on `HLT`.

v0.2 added control flow. Branches made `pc` a value the guest program could
change intentionally, not just a counter that always moves forward by four
bytes.

v0.3 added memory and a stack. The emulator had to decide whether an address was
inside the guest memory array and whether a load or store should be allowed.

v0.4 added function calls. `BL` saved a return address in `x30`, `RET` moved the
`pc` back to a caller, and stack frames became a convention rather than magic.

v0.5 added the debugger REPL. The same CPU state became something learners could
inspect step by step.

v0.6 made traces and disassembly readable. The emulator could explain what it
was about to execute.

v0.7 added toy syscalls. `SVC #0` became the first controlled doorway from guest
code into emulator-provided services such as `write` and `exit`.

v0.8 added ELF64 loading. Programs no longer had to be one flat raw blob loaded
at one fixed address.

v0.9 showed how tiny freestanding C programs can target the teaching emulator
when the compiler profile is intentionally small.

v1.0 stabilized the project with release checks, deterministic fixtures, and
clear limits.

v1.1 added a small Mach-O teaching loader.

v1.2 replaced the old "one big byte array" mental model with mapped pages and
permissions.

v1.3 added memory-mapped devices. UART, timer, and random devices live outside
ordinary RAM, so CPU loads and stores can talk to a tiny platform bus.

v1.4 connects all of those ideas with one new control-flow concept:
**exceptions**.

An exception is not just an error message. It is a controlled transfer from the
instruction that had a problem to a handler that can inspect what happened and
decide what to do next.

## 2. Why exceptions come after devices

Before v1.4, the emulator already had several situations where normal execution
could not continue:

- fetching from an unmapped or non-executable page,
- decoding an unsupported instruction,
- reading or writing a protected page,
- touching a device register with the wrong width or direction,
- using an unsupported `SVC` immediate,
- hitting a deliberate trap instruction.

In older versions, most of these situations were terminal host-side runtime
errors. That was useful while the emulator was small, but it is not enough for a
kernel lesson.

A tiny kernel needs to observe faults and traps. It needs a place to run handler
code. It needs a saved context saying "this is what was happening when execution
left normal code."

v1.4 begins that model without trying to become a production ARM exception
implementation.

## 3. The old mental model: fatal runtime errors

The old model looked like this:

```text
fetch/decode/execute instruction
if something goes wrong:
    stop the emulator
    print a host-side error message
```

That is still the default when no exception vector is configured. This preserves
older lesson behavior and keeps invalid beginner programs easy to understand.

For example, an invalid instruction without a configured vector is still a
program error, not a mysterious jump into nowhere.

## 4. The new mental model: exceptional control flow

The v1.4 model adds an optional second path:

```text
fetch/decode/execute instruction
if a catchable exception happens and a vector is configured:
    save exception context
    place useful context values in x0-x3
    jump pc to the exception vector
    let handler code run
    return with eret
```

The important difference is that the emulator does not simply stop. It captures
why the normal path was interrupted.

The first v1.4 implementation uses a single vector base. Every catchable
exception enters at the same address. A later version can turn this into a vector
table with cause-specific slots.

## 5. What v1.4 intentionally does not emulate

Real AArch64 exception handling includes exception levels, many system
registers, syndrome registers, interrupt masks, stack selection rules, vector
table offsets, and a full architectural `ERET` path.

v1.4 does not try to teach all of that at once.

It intentionally does not implement:

- full `EL0`/`EL1`/`EL2`/`EL3`,
- real `VBAR_ELx`, `ESR_ELx`, `ELR_ELx`, or `SPSR_ELx`,
- real `DAIF` interrupt masking semantics,
- a GIC or production interrupt controller,
- host asynchronous signal delivery,
- preemptive scheduling,
- nested exception levels,
- process isolation,
- a real OS ABI.

The goal is smaller: teach the shape of exceptional control flow before teaching
the architecture-accurate machinery.

## 6. The v1.4 exception controller

The emulator now has an exception controller inside `Emulator`.

It tracks:

```text
vector_configured        whether catchable exceptions may enter a handler
vector_base              the handler entry address
active                   whether a handler is currently running
interrupts_enabled       whether pending interrupts may be sampled
pending_timer_interrupt  whether a timer interrupt is waiting to enter
timer_interval           optional deterministic instruction-count interval
next_timer_deadline      next instruction-count deadline
context                  saved exception context
```

This state belongs to the emulator, not the raw `Cpu`, because exception entry is
more than normal instruction execution. It needs CPU state, memory permissions,
the deterministic timer policy, and guest-visible conventions.

## 7. Configuring a vector

An exception vector is the address where handler code begins.

There are three ways to configure the vector. The host C API is:

```c
bool emulator_configure_exception_vector(Emulator *emu,
                                         uint64_t vector_base,
                                         char *error,
                                         size_t error_size);
```

The vector address must be:

- 4-byte aligned,
- inside ordinary executable guest RAM,
- not inside a device range,
- readable as an instruction fetch target by the v1.2 permission model.

The CLI can also configure the vector before running a fixture:

```sh
./emulator trace examples/v1_4/cli_handled_brk.bin --exception-vector 0x1080
```

Guest code can configure the vector through the exception-controller MMIO device
at `0x09030000`:

```text
0x09030000 + 0x00  VECTOR          64-bit vector base
0x09030000 + 0x08  CONTROL         bit 0 vector enable, bit 1 interrupts enable
0x09030000 + 0x10  TIMER_INTERVAL  64-bit instruction-count interval
0x09030000 + 0x18  PENDING         32-bit timer pending flag
```

This means v1.4 reuses v1.2 and v1.3 instead of bypassing them. An exception
handler is still guest code. It must live in executable guest memory.

## 8. The saved exception context

When a catchable exception enters the vector, the emulator stores this context:

```text
cause
fault_address
interrupted_pc
resume_pc
flags
depth
```

`cause` is a stable teaching code.

`fault_address` is the relevant address for memory, fetch, or device faults. For
trap-like exceptions, it can hold the immediate value or zero.

`interrupted_pc` is where normal execution was interrupted.

`resume_pc` is where simplified `ERET` should continue.

`flags` stores the previous NZCV flags.

`depth` is currently one for an active exception and zero after returning. Nested
exceptions are rejected as a deterministic double fault.

## 9. Cause codes

The v1.4 cause codes are intentionally small and readable:

```text
0x01  INVALID_INSTRUCTION
0x02  BREAKPOINT_OR_TRAP
0x03  SVC_TRAP
0x10  FETCH_FAULT
0x11  READ_FAULT
0x12  WRITE_FAULT
0x13  EXEC_PERMISSION_FAULT
0x14  READ_PERMISSION_FAULT
0x15  WRITE_PERMISSION_FAULT
0x20  DEVICE_FAULT
0x30  DIVIDE_BY_ZERO
0x40  TIMER_INTERRUPT
```

Most causes now have a runtime path, while some remain reserved vocabulary for future lessons. The enum gives tests, examples, and future implementation work stable names.

## 10. Handler entry convention

When the emulator enters a handler, it sets:

```text
pc = vector_base
x0 = cause
x1 = fault_address
x2 = interrupted_pc
x3 = resume_pc
```

This is a teaching convention. It is not the real AArch64 exception ABI.

The point is to make early handler examples simple. A handler can inspect `x0`
to see why it ran, inspect `x2` to see where the interrupted code was, and use
`ERET` to return to `x3`.

The full saved context is also available through the C API:

```c
const EmuExceptionContext *emulator_get_exception_context(const Emulator *emu);
```

That API is useful for unit tests and debugger features.

## 11. Returning with simplified `ERET`

v1.4 decodes the AArch64 `ERET` opcode as a teaching return instruction.

When `ERET` runs inside an active exception:

1. the emulator reads the current `x3` as the return target,
2. validates that target as aligned and executable,
3. saves it back as the effective `resume_pc`,
4. restores saved flags,
5. sets `pc` to that target,
6. marks the exception inactive,
7. re-enables simplified interrupts.

If `ERET` runs without an active exception, execution stops with a clear error.

This makes `ERET` similar to `RET` in v0.4 from a teaching perspective: both are
control-flow returns, but they return from different kinds of calls.

`RET` returns from a function call.

`ERET` returns from exceptional control flow.

## 12. Resume policy

Different causes naturally resume differently.

For faults such as invalid instruction, fetch fault, read fault, write fault, and
permission fault, v1.4 records the faulting instruction as the default resume
target. Returning immediately will try the instruction again. A handler that wants
to skip the faulting instruction can add 4 to `x3` before `ERET`; a handler that
fixes a mapping or patches code can leave `x3` unchanged and retry.

For deliberate traps such as `BRK` and unsupported nonzero `SVC` traps, the
resume target is the next instruction. That is useful for handlers that treat the
trap as a call-like event.

For timer interrupts, the resume target is the current `pc`, because the
interrupt happens between instructions.

This distinction is one of the most important exception lessons: some exceptions
mean "retry this," while others mean "continue after this."

## 13. Catchable and non-catchable failures

v1.4 separates guest-visible exceptions from emulator bugs.

Catchable examples include:

- invalid instruction decode,
- `BRK` trap,
- unsupported nonzero `SVC` when a vector is configured,
- instruction fetch fault,
- data read/write fault,
- page permission fault,
- device access fault,
- queued timer interrupt.

Non-catchable examples include:

- invalid internal emulator state,
- exception entry when the configured vector is no longer executable,
- an exception while another exception is already active,
- `ERET` without active exception,
- invalid return target during exception return.

Those remain terminal host-side errors because a tiny teaching kernel should not
paper over emulator invariants.

## 14. `SVC #0` vs `SVC #imm`

v0.7 introduced fake syscalls through `SVC #0`.

That behavior stays intact. `SVC #0` still enters the emulator syscall
dispatcher for toy `write` and `exit`.

v1.4 treats nonzero `SVC` immediates as a possible trap source. If an exception
vector is configured, unsupported `SVC #imm` can enter the handler with cause
`SVC_TRAP` and resume at the next instruction.

If no vector is configured, the old fatal error behavior is preserved.

This preserves old examples while opening a path for toy kernel experiments.

## 15. `BRK` as an explicit trap

`BRK #imm` is decoded as a deliberate trap instruction.

With a vector configured, it enters the exception handler with cause
`BREAKPOINT_OR_TRAP`. The immediate value is placed in the fault-address/context
slot for simple inspection.

Without a vector, it is a runtime error. A later test suite can pin the exact CLI
text.

The debugger's ordinary breakpoint list is still separate from guest `BRK`.
Debugger breakpoints stop the REPL before executing an instruction. Guest `BRK`
is an instruction the guest program actually executes.

## 16. Memory faults reuse v1.2 permissions

v1.2 introduced mapped pages and permissions.

v1.4 does not create a new memory safety system. It asks the existing memory
layer whether an access is valid.

That matters because the same address can fail for different reasons:

```text
unmapped address       -> read/write/fetch fault
mapped r-- page write  -> write-permission fault
mapped rw- page fetch  -> exec-permission fault
device bad width       -> device fault
```

The exception model is only useful if it preserves these distinctions clearly.

## 17. Device faults reuse v1.3 device policy

v1.3 taught that devices are not RAM.

A device can reject an access because:

- the width is unsupported,
- the offset is not a valid register,
- the access crosses a device boundary,
- the guest reads a write-only register,
- the guest writes a read-only register,
- the guest tries to fetch instructions from a device range.

v1.4 keeps those rules. The difference is that a configured exception vector can
turn selected device access failures into a catchable `DEVICE_FAULT`.

This is the bridge from "the emulator says no" to "a tiny platform handler sees
what went wrong."

## 18. Timer interrupt skeleton

v1.3's timer device remains deterministic and advances when the guest reads timer registers.

v1.4 adds a separate deterministic instruction-count timer interrupt. It does not use host wall-clock time.

The C API includes:

```c
void emulator_configure_timer_interrupt(Emulator *emu, uint64_t interval);
bool emulator_queue_timer_interrupt(Emulator *emu);
void emulator_set_interrupts_enabled(Emulator *emu, bool enabled);
```

An interrupt is sampled after an instruction retires and before the next
instruction fetch.

That boundary is deliberate. It means an interrupt does not split an instruction
in half. A load either happens or it does not. A store either happens or it does
not. Then the emulator may enter an interrupt handler.

This is not a real interrupt controller. It is a deterministic teaching hook for future toy-kernel examples and can also be configured by guest code through the exception-controller MMIO device.

## 19. Slow walkthrough: a `BRK` trap

Imagine guest code reaches a `BRK #7` instruction at `0x1040`, and the vector is
configured to `0x2000`.

Before execution:

```text
pc = 0x1040
vector_base = 0x2000
exception_active = no
```

The emulator decodes `BRK #7`.

It creates this context:

```text
cause = BREAKPOINT_OR_TRAP
fault_address = 7
interrupted_pc = 0x1040
resume_pc = 0x1044
flags = old NZCV
depth = 1
```

It also sets registers for the handler:

```text
x0 = cause
x1 = 7
x2 = 0x1040
x3 = 0x1044
pc = 0x2000
```

The handler runs at `0x2000`.

When the handler executes `ERET`, the emulator sets:

```text
pc = 0x1044
exception_active = no
```

Normal execution resumes after the trap instruction.

## 20. Slow walkthrough: a read fault

Imagine code tries to load from an unmapped address at instruction `0x1100`.

For a fault, the resume policy is different:

```text
interrupted_pc = 0x1100
resume_pc = 0x1100
```

Returning would retry the same load.

That is useful if the handler can make the address valid. A tiny handler can also skip this specific fault by adding 4 to `x3` and then executing `ERET`. The `mmio_skip_device_fault.bin` example uses that trick to skip a read from the write-only UART DATA register.

## 21. Slow walkthrough: a timer interrupt

A timer interrupt is not caused by the current instruction failing.

Instead, the emulator samples pending interrupts between instructions:

```text
instruction retires
update deterministic timer interrupt state
if pending and enabled:
    enter vector
else:
    fetch next instruction
```

For timer interrupts:

```text
interrupted_pc = current pc
resume_pc = current pc
fault_address = 0
cause = TIMER_INTERRUPT
```

Returning resumes exactly where the next instruction would have been fetched.

## 22. Debugging exception state

The debugger now has an `exception` command.

It prints:

```text
exception_active
vector_configured
vector_base
interrupts_enabled
pending_timer_interrupt
cause
fault_address
interrupted_pc
resume_pc
saved_nzcv
depth
```

This follows the v0.5 debugger lesson pattern: when a concept becomes important,
make it inspectable.

The `regs` command still shows CPU registers. The `exception` command shows the
separate exception controller state.

## 23. Trace behavior

Trace prints the instruction at the current `pc` before execution, and v1.4 now adds explicit exception event lines.

That makes exception entry and return visible directly:

```text
trace pc=0x0000000000001000 ... brk #0x14
trace exception-enter cause=breakpoint-or-trap(0x02) ... vector=0x0000000000001080
trace pc=0x0000000000001080 ... eret
trace exception-return cause=breakpoint-or-trap(0x02) resume_pc=0x0000000000001004
```

## 24. What changed inside the emulator

The main implementation changes are:

- new `EmuExceptionCause`, `EmuExceptionContext`, and `EmuExceptionController`
  types,
- an exception controller field inside `Emulator`,
- public APIs to configure a vector, raise exceptions, return from exceptions,
  inspect context, enable/disable simplified interrupts, and queue/configure a
  timer interrupt,
- decode support for `BRK #imm`,
- decode support for simplified `ERET`,
- emulator-level dispatch for `BRK`, `ERET`, nonzero `SVC`, fetch faults, decode
  faults, selected memory faults, selected device faults, and pending timer
  interrupts,
- a guest-visible exception-controller MMIO device at `0x09030000`,
- debugger inspection through the `exception` command,
- explicit trace lines for exception entry and return,
- documentation for the teaching model.

The raw CPU stepper still handles ordinary instructions. The emulator wrapper is
responsible for exceptional control flow.

That separation is important. A CPU can add and branch. An emulator platform can
decide how faults, syscalls, and device interrupts are handled.

## 25. What tests should cover later

Tests are intentionally not added in this development slice yet, but the test
plan already describes the coverage expected next.

Future tests should prove:

- old behavior is preserved when no vector is configured,
- vector configuration rejects bad addresses,
- `BRK` enters the vector and `ERET` returns to the next instruction,
- invalid instruction entry saves the faulting `pc`,
- memory read/write/fetch failures become the right causes,
- device access failures become `DEVICE_FAULT`,
- nonzero `SVC` becomes `SVC_TRAP` only when a vector is configured,
- `SVC #0` fake syscalls still work,
- `ERET` without an active exception fails clearly,
- nested exceptions produce a deterministic double-fault-style error,
- timer interrupts are sampled only at the documented boundary,
- debugger `exception` output is stable.

## 26. Common mistakes

### Mistake 1: thinking exceptions are only errors

Some exceptions are errors, but traps and interrupts can be intentional control
flow.

A `BRK` trap can be used as a deliberate doorway into a handler.

A timer interrupt can run even when the interrupted instruction succeeded.

### Mistake 2: thinking the vector is a data structure

In v1.4, `vector_base` is an instruction address. The emulator
sets `pc` there.

A later vector table could contain many slots, but the first model is one common
entry point.

### Mistake 3: thinking `ERET` is just `RET`

`RET` reads a register target and returns from a function call.

`ERET` uses the saved exception context and returns from an exception.

They are both returns, but they return from different control-flow mechanisms.

### Mistake 4: forgetting that handlers are guest code

The vector must point to executable guest memory. It cannot point to a host C
function, a device range, or unmapped memory.

### Mistake 5: expecting real ARM exception levels

v1.4 is a teaching model. It prepares the mental model for real ARM exception
levels, but it does not implement them yet.

### Mistake 6: expecting interrupts to happen at random times

The interrupt skeleton is deterministic. Interrupts are sampled at a documented
boundary, not delivered by host wall-clock timing.

## 29. One-sentence summary

v1.4 turns selected guest faults, traps, and deterministic pending interrupts
from terminal runtime errors into inspectable, teachable control-flow events with
a saved context, a vector entry point, and a simplified `ERET` return path.
