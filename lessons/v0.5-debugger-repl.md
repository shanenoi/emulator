# v0.5 Lesson — Debugger REPL

This guide is beginner-first. It explains what the debugger is for and how each command maps to emulator state.

v0.5 adds an interactive debugger:

```sh
./emulator debug <raw-binary>
```

The debugger lets you pause, inspect, and control the emulator.

---

## 1. Why we need a debugger

Without the debugger, you run a program and only see the final result:

```sh
./emulator run examples/v0_1/add.bin
```

That tells you the end state, but not how the emulator got there.

A debugger lets you ask questions during execution:

```text
What is pc right now?
What are x0, x1, x2?
What bytes are in memory?
What happens if I execute one instruction?
Can I stop before a specific instruction?
```

This is especially useful for v0.2–v0.4 because control flow, stack, and functions are easier to understand step by step.

---

## 2. The core debugger mental model

The debugger controls the same emulator you already know:

```text
+-----------------------------+
| Debugger REPL               |
|                             |
| commands: run, step, regs   |
|                             |
| controls                    |
|       v                     |
| Emulator                    |
|  - CPU registers            |
|  - memory                   |
|  - pc                       |
|  - breakpoints              |
+-----------------------------+
```

The debugger does not change what instructions mean.

It just gives you control over when execution happens.

---

## 3. The three most important commands

The most confusing part is usually the difference between:

```text
run
step
continue
```

Here is the simple version:

```text
run       = restart program from the beginning
step      = execute exactly one instruction
continue  = keep executing from current pc until halt/error/breakpoint
```

### `run`

`run` resets CPU and memory to the initially loaded program state.

Use it when you want to start over.

```text
run = reset + execute
```

### `step`

`step` executes one instruction at the current `pc`.

Use it when you want to learn slowly.

```text
step = execute one instruction only
```

### `continue`

`continue` resumes execution from the current state.

Use it after hitting a breakpoint or after stepping manually.

```text
continue = keep going from here
```

## 4. The debugger prompt

When you run:

```sh
./emulator debug examples/v0_1/add.bin
```

you get a prompt like:

```text
emu>
```

You can type commands:

```text
emu> regs
emu> step
emu> regs
emu> quit
```

The debugger can also read commands from stdin, which makes it testable and scriptable:

```sh
printf 'step\nregs\nquit\n' | ./emulator debug examples/v0_1/add.bin
```

---

## 5. `regs`: inspect CPU registers

Command:

```text
regs
```

This prints the same stable CPU dump used by `run`.

You use `regs` to answer:

```text
What is x0?
What is pc?
What is sp?
How many instructions executed?
```

Example learning workflow:

```text
emu> regs
emu> step
emu> regs
```

This shows what changed after one instruction.

---

## 6. `step`: execute one instruction

Suppose the program is:

```asm
movz x0, #2
movz x1, #3
add  x2, x0, #3
hlt  #0
```

Initial state:

```text
pc = 0x1000
x0 = 0
x1 = 0
x2 = 0
```

Debugger session:

```text
emu> step
stepped pc=0x0000000000001000
emu> regs
```

After first step:

```text
pc = 0x1004
x0 = 2
x1 = 0
x2 = 0
```

Another step:

```text
emu> step
emu> regs
```

After second step:

```text
pc = 0x1008
x0 = 2
x1 = 3
x2 = 0
```

This is the most beginner-friendly way to understand emulator execution.

---

## 7. Breakpoints

A breakpoint means:

```text
Stop when pc reaches this address.
```

Command:

```text
break 0x1008
```

or alias:

```text
b 0x1008
```

If execution reaches address `0x1008`, the debugger stops **before** executing the instruction there.

This is important.

```text
breakpoint stops before instruction executes
```

---

## 8. Slow breakpoint walkthrough

Program:

```asm
movz x0, #2       // 0x1000
movz x1, #3       // 0x1004
add  x2, x0, #3   // 0x1008
hlt  #0           // 0x100c
```

Debugger commands:

```text
break 0x1008
run
regs
continue
regs
quit
```

What happens:

```text
1. break 0x1008 sets a breakpoint at the ADD instruction.
2. run starts from the beginning.
3. emulator executes 0x1000 and 0x1004.
4. pc becomes 0x1008.
5. debugger stops before ADD executes.
```

At the breakpoint:

```text
x0 = 2
x1 = 3
x2 = 0
pc = 0x1008
```

Why is `x2` still zero?

Because the `add` at `0x1008` has not executed yet.

Then:

```text
continue
```

continues past the breakpoint and eventually halts.

Final state:

```text
x0 = 2
x1 = 3
x2 = 5
```

---

## 9. Why `continue` after a breakpoint makes progress

This is subtle.

If you stop at a breakpoint at `0x1008`, and then run `continue`, the debugger must not immediately stop at the same breakpoint again forever.

So the debugger skips the current breakpoint once, executes the instruction, then resumes normal breakpoint checking.

Mental model:

```text
At breakpoint pc=0x1008
continue:
    allow this exact instruction to execute once
    then resume normal breakpoint behavior
```

---

## 10. `run` vs `continue` vs `step`

Here is the clearest comparison:

```text
Command     Starts over?   Executes how much?              Stops when?
-------     ------------   -----------------------------   ----------------------
run         yes            many instructions               halt/error/breakpoint
continue    no             many instructions from here      halt/error/breakpoint
step        no             exactly one instruction          immediately after step
```

Use `run` when:

```text
you want to restart from the beginning
```

Use `continue` when:

```text
you are paused and want to keep going
```

Use `step` when:

```text
you want to see one instruction's effect
```

---

## 11. `mem` / `x`: inspect memory

Commands:

```text
mem <address> <length>
x   <address> <length>
```

Example:

```text
mem 0xffff8 8
```

This dumps 8 bytes starting at address `0xffff8`.

Use this with v0.3 stack examples:

```text
emu> break 0x1008
emu> run
emu> mem 0xffff8 8
```

This lets you see what `STR` or `STP` wrote into memory.

Remember little-endian:

```text
value 42 stored as 64-bit = 2a 00 00 00 00 00 00 00
```

---

## 12. `trace on` and `trace off`

Trace prints each `pc` before execution.

Commands:

```text
trace on
run
trace off
```

Example output:

```text
trace pc=0x0000000000001000
trace pc=0x0000000000001004
trace pc=0x0000000000001008
```

Trace helps you understand control flow:

```text
straight-line program: pc increases by 4
loop: pc jumps backward
function call: pc jumps to function, then ret jumps back
```

---

## 13. `breakpoints` and `delete`

List breakpoints:

```text
breakpoints
```

Delete by ID or address:

```text
delete 1
delete 0x1008
```

Why IDs exist:

```text
Breakpoints are stored in a list.
Each one gets a small numeric id so it is easy to remove.
```

## 15. Debugging a loop

For a loop, trace is helpful.

Example v0.2 loop:

```asm
movz x0, #3
loop:
    sub  x0, x0, #1
    cbnz x0, loop
hlt #0
```

Debugger commands:

```text
trace on
run
quit
```

You should see addresses repeat:

```text
0x1004
0x1008
0x1004
0x1008
0x1004
0x1008
```

That repeated pattern is the loop.

---

## 16. Debugging a function call

For a function, set a breakpoint at the function address.

Example:

```asm
movz x0, #2       // 0x1000
bl add_three      // 0x1004
hlt #0            // 0x1008
add_three:        // 0x100c
    add x0,x0,#3
    ret
```

Commands:

```text
break 0x100c
run
regs
step
regs
continue
quit
```

At the breakpoint:

```text
pc = 0x100c
x30 = 0x1008
```

That proves `BL` saved the return address.

After `ret`, execution returns to `0x1008`.

---

## 17. Error recovery

The debugger should stay alive after runtime errors when possible.

Example:

```text
emu> run
error: unsupported instruction ...
emu> regs
emu> run
```

This is useful because errors are learning moments. You can inspect the state that caused the problem.

---

## 18. Strict command arguments

v0.5 rejects extra arguments.

Good:

```text
regs
trace on
break 0x1008
mem 0x1000 16
```

Bad:

```text
regs extra
trace on extra
break 0x1008 extra
mem 0x1000 16 extra
```

This strictness matters because debugger scripts should be predictable.

---

## 19. How v0.5 fits into emulator code

The debugger wraps the existing emulator.

Simplified idea:

```c
while (read_command(line)) {
    if (command == "step") {
        cpu_step(&cpu, &memory);
    } else if (command == "continue") {
        run_until_halt_error_or_breakpoint();
    } else if (command == "regs") {
        cpu_dump(&cpu);
    } else if (command == "break") {
        add_breakpoint(address);
    }
}
```

The debugger does not replace the emulator. It controls it.

## 20. Recommended learning workflow

For any example program, use this order:

```text
1. run the program normally
2. run it in debug mode
3. use regs before execution
4. step one instruction
5. use regs again
6. set a breakpoint near the interesting instruction
7. run
8. inspect regs/memory at the breakpoint
9. continue to the end
```

Example:

```sh
./emulator debug examples/v0_1/add.bin
```

Commands:

```text
regs
step
regs
step
regs
break 0x1008
run
regs
continue
regs
quit
```

## 21. Common mistakes

### Mistake 1: using `run` when you meant `continue`

`run` restarts from the beginning.

If you are paused and want to keep going, use:

```text
continue
```

### Mistake 2: expecting breakpoint instruction to already execute

Breakpoints stop before execution.

If you break at the `add` instruction, the destination register has not changed yet.

### Mistake 3: forgetting that `step` only executes one instruction

If the program has a loop or function, `step` does not “finish” it. It only moves one instruction.

### Mistake 4: using memory inspection before the store happened

If you inspect memory before `STR` executes, the data will not be there yet.

Use a breakpoint after the store, or step through the store first.

## 23. One-sentence summary

v0.5 adds a scriptable debugger that controls the existing emulator: `step` runs one instruction, `continue` resumes from the current state, `run` restarts from the beginning, breakpoints stop before an instruction executes, and `regs`/`mem` let you inspect CPU and memory state while learning.
