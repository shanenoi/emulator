# v0.8 Lesson — ELF64 Loader

This guide is beginner-first. It continues the learning chain from v0.1 through v0.7 and explains how the emulator moves from raw instruction bytes to a small real executable-file format.

In v0.7, guest programs learned to ask the emulator for tiny outside-world services:

```text
write bytes to stdout
write bytes to stderr
exit with a status code
```

But those programs were still loaded as raw `.bin` files.

v0.8 adds one new ability:

```text
The emulator can load a simple static AArch64 ELF64 executable.
```

Before v0.8, the loader always did this:

```text
copy the whole file into memory at 0x1000
set pc = 0x1000
set sp = top of memory
```

After v0.8, the loader can also do this:

```text
notice that the file is ELF
validate the ELF header
read program headers
copy each PT_LOAD segment to its guest address
zero-fill .bss bytes
set pc = e_entry
set sp = top of memory
```

The important shift is:

```text
raw .bin: file position decides where bytes go
ELF:      program headers decide where bytes go
```

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the emulator can load simple ELF64 executables
```

v0.8 does **not** make the CPU smarter.

It mostly makes the **loader** smarter.

The same CPU still executes the same supported instructions. The same fake syscalls still use:

```text
write = 64
exit  = 93
```

What changes is how the program bytes arrive in memory and where execution begins.

---

## 2. Why raw `.bin` files were simpler

A raw `.bin` file is just bytes.

It has no header.
It has no entry point.
It has no segment table.
It has no metadata saying what is code or data.

So the old loader had to make a simple rule:

```text
Put every byte at 0x1000.
Start pc at 0x1000.
```

That worked well for early lessons because it kept the mental model small:

```text
file byte 0 -> memory[0x1000]
file byte 1 -> memory[0x1001]
file byte 2 -> memory[0x1002]
...
```

A raw v0.7 hello program could place the message after the instructions and use a fixed address like `0x1020`:

```asm
movz x1, #0x1020   // address of message inside the raw .bin image
```

That is good for teaching, but it is not how normal executable files describe themselves.

---

## 3. What ELF adds

ELF means **Executable and Linkable Format**.

For v0.8, you do not need to understand the entire ELF specification. You only need this mental model:

```text
An ELF file is a container.

It contains:
1. an ELF header
2. a program-header table
3. bytes for loadable segments
```

The ELF header answers questions like:

```text
Is this really an ELF file?
Is it 64-bit?
Is it little-endian?
Is it for AArch64?
Where is the program-header table?
Where should execution begin?
```

The program headers answer questions like:

```text
Which bytes should be loaded into guest memory?
Where should those bytes be placed?
How much memory should the segment occupy?
Is the segment intended to be readable, writable, or executable?
```

This is the new loader-side mental model:

```text
ELF file
  |
  | read ELF header
  v
program-header table
  |
  | for each PT_LOAD segment
  v
guest memory
  |
  | set pc = e_entry
  v
CPU starts executing
```

---

## 4. The first rule: detect ELF by magic bytes

The loader decides between raw mode and ELF mode by looking at the first four bytes.

ELF files start with:

```text
7f 45 4c 46
```

As text, that is:

```text
\x7f E L F
```

v0.8 uses this rule:

```text
if file starts with \x7fELF:
    use the ELF loader
else:
    use the old raw-binary loader
```

This keeps v0.1 through v0.7 behavior compatible.

A tiny raw `.bin` file that is not ELF still loads at `0x1000` exactly like before.

## 5. Start with the visible result

Build and run the v0.8 hello ELF example:

```sh
make examples/v0_8/hello_elf.elf
./emulator run examples/v0_8/hello_elf.elf
```

Expected output:

```text
hello, v0.8!
```

That looks similar to the v0.7 hello program, but the file format is different.

v0.7 hello was a raw `.bin`:

```text
all bytes copied to 0x1000
pc starts at 0x1000
```

v0.8 hello is an ELF executable:

```text
ELF header says where the program headers are
program headers say where text/data segments go
e_entry says where pc starts
```

The fake syscall behavior is the same. The loading path is different.

---

## 6. The v0.8 hello assembly

The important assembly still looks familiar from v0.7:

```asm
.text
.global _start

_start:
    movz x0, #1              // fd = stdout
    movz x1, #0x2000         // message lives in the ELF data segment
    movz x2, #13             // length
    movz x8, #64             // fake write
    svc  #0
    movz x0, #0              // exit status
    movz x8, #93             // fake exit
    svc  #0

.data
message:
    .ascii "hello, v0.8!\n"
```

The syscall part is still v0.7 knowledge:

| Register | Meaning |
|---|---|
| `x8` | syscall number |
| `x0` | first argument / return value |
| `x1` | second argument |
| `x2` | third argument |

Before `SVC #0`, the first syscall has:

```text
x8 = 64       write
x0 = 1        stdout
x1 = 0x2000   guest address of message bytes
x2 = 13       number of bytes
```

The new v0.8 idea is why `0x2000` is a good address.

The linker script places `.data` at `0x2000`, and the ELF program header tells the loader to copy that data segment there.

## 7. Raw layout vs ELF layout

A raw `.bin` example is simple:

```text
memory
0x1000: instruction
0x1004: instruction
0x1008: instruction
...
0x1020: message bytes
```

Everything comes from one flat file image.

An ELF example can have a more explicit layout:

```text
memory
0x1000: .text instructions
0x1004: .text instructions
0x1008: .text instructions
...
0x2000: .data message bytes
```

The file itself might not store those bytes next to each other in the same way. The ELF program headers tell the loader how to place them.

Think of an ELF file like a moving box with labels:

```text
box item: bytes from file offset 0x1000
label:    put these at guest address 0x1000

box item: bytes from file offset 0x2000
label:    put these at guest address 0x2000
```

The loader follows the labels.

---

## 8. The two most important ELF concepts in v0.8

### Concept 1: `e_entry`

`e_entry` is the address where the CPU should start executing.

For raw files, the emulator decides:

```text
pc = 0x1000
```

For ELF files, the ELF header decides:

```text
pc = e_entry
```

In the v0.8 examples, `e_entry` is usually `0x1000`, but it does not have to be. The important behavior is that ELF owns the entry-point decision.

Because AArch64 instructions are 4 bytes wide, v0.8 also requires `e_entry` to be 4-byte aligned:

```text
accepted: e_entry = 0x1000
rejected: e_entry = 0x1001
```

That gives a clearer loader error instead of waiting for instruction fetch to fail later.

### Concept 2: `PT_LOAD`

`PT_LOAD` means:

```text
This program-header entry describes bytes that should be loaded into memory.
```

A simplified `PT_LOAD` entry says:

```text
p_offset = where bytes start in the ELF file
p_vaddr  = where bytes go in guest memory
p_filesz = how many bytes to copy from the file
p_memsz  = how many bytes the segment occupies in memory
p_flags  = intended permissions, like read/write/execute
```

The core loading action is:

```text
copy file[p_offset .. p_offset + p_filesz)
into memory[p_vaddr .. p_vaddr + p_filesz)
```

Then, if `p_memsz` is larger than `p_filesz`, the remaining bytes become zero.

That is how `.bss` works.

## 9. Slow walkthrough: loading a simple ELF

Imagine an ELF file with two loadable segments:

```text
text segment:
  p_offset = 0x1000
  p_vaddr  = 0x1000
  p_filesz = 0x20
  p_memsz  = 0x20
  flags    = R | X

data segment:
  p_offset = 0x2000
  p_vaddr  = 0x2000
  p_filesz = 0x0d
  p_memsz  = 0x0d
  flags    = R | W

e_entry = 0x1000
```

The loader does this:

```text
1. Check the magic bytes: yes, this is ELF.
2. Check ELF class: yes, ELF64.
3. Check endian: yes, little-endian.
4. Check machine: yes, AArch64.
5. Check type: yes, ET_EXEC.
6. Read the program-header table.
7. Find the text PT_LOAD segment.
8. Copy text bytes to memory at 0x1000.
9. Find the data PT_LOAD segment.
10. Copy message bytes to memory at 0x2000.
11. Check e_entry is mapped and 4-byte aligned.
12. Set pc = e_entry = 0x1000.
13. Set sp = 0x100000.
```

After loading, the emulator state is ready:

```text
pc = 0x1000
sp = 0x100000
memory[0x1000...] = instructions
memory[0x2000...] = "hello, v0.8!\n"
```

Then normal fetch/decode/execute begins.

## 10. Slow walkthrough: executing the ELF hello program

Once loading is done, execution is not special.

The CPU starts at `pc = 0x1000` and runs instructions just like a raw binary.

Trace mode shows this clearly:

```sh
./emulator trace examples/v0_8/hello_elf.elf
```

The early trace lines look like normal decoded instructions:

```text
trace pc=0x0000000000001000 ... movz x0, #0x1
trace pc=0x0000000000001004 ... movz x1, #0x2000
trace pc=0x0000000000001008 ... movz x2, #0xd
trace pc=0x000000000000100c ... movz x8, #0x40
trace pc=0x0000000000001010 ... svc #0x0
```

At `svc #0`, v0.7 behavior takes over:

```text
x8 = 64       fake write
x0 = 1        stdout
x1 = 0x2000   message address
x2 = 13       length
```

The emulator copies 13 bytes from guest memory at `0x2000` to host stdout.

So v0.8 is not a replacement for v0.7. It uses v0.7.

---

## 11. `.bss` and zero-fill

A normal executable often has memory that should exist at runtime even though the file does not store all those bytes.

This is commonly called `.bss`.

The ELF way to describe this is:

```text
p_filesz < p_memsz
```

Example:

```text
p_vaddr  = 0x3000
p_filesz = 4
p_memsz  = 16
```

The loader should:

```text
copy 4 bytes from the file to memory[0x3000..0x3004)
set memory[0x3004..0x3010) to zero
```

This rule matters because programs expect uninitialized global storage to start as zero.

v0.8 supports this zero-fill behavior for `PT_LOAD` segments.

---

## 12. What v0.8 rejects on purpose

v0.8 supports only a deliberately small ELF subset.

It rejects:

```text
ELF32 files
big-endian ELF files
non-AArch64 ELF files
ET_DYN / PIE files
ELF files with PT_INTERP
relocatable object files
core files
malformed headers
bad program-header tables
segments outside emulator memory
overlapping PT_LOAD segments
entry points outside loaded memory
```

This is not a weakness. It is the point of the version.

The goal is not:

```text
Run every Linux binary.
```

The goal is:

```text
Learn how an executable loader maps simple segments safely.
```

---

## 13. Why `ET_DYN`, PIE, and `PT_INTERP` are rejected

Modern Linux executables are often dynamically linked or position-independent.

Those files may need:

```text
a dynamic linker
relocations
shared libraries
a real process stack with auxv/envp/argv
more Linux syscalls
```

The v0.8 emulator does not provide those things.

So if an ELF file says it needs an interpreter through `PT_INTERP`, the loader rejects it with a clear unsupported-feature error.

If an ELF file is `ET_DYN` / PIE, the loader rejects it too because v0.8 does not define a load-bias policy.

This keeps behavior deterministic for learning.

---

## 14. Segment permissions are represented but not enforced yet

ELF program headers include flags such as:

```text
R = readable
W = writable
X = executable
```

The v0.8 loader records those flags in segment metadata.

But v0.8 still uses the old flat memory model:

```text
one byte array
no page table
no permission checks
```

So this version does **not** yet stop you from executing a writable segment or writing an executable segment.

That is future work for a virtual-memory version.

For now, the important lesson is:

```text
The loader knows what permissions the file requested,
but the memory system does not enforce them yet.
```

Two related policies are also intentionally simple in v0.8:

```text
Segment alignment:
  PT_LOAD virtual addresses do not have to be page-aligned.
  They only need to fit inside emulator memory and not overlap.

Stack overlap:
  sp starts at the top of flat memory.
  v0.8 does not reserve or protect a separate stack region.
  A guest program is responsible for not mapping segments where its stack will grow.
```

Later versions can make this stricter by adding pages, permissions, and a real process-stack layout.

## 15. Debugging an ELF program

The debugger works with ELF programs because it uses the same loader as `run`, `trace`, `regs`, and `dump`.

Try:

```sh
printf 'break 0x1010\nrun\nregs\nstep\nquit\n' | ./emulator debug examples/v0_8/hello_elf.elf
```

The breakpoint address is an ELF guest address.

If the text segment is loaded at `0x1000`, then `0x1010` means:

```text
the instruction at guest memory address 0x1010
```

In the hello example, that is the `SVC #0` instruction for `write`.

The mental model stays the same as v0.5:

```text
debugger controls emulator execution
emulator fetches instructions from guest memory
breakpoints are guest memory addresses
```

The only new part is that ELF decided where the code was loaded.

---

## 16. Dumping ELF-loaded memory

Because ELF segments are copied into the same guest memory array, `dump` also works.

For the hello ELF example:

```sh
./emulator dump examples/v0_8/hello_elf.elf 0x2000 13
```

You should see the bytes of the message:

```text
68 65 6c 6c 6f 2c 20 76 30 2e 38 21 0a
```

Those are ASCII bytes for:

```text
hello, v0.8!\n
```

This connects the ELF loader back to the v0.3 memory lesson:

```text
ELF loading is just another way bytes appear in memory.
```

## 17. What changed inside the emulator

At a high level, v0.8 adds a program loader that chooses between two paths:

```text
emulator_load_program(path):
    read file bytes

    if bytes start with \x7fELF:
        load as ELF64
    else:
        load as raw binary at 0x1000
```

The raw path still does the old behavior:

```text
copy whole file to memory[0x1000]
pc = 0x1000
sp = top of memory
```

The ELF path does more validation:

```text
validate ELF header
validate program-header table
validate each PT_LOAD segment
reject PT_INTERP
reject ET_DYN / PIE
reject bad ranges or overlaps
copy segment bytes
zero-fill .bss bytes
pc = e_entry
sp = top of memory
```

In C-like pseudocode:

```c
if (is_elf_magic(file)) {
    load_elf64(emu, file, &program);
} else {
    load_raw(emu, file, &program);
}
```

The CLI and debugger now call this common loader instead of directly calling the raw loader.

That means all user-facing commands share the same program-loading behavior:

```text
run
trace
regs
dump
debug
```

## 18. Common mistakes

### Mistake 1: thinking ELF makes the CPU execute differently

ELF changes loading, not instruction meaning.

After the file is loaded and `pc` is set, the CPU still fetches, decodes, and executes instructions the same way.

### Mistake 2: thinking file offsets are guest addresses

`p_offset` is where bytes are inside the ELF file.

`p_vaddr` is where those bytes go in guest memory.

They can be the same in tiny examples, but they are different ideas.

### Mistake 3: thinking `.bss` bytes must exist in the file

`.bss` is represented by memory size larger than file size:

```text
p_memsz > p_filesz
```

The loader creates the extra zero bytes in memory.

### Mistake 4: trying to run a normal Linux program

A normal Linux program may need libc, a dynamic linker, many syscalls, relocations, and process startup data.

v0.8 intentionally supports only simple static AArch64 `ET_EXEC` examples.

### Mistake 5: thinking permissions are enforced

The loader records segment flags, but the memory model does not enforce read/write/execute permissions yet.

That belongs to a later virtual-memory milestone.

### Mistake 6: thinking v0.8 reserves a protected stack area

v0.8 initializes `sp` to the top of emulator memory, but it does not protect that region from ELF segments.

If a simple ELF maps memory near the top of the emulator address space, stack collision is the guest program's responsibility.

### Mistake 7: thinking every ELF alignment rule is enforced

v0.8 rejects a misaligned `e_entry` because instruction fetch needs 4-byte alignment.

But v0.8 does not require every `PT_LOAD` segment address to be page-aligned. The segment range must fit inside memory and avoid overlap with other loaded segments.

## 19. Exercises

### Exercise 1

A file does not start with `\x7fELF`.

Which loader path does v0.8 use?

Answer:

```text
The raw-binary path.
The file is copied to memory at 0x1000 and pc starts at 0x1000.
```

### Exercise 2

An ELF has:

```text
e_entry = 0x1800
```

What should `pc` be after loading?

Answer:

```text
pc = 0x1800
```

### Exercise 3

A `PT_LOAD` segment has:

```text
p_offset = 0x400
p_vaddr  = 0x2000
p_filesz = 0x20
p_memsz  = 0x30
```

What does the loader do?

Answer:

```text
Copy 0x20 bytes from file offset 0x400 into memory at 0x2000.
Then zero-fill 0x10 more bytes from 0x2020 through 0x202f.
```

### Exercise 4

Why does v0.8 reject `PT_INTERP`?

Answer:

```text
PT_INTERP means the program expects a dynamic linker.
v0.8 does not emulate dynamic linking, so it rejects the file clearly.
```

### Exercise 5

What is the difference between raw loading and ELF loading?

Answer:

```text
Raw loading puts the whole file at one fixed address, 0x1000.
ELF loading reads metadata and maps each PT_LOAD segment to the guest address requested by the file.
```

---

## 20. One-sentence summary

v0.8 teaches the emulator to recognize simple AArch64 ELF64 executables, validate their headers and program headers, load `PT_LOAD` segments into guest memory, zero-fill `.bss`, reject invalid entry points early, and start at `e_entry` while preserving the raw-binary path from earlier versions.
