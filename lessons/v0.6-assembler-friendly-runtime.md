# v0.6 Lesson — Assembler-Friendly Runtime

v0.6 does not add a new CPU concept like branches, stack, or function calls.

Instead, v0.6 makes the emulator easier to **read**, **trace**, and **use while learning**.

Before v0.6, trace output looked like this:

```text
trace pc=0x0000000000001000
trace pc=0x0000000000001004
trace pc=0x0000000000001008
```

That told us where the CPU was, but not what instruction lived there.

After v0.6, trace output looks like this:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
trace pc=0x0000000000001004 0x0000000000001004: 0xd2800061  movz x1, #0x3
trace pc=0x0000000000001008 0x0000000000001008: 0x91000c02  add x2, x0, #0x3
```

Now one trace line answers three questions:

```text
Where am I?       pc/address
What bytes?       raw opcode
What does it do?  decoded instruction text
```

## The three columns

Take this line:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
```

Break it apart:

| Part | Meaning |
|---|---|
| `trace pc=0x1000` | the CPU is about to execute the instruction at `0x1000` |
| `0x0000000000001000:` | same instruction address in listing format |
| `0xd2800040` | the raw 32-bit ARM64 opcode |
| `movz x0, #0x2` | human-readable decoded instruction |

This is still not a full disassembler. It only formats instructions that our emulator already supports.

## Why readable traces help

In v0.1, you learned:

```text
fetch -> decode -> execute
```

v0.6 makes that visible:

```text
fetch address:      0x1000
fetch raw opcode:   0xd2800040
decode as:          movz x0, #0x2
execute effect:     x0 = 2
```

## `run` vs `trace` vs `regs` vs `debug`

v0.6 gives you several ways to view the same program.

### `run`

```sh
./emulator run examples/v0_1/add.bin
```

Use this when you only care about the final result.

It prints:

```text
halted
x0 = ...
x1 = ...
...
```

### `trace`

```sh
./emulator trace examples/v0_1/add.bin
```

Use this when you want to see every instruction before it executes.

This is best for learning control flow.

### `regs`

```sh
./emulator regs examples/v0_1/add.bin
```

Use this when you want only the final CPU state.

It runs the program like `run`, but skips the extra `halted` line.

### `debug`

```sh
./emulator debug examples/v0_1/add.bin
```

Use this when you want to stop, step, inspect registers, inspect memory, or set breakpoints.

Inside the debugger, `trace on` now uses the same readable trace format as the normal `trace` command.

## Example: branch trace

A loop is easier to understand when repeated addresses are visible:

```sh
./emulator trace examples/v0_2/cbnz_countdown.bin
```

You will see some addresses repeat. That is the loop.

A repeated address means:

```text
a branch changed pc backward
```

## Example: memory trace

```sh
./emulator trace examples/v0_3/stack_push_pop.bin
```

Now load/store instructions are visible as text:

```text
str x0, [sp, #-8]!
ldr x1, [sp], #8
```

That makes it easier to connect the trace to the v0.3 stack lesson.

## Example: function trace

```sh
./emulator trace examples/v0_4/simple_call.bin
```

You will see a `bl` jump to the function and a `ret` return to the caller.

The trace still does not know function names. It only shows addresses and decoded instructions.

## How examples are built

The emulator does not read `.s` files directly.

The flow is:

```text
examples/v0_1/add.s
  -> assembler
examples/v0_1/add.o
  -> objcopy extracts raw .text
examples/v0_1/add.bin
  -> emulator runs raw bytes
```

Build all examples with:

```sh
make examples
```

## What v0.6 intentionally does not do

v0.6 does not add:

```text
ELF loading
Mach-O loading
source-level debugging
symbol names
DWARF
full ARM64 disassembly
new CPU instructions
```

It is a readability and workflow release.

## Small exercises

### Exercise 1

Given this trace line:

```text
trace pc=0x0000000000001008 0x0000000000001008: 0x91000c02  add x2, x0, #0x3
```

Which register changes?

Answer: `x2` changes.

### Exercise 2

Why does `trace` help more than `run` when learning loops?

Answer: `trace` shows repeated instruction addresses, so you can see when `pc` jumps backward.

### Exercise 3

What does `regs` show?

Answer: the final CPU state after the program finishes.
