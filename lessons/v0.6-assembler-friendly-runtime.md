# v0.6 Lesson — Readable Traces and Disassembly

This guide is beginner-first. It explains how v0.6 makes raw machine-code programs easier to read while keeping the emulator behavior the same.

In v0.5, you learned to pause and inspect execution with the debugger.

v0.6 adds one new learner-facing ability:

```text
The emulator can now explain the instruction it is about to execute.
```

Before v0.6, trace mostly answered:

```text
Where is the CPU?
```

After v0.6, trace answers:

```text
Where is the CPU?
What raw 32-bit instruction word is there?
How does the emulator decode that word into readable instruction text?
```

In one phrase, v0.6 adds **readable traces**.

---

## 1. Why we need readable traces

By v0.5, the emulator could already run useful examples:

```text
v0.1: straight-line instructions
v0.2: branches and loops
v0.3: memory and stack
v0.4: functions and returns
v0.5: debugger control
```

But the output was still hard to connect back to the program.

A trace line could tell you only this:

```text
trace pc=0x0000000000001000
trace pc=0x0000000000001004
trace pc=0x0000000000001008
```

That is useful, but it leaves beginner questions unanswered:

```text
What instruction is at 0x1000?
Was it an add, load, branch, call, return, or halt?
What raw machine-code bits did the emulator fetch?
```

v0.6 fixes that.

A v0.6 trace line looks like this:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
```

Now the trace tells a fuller story:

```text
pc/address:       0x1000
raw instruction:  0xd2800040
decoded text:     movz x0, #0x2
```

---

## 2. Running example

Use this tiny program as the main mental example:

```asm
movz x0, #2
movz x1, #3
add  x2, x0, #3
hlt  #0
```

The source file is human-readable assembly.

The `.bin` file is raw machine code.

The emulator executes the `.bin`, not the `.s` file.

When you run trace mode:

```sh
./emulator trace examples/v0_1/add.bin
```

you should think:

```text
The emulator is reading raw bytes,
but v0.6 prints a readable version for me.
```

---

## 3. The core mental model

The emulator still follows the same loop from v0.1:

```text
fetch -> decode -> execute
```

v0.6 makes the **decode** step visible to humans.

For this trace line:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
```

think:

```text
fetch address:      pc = 0x1000
fetch raw opcode:   0xd2800040
decode as:          movz x0, #0x2
execute effect:     x0 becomes 2
```

A useful picture:

```text
CPU
 pc = 0x1000
      |
      v
Memory bytes at 0x1000
      |
      v
32-bit instruction word
  0xd2800040
      |
      v
Decoder says
  movz x0, #0x2
      |
      v
Executor updates
  x0 = 2
```

Formatting the instruction helps you learn.

It does not change what the instruction does.

## 4. Slow walkthrough: one trace line

Take the full line:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
```

Break it into pieces:

| Piece | Meaning |
|---|---|
| `trace` | This came from trace mode. |
| `pc=0x0000000000001000` | The CPU is about to execute the instruction at `0x1000`. |
| `0x0000000000001000:` | The same address, formatted like a disassembly listing. |
| `0xd2800040` | The raw 32-bit instruction word fetched from memory. |
| `movz x0, #0x2` | The emulator's readable decoding of that instruction. |

The address appears twice on purpose:

```text
pc=...        helps you think like the CPU
address:      helps you read it like a disassembly listing
```

The most important timing rule is:

```text
trace prints the instruction before it executes.
```

So if the trace line says `movz x0, #0x2`, that line appears before `x0` changes to `2`.

---

## 5. Disassembly is decoded text, not the original source

This is one of the easiest ideas to misunderstand.

When the emulator prints:

```text
movz x0, #0x2
```

it is **not** reading that line from your `.s` file.

The flow is:

```text
human assembly source
  examples/v0_1/add.s
        |
        v
assembler turns text into machine code
        |
        v
raw binary bytes
  examples/v0_1/add.bin
        |
        v
emulator loads bytes into memory
        |
        v
emulator decodes 32-bit instruction words
        |
        v
emulator prints readable instruction text
```

So this trace text:

```text
add x2, x0, #0x3
```

means:

```text
The emulator decoded the raw 32-bit instruction word
and formatted it into a readable ARM64-like string.
```

It does **not** mean:

```text
The emulator found the original source-code line.
```

That distinction matters because raw `.bin` files do not contain comments, source labels, function names, or source line numbers.

---

## 6. Why raw `.bin` files are different from normal executables

This project still runs raw binary instruction bytes.

A `.bin` file is close to:

```text
instruction
instruction
instruction
instruction
```

A normal executable format, such as ELF or Mach-O, can contain extra information:

```text
entry point
sections
symbol names
relocations
debug info
source line mapping
metadata for the operating system loader
```

A raw `.bin` in this project does not have those things.

So the emulator uses simple rules:

```text
load raw bytes at address 0x1000
set pc to 0x1000
start executing 4 bytes at a time
stop when the program halts, exits, errors, or hits the instruction limit
```

That is why v0.6 trace output is useful: it gives you a readable view even though the input file is just raw bytes.

## 7. `run`, `regs`, `trace`, and `debug`

These modes all use the same emulator core. They differ in **how much they show you** and **how much control they give you**.

### `run`

Use `run` when you want to execute the program normally.

```sh
./emulator run examples/v0_1/add.bin
```

Good question for `run`:

```text
Did the program finish, and what final state did it leave behind?
```

### `regs`

Use `regs` when you mostly care about final register values.

```sh
./emulator regs examples/v0_1/add.bin
```

Good question for `regs`:

```text
What are x0, x1, x2, pc, sp, and flags after execution?
```

### `trace`

Use `trace` when you want to see every instruction before it executes.

```sh
./emulator trace examples/v0_2/cbnz_countdown.bin
```

Good question for `trace`:

```text
How did the program get from the start state to the final state?
```

Starting in v0.6, each trace line includes readable decoded instruction text.

### `debug`

Use `debug` when you want interactive control.

```sh
./emulator debug examples/v0_4/simple_call.bin
```

Inside the debugger you can:

```text
step one instruction
continue until a breakpoint
print registers
dump memory
set/delete/list breakpoints
turn trace on/off
```

Good question for `debug`:

```text
Can I stop exactly where I am confused and inspect the machine state?
```

## 8. Slow walkthrough: trace through a loop

In v0.2, branches changed the `pc`.

Straight-line execution usually moves forward by 4 bytes:

```text
0x1000
0x1004
0x1008
0x100c
```

A loop may look like this:

```text
0x1000
0x1004
0x1008
0x1004
0x1008
0x1004
0x1008
0x100c
```

The repeated addresses are the clue.

A repeated address usually means:

```text
A branch changed pc backward.
```

For example:

```asm
loop:
    sub  x0, x0, #1
    cbnz x0, loop
```

The assembler turns `loop` into an encoded branch target. The raw `.bin` does not contain the name `loop`.

The emulator trace can still show the branch target address:

```text
cbnz x0, 0x0000000000001000
```

That means:

```text
if x0 is not zero, set pc back to 0x1000
otherwise, continue to the next instruction
```

## 9. Reading memory, stack, calls, and returns

v0.6 does not replace earlier lessons. It makes their instructions easier to recognize while they run.

From v0.3, load/store instructions may appear as:

```text
str x0, [sp, #-8]!
ldr x1, [sp], #8
```

Use the memory mental model from v0.3:

```text
memory is a big byte array
registers hold values or addresses
STR copies register data into memory
LDR copies memory data into a register
```

So this:

```text
str x0, [sp, #-8]!
```

roughly means:

```text
move sp down by 8 bytes
store x0 at the new sp address
```

From v0.4, function calls may appear as:

```text
bl 0x0000000000001010
ret
```

Read them as:

```text
BL saves a return address in x30 and jumps.
RET sets pc from x30.
```

Important limitation:

```text
The trace does not know function names.
```

If your source said:

```asm
bl add_three
```

v0.6 trace may show only:

```text
bl 0x0000000000001010
```

because the raw `.bin` has the encoded target address, not the source name `add_three`.

## 10. What changed inside the emulator

Conceptually, v0.6 adds a formatting step near tracing and errors.

Before:

```text
fetch opcode
execute opcode
maybe print pc
```

After:

```text
fetch opcode
format opcode as readable text when needed
execute opcode
```

Small pseudocode:

```c
opcode = fetch_u32(memory, pc);
decoded = decode(opcode);
text = format(decoded, pc);
printf("trace pc=%... %s\n", pc, text);
execute(decoded);
```

The important separation is:

```text
Formatting answers: "How do I print this instruction?"
Execution answers:  "How does this instruction change CPU/memory state?"
```

Keeping those separate makes the emulator easier to test and easier to extend.

---

## 11. What v0.6 intentionally does not do

v0.6 is not a full disassembler.

It only formats instructions this emulator already understands.

v0.6 does **not** add:

```text
ELF loading
Mach-O loading
source-level debugging
symbol names
DWARF debug info
full ARM64 disassembly
new CPU instructions
real operating-system services
```

That is why this version is a readability and workflow release, not a new CPU-instruction release.

## 13. Common mistakes

### Mistake 1: thinking disassembly is the original source

The emulator reads the `.bin` file, not the `.s` file.

Readable instruction text is reconstructed from raw instruction bits.

### Mistake 2: thinking repeated addresses mean duplicate instructions

Usually, repeated addresses mean a branch changed `pc` backward.

That is how loops appear in trace output.

### Mistake 3: forgetting trace prints before execution

A trace line tells you:

```text
This is the instruction about to run.
```

After it runs, registers, memory, flags, or `pc` may change.

### Mistake 4: expecting source labels in raw binary trace

Labels are source-level names. The raw `.bin` file does not keep them.

The emulator can show target addresses, but not original label names.

### Mistake 5: thinking v0.6 changes instruction behavior

v0.6 changes how instructions are shown to humans.

It should not change how existing programs execute.

## 14. Exercises

### Exercise 1

Given this line:

```text
trace pc=0x0000000000001008 0x0000000000001008: 0x91000c02  add x2, x0, #0x3
```

Which register changes?

Answer:

```text
x2 changes.
```

### Exercise 2

Why does this sequence suggest a loop?

```text
0x1000
0x1004
0x1008
0x1004
0x1008
```

Answer:

```text
Because pc went backward from 0x1008 to 0x1004.
A branch probably jumped back to an earlier instruction.
```

### Exercise 3

Why can v0.6 show `bl 0x1010` but not necessarily `bl add_three`?

Answer:

```text
The raw .bin contains the encoded target address, not the original source label name.
```

### Exercise 4

What is the difference between these two ideas?

```text
format instruction
execute instruction
```

Answer:

```text
Formatting produces readable text for humans.
Execution changes CPU registers, memory, flags, pc, or halt state.
```

## 15. One-sentence summary

v0.6 teaches the emulator to print readable traces: each trace line shows the current `pc`, the raw 32-bit instruction word, and decoded instruction text, so raw `.bin` programs are easier to understand without changing how instructions execute.
