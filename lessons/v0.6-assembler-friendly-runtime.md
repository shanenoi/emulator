# v0.6 Lesson — Assembler-Friendly Runtime

This guide is written for someone who already knows the basic emulator model from v0.1–v0.5, but does **not** yet feel fully comfortable reading trace output, raw opcodes, or debugger output.

v0.6 does not add a new CPU feature like branches, stack, or function calls.

Instead, v0.6 makes the emulator easier to **read**, **trace**, and **use while learning**.

In one phrase, v0.6 adds **readable traces** for the raw binaries you were already running.

The short version:

```text
Before v0.6:
    trace told you only where the CPU was.

After v0.6:
    trace tells you where the CPU is,
    what raw 32-bit instruction word is there,
    and how the emulator decodes that instruction as readable text.
```

---

## 1. The problem v0.6 solves

Earlier versions could run examples, but trace output was hard to connect back to the program.

Before v0.6, trace output looked like this:

```text
trace pc=0x0000000000001000
trace pc=0x0000000000001004
trace pc=0x0000000000001008
```

That answers only one question:

```text
Where is the CPU about to execute?
```

It does **not** answer:

```text
What instruction is at that address?
What raw machine-code bits were fetched?
Is this an add, branch, load, store, call, or return?
```

After v0.6, trace output looks like this:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
trace pc=0x0000000000001004 0x0000000000001004: 0xd2800061  movz x1, #0x3
trace pc=0x0000000000001008 0x0000000000001008: 0x91000c02  add x2, x0, #0x3
```

Now each trace line answers three beginner-important questions:

```text
Where am I?       pc/address
What bits?        raw 32-bit opcode
What does it do?  decoded instruction text
```

This is why v0.6 is called an **assembler-friendly runtime** release. It makes raw machine-code examples feel closer to the assembly you wrote.

---

## 2. The most important mental model

The emulator still follows the same loop from v0.1:

```text
fetch -> decode -> execute
```

v0.6 makes that loop visible in the output.

For this line:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
```

think:

```text
fetch address:      pc = 0x1000
fetch raw opcode:   0xd2800040
decode as:          movz x0, #0x2
execute effect:     x0 becomes 2
```

A useful picture:

```text
CPU
 pc = 0x1000
      |
      v
Memory bytes at 0x1000
      |
      v
32-bit instruction word
  0xd2800040
      |
      v
Decoder says
  movz x0, #0x2
      |
      v
Executor updates
  x0 = 2
```

v0.6 mostly improves the **decoder/formatting** part of that picture.

---

## 3. Disassembly is decoded text, not the original source file

This is one of the easiest ideas to accidentally misunderstand.

When the emulator prints:

```text
movz x0, #0x2
```

it is **not** reading that line from your `.s` file.

The flow is:

```text
human assembly source
  examples/v0_1/add.s
        |
        v
assembler turns text into machine code
        |
        v
raw binary bytes
  examples/v0_1/add.bin
        |
        v
emulator loads bytes into memory
        |
        v
emulator decodes 32-bit instruction words
        |
        v
emulator prints readable instruction text
```

So this trace text:

```text
add x2, x0, #0x3
```

means:

```text
The emulator decoded the raw 32-bit instruction word
and formatted it into a readable ARM64-like string.
```

It does **not** mean:

```text
The emulator found the original source-code line.
```

That distinction matters because the raw `.bin` file does not contain comments, labels, function names, or source line numbers.

## 4. Breaking down one trace line

Take this complete trace line:

```text
trace pc=0x0000000000001000 0x0000000000001000: 0xd2800040  movz x0, #0x2
```

Break it into pieces:

| Piece | Meaning |
|---|---|
| `trace` | This came from trace mode. |
| `pc=0x0000000000001000` | The CPU is about to execute the instruction at address `0x1000`. |
| `0x0000000000001000:` | The same address, formatted like a disassembly listing. |
| `0xd2800040` | The raw 32-bit instruction word fetched from memory. |
| `movz x0, #0x2` | The emulator's readable decoding of that instruction. |

The address appears twice on purpose:

```text
trace pc=...       helps you think like the CPU
address: opcode    helps you read it like a disassembly listing
```

## 5. Why raw `.bin` files are different from normal executables

This project still runs raw binary instruction bytes.

That means a `.bin` file is close to:

```text
instruction
instruction
instruction
instruction
```

It does **not** include the extra information found in a normal executable format such as ELF or Mach-O.

A normal executable can contain things like:

```text
entry point
sections
symbol names
relocations
debug info
source line mapping
metadata for the operating system loader
```

A raw `.bin` in this project does not have those things.

So the emulator uses simple rules:

```text
load raw bytes at address 0x1000
set pc to 0x1000
start executing 4 bytes at a time
stop when the program halts, exits, errors, or hits the instruction limit
```

That is why v0.6 trace output is useful: it gives you a readable view even though the input file is just raw bytes.

## 6. `run`, `regs`, `trace`, and `debug`

These modes all use the same emulator core. They differ in **how much they show you** and **how much control they give you**.

### `run`

Use `run` when you want to execute the program normally.

```sh
./emulator run examples/v0_1/add.bin
```

For older `HLT`-based examples, `run` prints a final state summary.

Good question for `run`:

```text
Did the program finish, and what final state did it leave behind?
```

### `regs`

Use `regs` when you mostly care about final register values.

```sh
./emulator regs examples/v0_1/add.bin
```

Good question for `regs`:

```text
What are x0, x1, x2, pc, sp, and flags after execution?
```

### `trace`

Use `trace` when you want to see every instruction before it executes.

```sh
./emulator trace examples/v0_2/cbnz_countdown.bin
```

Good question for `trace`:

```text
How did the program get from the start state to the final state?
```

Starting in v0.6, each trace line includes readable decoded instruction text.

### `debug`

Use `debug` when you want interactive control.

```sh
./emulator debug examples/v0_4/simple_call.bin
```

Inside the debugger you can:

```text
step one instruction
continue until a breakpoint
print registers
dump memory
set/delete/list breakpoints
turn trace on/off
```

Good question for `debug`:

```text
Can I stop exactly where I am confused and inspect the machine state?
```

## 7. Reading branches and loops in trace output

In v0.2, branches changed the `pc`.

Normal straight-line execution looks like this:

```text
0x1000
0x1004
0x1008
0x100c
```

A loop may look like this:

```text
0x1000
0x1004
0x1008
0x1004
0x1008
0x1004
0x1008
0x100c
```

The repeated addresses are the clue.

A repeated address usually means:

```text
A branch changed pc backward.
```

For example:

```asm
loop:
    sub  x0, x0, #1
    cbnz x0, loop
```

The assembler turns `loop` into an encoded branch target. The raw `.bin` does not contain the name `loop`.

The emulator trace can still show the branch target address:

```text
cbnz x0, 0x0000000000001000
```

That means:

```text
if x0 is not zero, set pc back to 0x1000
otherwise, continue to the next instruction
```

## 8. Reading memory and stack instructions

v0.3 added load/store instructions. v0.6 makes them easier to recognize in trace output.

You may see lines like:

```text
str x0, [sp, #-8]!
ldr x1, [sp], #8
```

Use the v0.3 memory mental model:

```text
memory is a big byte array
registers hold values or addresses
STR copies register data into memory
LDR copies memory data into a register
```

So this:

```text
str x0, [sp, #-8]!
```

roughly means:

```text
move sp down by 8 bytes
store x0 at the new sp address
```

And this:

```text
ldr x1, [sp], #8
```

roughly means:

```text
load 8 bytes from memory at sp into x1
then move sp up by 8 bytes
```

Trace output does not replace the memory lesson. It gives you a way to connect each running instruction to that lesson.

---

## 9. Reading function calls and returns

v0.4 added function calls using `BL`, `RET`, and `x30`.

In trace output, a call may look like:

```text
bl 0x0000000000001010
```

That means:

```text
save return address in x30
set pc to 0x1010
```

Later, a return may look like:

```text
ret
```

That means:

```text
set pc to x30
```

Important limitation:

```text
The trace does not know function names.
```

If your source said:

```asm
bl add_three
```

v0.6 trace may show only:

```text
bl 0x0000000000001010
```

because the raw `.bin` has the encoded target address, not the name `add_three`.

## 10. What changed inside the emulator

Conceptually, v0.6 adds a formatting step near tracing and errors.

Before:

```text
fetch opcode
execute opcode
maybe print pc
```

After:

```text
fetch opcode
format opcode as readable text when needed
execute opcode
```

Small pseudocode:

```c
opcode = fetch_u32(memory, pc);
decoded = decode(opcode);
text = format(decoded, pc);
printf("trace pc=%... %s\n", pc, text);
execute(decoded);
```

The important thing is that formatting is a teaching/debugging feature.

It should not change what the instruction does.

```text
Formatting answers: "How do I print this instruction?"
Execution answers:  "How does this instruction change CPU/memory state?"
```

Keeping those separate makes the emulator easier to test and easier to extend.

---

## 11. What v0.6 intentionally does not do

v0.6 is not a full disassembler.

It only formats instructions this emulator already understands.

v0.6 does **not** add:

```text
ELF loading
Mach-O loading
source-level debugging
symbol names
DWARF debug info
full ARM64 disassembly
new CPU instructions
real operating-system services
```

That is why this version is a workflow/readability release, not a new CPU-feature release.

## 13. Common confusions

### Confusion: “Is the emulator reading my `.s` source file?”

No.

The emulator reads the `.bin` file.

The readable instruction text is reconstructed from raw instruction bits.

### Confusion: “Why are there two addresses in one trace line?”

They are the same instruction address shown in two useful styles:

```text
pc=...        CPU state style
address:      disassembly listing style
```

### Confusion: “Why do addresses repeat?”

Usually because a branch changed `pc` backward.

That is how loops appear in trace output.

### Confusion: “Why are labels missing?”

Labels are source-level names. The raw `.bin` file does not keep them.

The emulator can show target addresses, but not original label names.

### Confusion: “Why does trace show the instruction before effects happen?”

Trace is printed before execution.

That means the line tells you:

```text
This is the instruction about to run.
```

After it runs, registers/memory may change.

## 14. Mini exercises

### Exercise 1

Given this line:

```text
trace pc=0x0000000000001008 0x0000000000001008: 0x91000c02  add x2, x0, #0x3
```

Which register changes?

Answer:

```text
x2 changes.
```

### Exercise 2

Why does this sequence suggest a loop?

```text
0x1000
0x1004
0x1008
0x1004
0x1008
```

Answer:

```text
Because pc went backward from 0x1008 to 0x1004.
A branch probably jumped back to an earlier instruction.
```

### Exercise 3

Why can v0.6 show `bl 0x1010` but not necessarily `bl add_three`?

Answer:

```text
The raw .bin contains the encoded target address, not the original source label name.
```

### Exercise 4

What is the difference between these two ideas?

```text
format instruction
execute instruction
```

Answer:

```text
Formatting produces readable text for humans.
Execution changes CPU registers, memory, flags, pc, or halt state.
```
