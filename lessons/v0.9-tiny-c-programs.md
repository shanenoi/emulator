# v0.9 Lesson — Tiny Freestanding C Programs

This guide is beginner-first. It continues the learning chain from v0.1 through v0.8 and explains how the emulator moves from hand-written assembly examples to very small C programs.

In v0.8, the emulator learned to load simple static AArch64 ELF64 executables:

```text
read ELF header
load PT_LOAD segments
zero-fill .bss
start at e_entry
```

But the programs were still mostly hand-written assembly.

v0.9 adds one new ability:

```text
The emulator can run tiny freestanding C programs compiled to AArch64 ELF64.
```

The important phrase is **freestanding C**.

A normal C program often expects an operating system and libc. A v0.9 C program does not. It is still a tiny bare program that runs inside this teaching emulator.

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
```

v0.9 does **not** turn the emulator into Linux.

It adds enough instruction support and example build rules so a carefully constrained C program can become:

```text
C source -> AArch64 object file -> static ELF64 executable -> emulator
```

The CPU still has the same flat memory model, the same fake syscalls, and the same debugger.

---

## 2. Hosted C vs freestanding C

Most beginner C examples are **hosted C**:

```c
#include <stdio.h>

int main(void) {
    printf("hello\n");
    return 0;
}
```

That program quietly depends on a lot of outside help:

```text
operating-system process setup
libc startup code
printf implementation
file descriptors from the real OS
dynamic linker or static libc code
argv, envp, and auxiliary vector
```

This emulator does not provide those things.

v0.9 uses **freestanding C** instead. A freestanding program avoids libc and talks only to the tiny runtime we provide:

```c
int main(void) {
    return 42;
}
```

That small `main` can compile to AArch64 instructions, and our own `_start` assembly stub can call it.

The mental model is:

```text
normal hosted C:
    OS/libc startup -> main -> libc/OS exit

v0.9 freestanding C:
    tiny _start -> main -> fake exit syscall
```

## 4. The tiny `_start` stub

A normal hosted C program does not start directly at `main`.

Something has to call `main` first.

In hosted C, libc usually provides that startup code. In v0.9, this repository provides a tiny version in `examples/v0_9/start.s`:

```asm
.text
.global _start
.extern main

_start:
    bl main
    movz x8, #93
    svc #0
```

Slowly:

```text
bl main
```

uses the v0.4 function-call rule:

```text
x30 = address after the BL
pc  = address of main
```

Then C `main` eventually executes:

```text
ret
```

That returns to `_start`.

By the AArch64 calling convention, the integer return value from `main` is in `w0` / `x0`.

Then `_start` runs:

```asm
movz x8, #93
svc #0
```

That uses the v0.7 toy syscall rule:

```text
x8 = 93 means fake exit
x0 = exit status
SVC #0 asks the emulator to handle it
```

So the whole path is:

```text
ELF e_entry
  -> _start
  -> BL main
  -> C code returns value in x0
  -> _start sets x8 = 93
  -> SVC #0
  -> emulator exits with low 8 bits of x0
```

## 5. Slow walkthrough: `_start -> main -> exit`

For `return_42.c`, trace output begins like this:

```text
trace pc=0x0000000000001000 ... bl 0x000000000000100c
trace pc=0x000000000000100c ... movz w0, #0x2a
trace pc=0x0000000000001010 ... ret
trace pc=0x0000000000001004 ... movz x8, #0x5d
trace pc=0x0000000000001008 ... svc #0x0
```

Line by line:

```text
pc=0x1000: bl main
```

The startup stub calls `main`.

```text
pc=0x100c: movz w0, #0x2a
```

The compiled C function puts `42` into `w0`.

```text
pc=0x1010: ret
```

`main` returns to `_start` using `x30`.

```text
pc=0x1004: movz x8, #0x5d
```

`0x5d` is decimal `93`, the fake exit syscall number.

```text
pc=0x1008: svc #0
```

The emulator sees `x8 = 93` and exits with status `x0 & 0xff`.

## 6. The compiler emits instructions, not C concepts

The emulator does not execute C source code directly.

It executes AArch64 instructions produced by the compiler.

For example, this C code:

```c
int main(void) {
    return 42;
}
```

may become something like:

```asm
movz w0, #42
ret
```

The emulator only sees the machine-code version.

That means v0.9 needs more instruction support, not because the emulator understands C, but because the compiler tends to emit more varied instructions than our hand-written early examples.

v0.9 adds compiler-oriented instructions such as:

```text
MOVN / MOVK        build constants
ADD / SUB register arithmetic
ADD / SUB immediate with SP support for stack frames
ADDS / SUBS        arithmetic that also updates flags
AND / ORR / EOR    bitwise logic
LSL / LSR / ASR    shifts
MUL / UDIV / SDIV  multiply and divide
ADR / ADRP         address generation for code/data
LDRB / STRB        byte memory access
LDRH / STRH        halfword memory access
```

The goal is not to support every instruction a compiler could ever emit.

The goal is to support a documented, constrained compiler profile for small teaching examples. Logical-immediate instructions and sign-extension aliases such as `UXTB`, `UXTH`, and `SXTW` are intentionally outside the selected v0.9 profile unless a later version adds them.

## 7. The documented compiler profile

The v0.9 examples use this kind of compile command. The Makefile recipes use `clang` and `ld.lld` when available; if those tools are missing, v0.9 example targets print a clear skip message rather than failing just because the optional cross-toolchain is absent:

```sh
clang --target=aarch64-none-elf -ffreestanding -nostdlib \
  -fno-stack-protector -fno-pic -fno-pie -O0 \
  -c examples/v0_9/fib.c -o examples/v0_9/fib.o
```

The important flags are:

```text
--target=aarch64-none-elf
```

Compile for AArch64 ELF, not for your host machine.

```text
-ffreestanding
```

Tell the compiler this is not a normal hosted C environment.

```text
-nostdlib
```

Do not link libc or operating-system startup files.

```text
-fno-stack-protector
```

Avoid compiler-inserted stack-check helper calls that need runtime support we do not have.

```text
-fno-pic -fno-pie
```

Avoid position-independent executable assumptions. v0.8 supports static `ET_EXEC`, not PIE/`ET_DYN`.

```text
-O0
```

Use the documented optimization profile for examples. This often produces smaller code and avoids some bulky debug-style stack frames.

Then the object is linked with our tiny `_start` and linker script:

```sh
ld.lld -static -nostdlib -T examples/v0_9/linker.ld \
  examples/v0_9/start.o examples/v0_9/fib.o \
  -o examples/v0_9/fib.elf
```

## 8. Example: Fibonacci from C

The v0.9 Fibonacci example is intentionally recursive so you can see normal C calls use the same `BL`, `RET`, `x30`, and stack-frame ideas from v0.4:

```c
static volatile int fib_input = 10;

static int fib_recursive(int n) {
    if (n <= 1) {
        return n;
    }
    return fib_recursive(n - 1) + fib_recursive(n - 2);
}

int main(void) {
    return fib_recursive(fib_input);
}
```

Build and run it:

```sh
make examples/v0_9/fib.elf
./emulator run examples/v0_9/fib.elf
echo $?
```

Expected exit status:

```text
55
```

Why `volatile`?

Without it, the compiler may decide that `fib_recursive(10)` can be computed at compile time and replace the whole function with:

```asm
movz w0, #55
ret
```

That is legal compiler behavior, but it hides the recursive calls. `volatile` makes the compiler load the input from memory so the example shows more of the emulator's call, stack, branch, and instruction support.

## 9. Address generation: `ADR` and `ADRP`

C programs often refer to global data:

```c
static volatile int fib_input = 10;
```

The compiled code needs to compute the address of that data.

For nearby data, the compiler may use:

```asm
adr x8, some_label
```

Mental model:

```text
x8 = current pc + signed byte offset
```

For page-based addressing, the compiler may use:

```asm
adrp x8, some_global
ldr  w8, [x8]
```

Mental model:

```text
ADRP starts from the current 4 KiB page of pc.
Then it adds a signed page offset.
```

Important detail:

```text
ADR  is byte-relative.
ADRP is page-relative.
```

v0.9 computes `ADRP` from the instruction `pc` page, not `pc + 4`.

## 10. Byte and halfword memory

Earlier memory lessons focused mostly on 32-bit and 64-bit loads/stores:

```asm
ldr w0, [x1]
str x0, [sp, #-8]!
```

C string-like code needs byte access:

```c
while (message[len] != '\0') {
    len++;
}
```

That can compile into:

```asm
ldrb w9, [x8], #1
cbnz w9, loop
```

`LDRB` means:

```text
load 1 byte from memory
zero-extend it into a W register
```

So if memory contains byte `0xff`, then:

```text
ldrb w0, [x1]
```

sets:

```text
w0 = 0x000000ff
```

not:

```text
w0 = 0xffffffff
```

`LDRH` is similar, but loads 2 bytes and zero-extends them.

## 11. What `hello_c.c` proves

The C hello example still uses the v0.7 fake syscall ABI.

The C code contains a tiny wrapper:

```c
static long fake_write(long fd, const char *buffer, unsigned long length) {
    register long x0 asm("x0") = fd;
    register const char *x1 asm("x1") = buffer;
    register unsigned long x2 asm("x2") = length;
    register long x8 asm("x8") = 64;
    asm volatile("svc #0" : "+r"(x0) : "r"(x1), "r"(x2), "r"(x8) : "memory");
    return x0;
}
```

That wrapper says:

```text
put fd in x0
put guest buffer address in x1
put byte length in x2
put syscall number write = 64 in x8
execute SVC #0
```

The same fake syscall table from v0.7 is still in use here:

```text
write = 64
exit = 93
```

So v0.9 does not add real `printf`.

It shows how C code can still use the tiny syscall agreement you learned in v0.7.

Run it:

```sh
make examples/v0_9/hello_c.elf
./emulator run examples/v0_9/hello_c.elf
```

Expected output:

```text
hello from c
```

## 12. Slow walkthrough: C string length

`string_len.c` scans bytes until a NUL terminator:

```c
static const char message[] = "hello";

int main(void) {
    const char *p = message;
    int len = 0;
    while (*p != '\0') {
        len++;
        p++;
    }
    return len;
}
```

A typical trace includes a loop that loads one byte, checks it, increments the count, and advances the pointer:

```text
adr  x8, message
ldrb w9, [x8]
add  w0, w0, #1
add  x8, x8, #1
cbnz w9, loop
ret
```

The memory mental model is:

```text
address     byte      meaning
--------    ----      -------
message+0   68        'h'
message+1   65        'e'
message+2   6c        'l'
message+3   6c        'l'
message+4   6f        'o'
message+5   00        NUL terminator
```

Each loop iteration:

```text
1. LDRB reads one byte.
2. ADD increments the length.
3. ADD advances the pointer by 1.
4. CBNZ loops if the byte was not zero.
```

This combines older lessons:

```text
v0.2: CBNZ loop
v0.3: memory load
v0.6: readable trace
v0.8: data loaded from ELF segment
v0.9: byte-sized C string access
```

## 13. What is still out of scope

v0.9 intentionally does not support normal hosted C.

These are still out of scope:

```text
printf
malloc
files
argc / argv
environment variables
auxiliary vector
libc startup files
shared libraries
dynamic linker
PIE / ET_DYN loading
floating-point / SIMD
C++ runtime
exceptions / unwinding
real Linux syscall compatibility
```

If you try to compile a normal Linux C program, it may produce an ELF file that the loader rejects, or it may contain unsupported instructions or calls to missing runtime functions.

That is expected.

The promise is narrower:

```text
small freestanding C + documented flags + static ET_EXEC ELF
```

## 14. What changed inside the emulator

At a high level, v0.9 changes three areas.

First, instruction decoding learned more compiler-style patterns:

```text
raw opcode -> decoded instruction kind + registers + immediates
```

Second, instruction execution learned how those operations update registers and memory:

```text
ADD register: xD = xN + xM
LDRB:         wD = zero_extend(memory[address])
ADRP:         xD = pc_page + signed_page_offset
SDIV:         signed division, divide-by-zero gives zero
```

Third, examples now include a tiny C runtime shape and several C-originated behaviors:

```text
examples/v0_9/start.s
examples/v0_9/return_42.c
examples/v0_9/fib.c              recursive calls
examples/v0_9/sum_array.c        global data, .bss, pointer loop
examples/v0_9/string_len.c       pointer loop and byte loads
examples/v0_9/byte_copy.c        memcpy-style byte loop
examples/v0_9/static_local.c     static local storage
examples/v0_9/stderr_c.c         fd 2 writes
examples/v0_9/bad_fd_c.c         -EBADF handling
examples/v0_9/unknown_syscall_c.c -ENOSYS handling
examples/v0_9/invalid_write_c.c  runtime error path
examples/v0_9/linker.ld
```

The loader is still the v0.8 loader. v0.9 reuses it.

## 15. Common mistakes

### Mistake 1: thinking the emulator runs C source

The emulator does not read `.c` files.

It runs AArch64 machine code inside `.elf` files.

C is only the language used before compilation.

### Mistake 2: thinking freestanding C means normal C with fewer libraries

Freestanding C is a different environment assumption.

There is no automatic libc startup and no operating-system process setup.

You need `_start` because something must call `main`.

### Mistake 3: expecting `printf`

`printf` is a libc function. v0.9 does not implement libc.

Use the tiny fake syscall wrapper if you want visible output.

### Mistake 4: thinking `main` exits by itself

`main` only returns to whoever called it.

In v0.9, `_start` calls `main`, receives the return value in `x0`, and then performs fake `exit`.

### Mistake 5: assuming every compiler output is supported

Different flags, optimization levels, and compiler versions may emit different instructions.

v0.9 supports a deliberately constrained profile. Unsupported instructions should fail clearly rather than silently doing the wrong thing. In this profile, logical-immediate instructions and sign-extension aliases such as `UXTB`, `UXTH`, and `SXTW` are documented as unsupported unless added later.
