# v1.0 Lesson — Stable Learning Emulator

## What v0.9 could already do

By the end of v0.9, the emulator could run tiny freestanding C programs:

```text
raw .bin -> branches -> memory -> calls -> debugger -> readable traces -> fake syscalls -> ELF -> tiny C
```

That is a full learning ladder. You can start with a few raw ARM64 instructions, then eventually run a small C `main` that exits through the toy syscall ABI.

v1.0 does **not** add a large new CPU feature. Instead, v1.0 turns the project into a stable learning release.

The new ability is confidence:

```text
clone -> build -> test -> run examples -> debug examples -> understand the limits
```

## The v1.0 mental model

Earlier versions asked, “Can the emulator do one more thing?”

v1.0 asks, “Can a new learner trust the whole path?”

That means v1.0 focuses on release quality:

- the README should match the code,
- command-line usage should be clear,
- errors should explain what went wrong,
- optional tools should skip clearly,
- generated files should be cleaned,
- examples should represent the supported features,
- the project should be honest about what it still does **not** support.

## The stable CLI surface

The public commands are:

```sh
./emulator run <program>
./emulator trace <program>
./emulator regs <program>
./emulator dump <program> <address> <length>
./emulator debug <program>
./emulator help
```

`<program>` can be either:

```text
raw .bin file              loaded at 0x1000
supported ELF64 ET_EXEC    loaded through PT_LOAD segments
```

The old lessons often say `<raw-binary>`. That wording is still useful for v0.1 through v0.7, but the v1.0 wording is broader: `<program>`.

## Slow walkthrough: the release smoke path

A new learner should be able to do this from the repository root:

```sh
make clean
make test
make examples
```

Then run representative examples:

```sh
./emulator run examples/v0_1/add.bin
./emulator trace examples/v0_2/trace_loop.bin
./emulator dump examples/v0_3/memory_store_load.bin 0xffff8 8
./emulator debug examples/v0_1/add.bin < examples/v0_5/debug_add_script.txt
./emulator run examples/v0_7/hello.bin
./emulator run examples/v0_8/hello_elf.elf
./emulator run examples/v0_9/return_42.elf
```

This is not random repetition. Each command checks one layer of the learning ladder:

| Command | What it proves |
|---|---|
| `run examples/v0_1/add.bin` | raw instruction sandbox still works |
| `trace examples/v0_2/trace_loop.bin` | branches and readable traces still work |
| `dump examples/v0_3/memory_store_load.bin ...` | memory and dump mode still work |
| `debug ... < debug_add_script.txt` | debugger scripting still works |
| `run examples/v0_7/hello.bin` | fake syscall output still works |
| `run examples/v0_8/hello_elf.elf` | ELF loading still works |
| `run examples/v0_9/return_42.elf` | freestanding C path still works when built |

## Optional v0.9 C examples

The v0.9 C examples need extra tools:

```text
clang --target=aarch64-none-elf
ld.lld
```

Those tools are optional for the main test suite. The deterministic v0.9 tests generate ELF fixtures directly, so `make test` does not require a cross compiler.

But `make examples` may try to build real C examples. If the required tools are unavailable, the recipe prints a skip message. A skip message means the requested `.elf` file was **not** produced.

So this sequence may fail for an understandable reason:

```sh
make examples/v0_9/fib.elf
./emulator run examples/v0_9/fib.elf
```

If the first command skipped, the second command has no file to run.

## Slow walkthrough: a bad `dump` argument

v1.0 also tightens beginner-facing error behavior. For example:

```sh
./emulator dump examples/v0_1/add.bin -1 16
```

Earlier, a negative string could accidentally pass through unsigned parsing on some C libraries. v1.0 treats signed-looking dump values as invalid input.

The important lesson is not just “reject `-1`.” The important lesson is:

```text
user input should fail clearly before it becomes a huge wrapped address
```

`dump` accepts decimal and `0x`-prefixed hexadecimal values:

```sh
./emulator dump examples/v0_1/add.bin 4096 16
./emulator dump examples/v0_1/add.bin 0x1000 16
```

## Known limitations remain part of the release

v1.0 is stable, not complete. It still intentionally does not support:

- dynamic linking,
- PIE / `ET_DYN`,
- relocations,
- normal hosted C startup,
- `printf`,
- `malloc`,
- real Linux syscall handling beyond the fake `write` and `exit` ABI,
- `argv`, `envp`, or auxiliary vectors,
- MMU/page-table behavior,
- memory protection enforcement,
- floating point, SIMD, or NEON,
- Mach-O loading.

This honesty is part of the teaching design. A learner should know exactly where the emulator stops.

## What changed inside the project

v1.0 development work is mostly polish:

- command usage now advertises `<program>` instead of only `<raw-binary>`,
- `emulator help` prints the same usage text successfully,
- `dump` rejects signed-looking addresses and lengths,
- the README links the v1.0 lesson and release test plan,
- `examples/v1_0/` contains a smoke manifest for representative release examples,
- `make release-check` is a named release gate alias for the full test suite.

No new architecture subsystem is added. That is intentional.

## Common mistakes

### Mistake 1: thinking v1.0 means “complete ARM64 emulator”

v1.0 means stable learning release, not complete architecture support.

### Mistake 2: thinking optional C examples are required for `make test`

The real C examples are useful, but optional. The main deterministic test suite does not depend on an installed cross compiler.

### Mistake 3: confusing `<program>` with only raw `.bin`

In v1.0, `<program>` means raw `.bin` **or** supported ELF64 `ET_EXEC`.

### Mistake 4: assuming unsupported hosted C should work

Programs that need libc startup, dynamic linking, `printf`, or `malloc` are still outside the emulator’s contract.

## Exercises

1. Run `./emulator help` and identify the five execution/inspection modes.
2. Run one raw example, one ELF example, and one debugger script.
3. Try `dump` with both `4096` and `0x1000` as the address.
4. Try a negative `dump` address and read the error before fixing it.
5. Read `examples/v1_0/smoke_manifest.txt` and explain what each line is meant to represent.

## One-sentence summary

v1.0 does not expand the emulator into a full operating-system emulator; it stabilizes the existing raw, debugger, syscall, ELF, and tiny-C learning path so a new user can build it, run it, inspect it, and understand its limits.
