# v1.0 Lesson — Stable Learning Emulator

This guide is beginner-first. It continues the learning chain from v0.1 through v0.9 and explains why v1.0 is a stability milestone instead of another big CPU-feature milestone.

In v0.9, the emulator learned to run tiny freestanding C programs compiled to AArch64 ELF64:

```text
C source -> object file -> static ELF64 executable -> emulator
```

That completed the first big learning ladder:

```text
raw instructions -> branches -> memory -> calls -> debugger -> readable traces -> fake syscalls -> ELF -> tiny C
```

v1.0 adds one new ability:

```text
A new learner can trust the whole supported path.
```

That may sound less exciting than adding a new instruction, but it matters. A teaching emulator is only useful if the commands, examples, docs, errors, and tests all agree with each other.

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
v1.0: the whole learning path becomes a stable release surface
```

Earlier versions asked:

```text
Can the emulator do one more thing?
```

v1.0 asks a different question:

```text
Can someone clone the project, follow the docs, run the examples,
understand the limits, and get clear errors when something goes wrong?
```

That is the v1.0 mental model.

---

## 2. What v1.0 does and does not add

v1.0 does **not** add a large new architecture subsystem.

It does not add:

```text
new operating-system behavior
new executable format support
new libc support
a full ARM64 instruction set
```

Instead, v1.0 stabilizes the project around the features that already exist.

The new ability is confidence:

```text
clone -> build -> test -> run examples -> debug examples -> understand the limits
```

For a learner, this means:

```text
README wording should match the code.
usage text should explain the supported commands.
errors should explain what went wrong.
examples should represent the supported features.
optional tools should skip clearly.
generated files should clean up correctly.
known limitations should be honest and easy to find.
```

In other words, v1.0 turns a sequence of learning prototypes into a stable learning emulator.

---

## 3. The stable command surface

The public command shape is:

```sh
./emulator run <program>
./emulator trace <program>
./emulator regs <program>
./emulator dump <program> <address> <length>
./emulator debug <program>
./emulator help
```

The important v1.0 wording is `<program>`.

In v0.1 through v0.7, the lessons often use `<raw-binary>` because those versions only loaded raw `.bin` files.

By v1.0, a program can be either:

```text
raw .bin file
    loaded at 0x1000

supported ELF64 ET_EXEC file
    loaded through PT_LOAD program headers
```

So this older wording is still useful when studying early lessons:

```sh
./emulator run <raw-binary>
```

But the stable v1.0 wording is broader:

```sh
./emulator run <program>
```

A beginner-friendly command should tell you the contract without making you remember the whole project history.

## 4. Slow walkthrough: from help text to a real run

Start with help:

```sh
./emulator help
```

The help text is the front door of the project. It should answer:

```text
What commands exist?
What arguments do they take?
What file formats are supported?
What is still unsupported?
```

Then build the deterministic suite:

```sh
make clean
make test
```

This checks the supported behavior from v0.1 through v0.9.

Then build examples:

```sh
make examples
```

Finally, run one small example:

```sh
./emulator run examples/v0_1/add.bin
```

This simple path proves the most basic release promise:

```text
A new learner can build the project and run the first example.
```

That promise is small, but everything else depends on it.

## 5. The release smoke path

v1.0 includes a smoke manifest in:

```text
examples/v1_0/smoke_manifest.txt
```

A smoke test is not meant to prove every edge case. It is a quick representative path through the project.

A good v1.0 smoke path touches each layer of the learning ladder:

```sh
./emulator run examples/v0_1/add.bin
./emulator trace examples/v0_2/trace_loop.bin
./emulator dump examples/v0_3/memory_store_load.bin 0xffff8 8
./emulator debug examples/v0_1/add.bin < examples/v0_5/debug_add_script.txt
./emulator run examples/v0_7/hello.bin
./emulator run examples/v0_8/hello_elf.elf
./emulator run examples/v0_9/return_42.elf
```

Each command represents a different layer:

| Command | What it represents |
|---|---|
| `run examples/v0_1/add.bin` | raw instruction execution still works |
| `trace examples/v0_2/trace_loop.bin` | branches plus readable traces still work |
| `dump examples/v0_3/memory_store_load.bin ...` | memory inspection still works |
| `debug ... < debug_add_script.txt` | debugger scripting still works |
| `run examples/v0_7/hello.bin` | fake syscall output still works |
| `run examples/v0_8/hello_elf.elf` | ELF loading still works |
| `run examples/v0_9/return_42.elf` | tiny C ELF path still works when built |

The smoke path is useful because it tells a story:

```text
The old beginner examples still work,
and the newer ELF/C examples still fit into the same command surface.
```

## 6. Optional v0.9 C examples

The v0.9 C examples need extra tools:

```text
clang --target=aarch64-none-elf
ld.lld
```

Those tools are optional for the deterministic test suite. The v0.9 automated tests can generate ELF fixtures directly, so `make test` does not need a cross compiler.

But `make examples` may try to build real C examples. If the required tools are unavailable, the recipe prints a skip message.

A skip message means:

```text
The .elf file was not produced.
```

So this sequence can fail for an understandable reason:

```sh
make examples/v0_9/fib.elf
./emulator run examples/v0_9/fib.elf
```

If the first command skipped, the second command has no file to run.

This is not an emulator failure. It means the optional external build tool was missing.

The v1.0 goal is that this situation should be clear, not mysterious.

## 7. Slow walkthrough: a bad `dump` argument

v1.0 tightens beginner-facing error behavior.

Consider this command:

```sh
./emulator dump examples/v0_1/add.bin -1 16
```

The user probably means an invalid address. But in C, careless unsigned parsing can accidentally turn `-1` into a huge positive number.

v1.0 rejects signed-looking dump values before they become wrapped addresses.

The important lesson is:

```text
Command-line input should fail clearly before it becomes confusing internal state.
```

Valid dump values can still be decimal:

```sh
./emulator dump examples/v0_1/add.bin 4096 16
```

or hexadecimal with `0x`:

```sh
./emulator dump examples/v0_1/add.bin 0x1000 16
```

The command should accept clear valid input and reject confusing invalid input.

That is v1.0-style polish.

---

## 8. Known limitations are part of the lesson

v1.0 is stable, not complete.

It still intentionally does not support:

- dynamic linking,
- PIE / `ET_DYN`,
- relocations,
- normal hosted C startup,
- `printf`,
- `malloc`,
- real Linux syscall handling beyond the fake `write` and `exit` ABI,
- `argv`, `envp`, or auxiliary vectors,
- MMU/page-table behavior,
- memory protection enforcement,
- floating point, SIMD, or NEON,
- Mach-O loading.

This honesty is part of the teaching design.

A learner should not have to guess whether something failed because:

```text
the program is wrong,
the emulator has a bug,
or the feature is intentionally unsupported.
```

v1.0 should make the third category visible.

## 9. What changed inside the project

v1.0 development work is mostly release polish.

Conceptually, the flow is:

```text
main.c
  parses command-line arguments
  prints stable usage/help text
  rejects confusing dump values
  calls the existing emulator/debugger paths

Makefile
  keeps the deterministic test command easy to run
  adds a named release-check target
  adds release docs, hygiene, and archive-build checks

README + lessons + examples
  describe the same supported surface as the code
```

The user-facing changes are:

```text
./emulator help works
usage text says <program>
dump rejects signed-looking address/length values
make release-check names the release gate
make release-archive packages committed HEAD with .git included
examples/v1_0/ documents a representative smoke path
known limitations are collected clearly
```

No new instruction family is added here. No new loader format is added here. No new syscall is added here.

That restraint is intentional: v1.0 is about making the existing learning path dependable.

## 10. How to read v1.0 errors

A good v1.0 error should help you decide what to do next.

For example:

| Situation | Helpful interpretation |
|---|---|
| missing file | check the path or build the example first |
| unsupported ELF type | the file is outside the v0.8/v1.0 ELF contract |
| unsupported instruction | the program uses an instruction the teaching CPU has not implemented |
| invalid dump address | the CLI argument itself is not valid |
| invalid guest memory | the guest program asked the emulator to read/write outside guest memory |
| optional C example skipped | the cross compiler/linker was unavailable |

The emulator is still small. It cannot fix those situations automatically.

But by v1.0, it should make them understandable.

## 11. Commands to try

Print help:

```sh
./emulator help
```

Run the deterministic suite:

```sh
make test
```

Run the named release gate:

```sh
make release-check
```

That target runs the deterministic v0.1-v0.9 suite and then runs v1.0 release helper scripts for documentation status, repository hygiene, and fresh-archive build checks. Dedicated `tests/v1_0/` wrappers can be added later, but the release surface is already exposed through the named gate.

Create a release archive from committed `HEAD`:

```sh
make release-archive
```

This produces an `emulator_<timestamp>.zip` archive and includes `.git`, matching the project handoff convention. Because it packages `HEAD`, commit your changes before using it as the final handoff archive.

Build examples:

```sh
make examples
```

Run representative examples:

```sh
./emulator run examples/v0_1/add.bin
./emulator trace examples/v0_2/trace_loop.bin
./emulator run examples/v0_7/hello.bin
./emulator run examples/v0_8/hello_elf.elf
./emulator run examples/v0_9/return_42.elf
```

Try valid dump arguments:

```sh
./emulator dump examples/v0_1/add.bin 4096 16
./emulator dump examples/v0_1/add.bin 0x1000 16
```

Try an invalid dump argument:

```sh
./emulator dump examples/v0_1/add.bin -1 16
```

Then read the error and explain why rejecting it early is better than treating it as a huge address.

## 12. Common mistakes

### Mistake 1: thinking v1.0 means “complete ARM64 emulator”

v1.0 means stable learning release, not complete architecture support.

The emulator is still intentionally small.

### Mistake 2: thinking v1.0 adds a new CPU subsystem

v0.2 added control flow. v0.3 added memory. v0.8 added ELF. v0.9 added tiny C support.

v1.0 is different: it stabilizes the path instead of adding a new CPU concept.

### Mistake 3: thinking optional C examples are required for `make test`

The real C examples are useful, but optional. The main deterministic test suite does not depend on an installed cross compiler.

### Mistake 4: confusing `<program>` with only raw `.bin`

In v1.0, `<program>` means raw `.bin` **or** supported ELF64 `ET_EXEC`.

### Mistake 5: assuming unsupported hosted C should work

Programs that need libc startup, dynamic linking, `printf`, or `malloc` are still outside the emulator’s contract.

### Mistake 6: treating a skipped optional build as a successful output file

If a v0.9 example recipe prints a skip message, the target `.elf` was not produced. Build tools must be installed before that file can be run.

## 13. Exercises

1. Run `./emulator help` and list the five execution/inspection modes: `run`, `trace`, `regs`, `dump`, and `debug`.
2. Run one raw `.bin` example and one ELF example. Explain how the loader treats them differently.
3. Run `./emulator trace examples/v0_2/trace_loop.bin` and point to one repeated `pc` address.
4. Run `./emulator dump examples/v0_1/add.bin 4096 16`, then run the same command with `0x1000`. Explain why the output should match.
5. Try a negative `dump` address and explain why v1.0 rejects it before execution starts.
6. Read `examples/v1_0/smoke_manifest.txt` and explain which learning layer each command represents.
7. Read the known limitations list and choose one unsupported feature. Explain why supporting it would be a future version, not a v1.0 polish task.

---

## One-sentence summary

v1.0 does not expand the emulator into a full operating-system emulator; it stabilizes the existing raw, debugger, syscall, ELF, and tiny-C learning path so a new user can build it, run it, inspect it, and understand its limits.
