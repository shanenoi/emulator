# v1.0 Lesson — Stable Learning Emulator

This guide is beginner-first. It continues the learning chain from `lessons/v0.1-instruction-sandbox.md` through `lessons/v0.9-tiny-c-programs.md`.

v1.0 is different from the earlier versions.

Most earlier versions added a new emulator capability:

```text
v0.2 added branches.
v0.3 added memory and stack.
v0.7 added toy syscalls.
v0.8 added ELF loading.
v0.9 added tiny freestanding C programs.
```

v1.0 does not try to make the emulator bigger in that way.

v1.0 asks a release question instead:

```text
Can a new learner trust the supported path?
```

The short version:

```text
v1.0 turns the v0.1-v0.9 learning emulator into a stable teaching release.

It checks that the commands, examples, docs, tests, errors, cleanup rules,
and release archive all agree with each other.
```

That may sound less exciting than adding a new instruction, but it matters. A learning project becomes hard to trust when the code says one thing, the README says another thing, examples leave generated files behind, or errors are confusing.

v1.0 is about removing that confusion.

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
v1.0: the whole learning path becomes stable enough to release from
```

The old mental model still applies:

```text
program file
    -> loader copies bytes into fake memory
    -> loader chooses initial pc and sp
    -> CPU fetches one 32-bit instruction at pc
    -> CPU decodes and executes it
    -> debugger/trace/syscalls help us observe what happened
```

v1.0 does not replace that model.

It makes the model dependable.

---

## 2. What “stable” means in this project

In a large production emulator, “stable” might mean thousands of instructions, real process loading, memory protection, threads, signals, and operating-system compatibility.

That is **not** what v1.0 means here.

In this teaching project, stable means:

```text
The supported beginner workflow is predictable.
```

More concretely:

```text
README links point to real files.
lesson docs match the code.
examples build or skip clearly.
make test is deterministic.
make clean removes generated artifacts.
release checks catch stale docs and packaging mistakes.
errors explain unsupported input instead of failing mysteriously.
```

The v1.0 goal is not:

```text
Run every ARM64 program.
Run Linux.
Run macOS.
Run normal hosted C with printf and malloc.
```

The v1.0 goal is:

```text
Run the supported examples reliably and explain the unsupported ones clearly.
```

That difference is important.

## 3. The supported program types at v1.0

Earlier lessons often said `<raw-binary>` because the emulator originally loaded only raw `.bin` files.

By v1.0, the public wording is broader:

```sh
./emulator run <program>
./emulator trace <program>
./emulator regs <program>
./emulator dump <program> <address> <length>
./emulator debug <program>
./emulator help
```

The word `<program>` matters.

At v1.0, a supported program can be:

```text
1. a raw .bin file
2. a supported ELF64 ET_EXEC file
```

These two file types are loaded differently.

### Raw `.bin`

A raw `.bin` file has no metadata.

It does not say:

```text
where it should be loaded
where execution should begin
which bytes are code
which bytes are data
which CPU it was built for
```

So the emulator uses a fixed rule:

```text
copy the whole file to guest address 0x1000
set pc = 0x1000
set sp = 0x100000
```

This is perfect for early lessons because it is easy to see.

### ELF64 `ET_EXEC`

An ELF file has metadata.

The loader can read:

```text
ELF header
program headers
PT_LOAD segment descriptions
entry point e_entry
```

So the emulator follows the file's own loading instructions:

```text
copy each PT_LOAD segment to its p_vaddr
zero-fill bytes where p_memsz > p_filesz
set pc = e_entry
set sp = 0x100000
```

The important contrast is:

```text
raw .bin: the emulator invents the loading rule
ELF64:    the file describes the loading rule
```

v1.0 makes sure both paths are documented, tested, and reachable from the same command surface.

## 4. The important zero-fill idea

Your earlier answers showed that zero-fill deserves a slower explanation.

Zero-fill appears in v0.8 ELF loading, stays important in v1.0 stability, and returns again in v1.1 Mach-O loading.

Here is the problem it solves.

A program sometimes needs memory that should exist when it starts, but that memory does not need meaningful bytes stored in the executable file.

For example, imagine a program wants this global buffer:

```c
static char buffer[16];
```

At program start, C expects those bytes to be zero:

```text
00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
```

The executable file does not need to physically store sixteen zero bytes. It can say something like:

```text
I have 4 bytes of initialized file data,
but I need 20 bytes of memory at runtime.
```

That means:

```text
file size   = 4 bytes
memory size = 20 bytes
```

The loader performs two steps:

```text
1. copy the 4 real file bytes into guest memory
2. fill the remaining 16 guest memory bytes with zero
```

Visually:

```text
Executable file bytes:

file offset
0x200:  aa bb cc dd

Loaded guest memory:

guest address
0x4000: aa bb cc dd 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
        ^^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
        copied      zero-filled by the loader
```

This is why loader records have two sizes.

For ELF, the names are:

```text
p_filesz = how many bytes exist in the file
p_memsz  = how many bytes should exist in memory
```

The rule is:

```text
copy p_filesz bytes
then zero-fill p_memsz - p_filesz bytes
```

But only if:

```text
p_memsz >= p_filesz
```

If a file claims:

```text
file size   = 20
memory size = 4
```

that is invalid for this loader because the file is asking the loader to copy more initialized bytes than the mapped memory region can hold.

Zero-fill is not a CPU feature. It happens before the first instruction executes.

The CPU simply sees memory that already contains:

```text
copied bytes + zero bytes
```

## 5. Why release checks matter

By v1.0, the project has enough pieces that a single `make test` is necessary but not enough.

The project also needs release checks.

Think of the checks as different questions.

```text
make test
    Do the deterministic behavior tests pass?

make release-docs-check
    Do the required docs, lessons, test plans, and examples exist?
    Do README claims match the implemented versions?

make release-hygiene-check
    Are unwanted generated files or forbidden directories present?

make release-clean-check
    Does make clean remove generated artifacts after examples/tests run?

make release-archive-check
    Can a fresh archive from committed HEAD pass the deterministic suite?

make release-check
    Run the release gate that combines those promises.
```

The mental model is:

```text
tests protect emulator behavior
docs checks protect learner instructions
clean checks protect repository hygiene
archive checks protect handoff reproducibility
```

A teaching project especially needs this because learners often follow commands exactly.

If the README says:

```sh
make examples
```

but the examples leave confusing temporary files behind, the learner may not know whether they made a mistake.

v1.0 tries to make the supported path boring and reliable.

That is a good thing.

## 6. Deterministic fixtures

Tests need input programs.

There are two broad ways to get those programs:

```text
1. build them using external tools
2. generate known byte sequences inside the test/project
```

External tools are useful, but they are not always installed.

For example, tiny C examples may need:

```text
clang --target=aarch64-none-elf
ld.lld
```

If those tools are missing, an optional example can skip clearly.

But deterministic tests should not depend on a learner's local toolchain.

That is why fixtures matter.

A deterministic fixture is an input file produced the same way every time:

```text
same generator
same bytes
same expected behavior
same cleanup rule
```

Good fixture behavior looks like this:

```text
make test creates exactly the files it needs
the generated bytes are predictable
make clean removes those generated files
release checks verify the cleanup
```

This makes failures easier to understand.

If a deterministic fixture fails, the cause is likely inside the emulator, test, or generator.

If a toolchain-built optional example fails, the cause may simply be the host environment.

v1.0 keeps those categories separate.

## 7. The v1.0 smoke path

v1.0 includes a representative smoke manifest:

```text
examples/v1_0/smoke_manifest.txt
```

A smoke path is not meant to test every edge case. It is meant to touch the main learning layers quickly.

Representative commands include:

```sh
./emulator run examples/v0_1/add.bin
./emulator trace examples/v0_2/trace_loop.bin
./emulator dump examples/v0_3/memory_store_load.bin 0xffff8 8
./emulator debug examples/v0_1/add.bin < examples/v0_5/debug_add_script.txt
./emulator run examples/v0_7/hello.bin
./emulator run examples/v0_8/hello_elf.elf
./emulator run examples/v0_9/return_42.elf
```

Each command represents a layer:

| Command | What it proves at a high level |
|---|---|
| `run examples/v0_1/add.bin` | raw instruction execution still works |
| `trace examples/v0_2/trace_loop.bin` | branches and readable traces still work |
| `dump examples/v0_3/memory_store_load.bin ...` | memory inspection still works |
| `debug ... < debug_add_script.txt` | debugger scripting still works |
| `run examples/v0_7/hello.bin` | fake syscall output still works |
| `run examples/v0_8/hello_elf.elf` | ELF loading still works |
| `run examples/v0_9/return_42.elf` | tiny C ELF path still works when built |

The story is:

```text
The early raw-binary lessons still work,
and the later ELF/C lessons still fit into the same command-line tool.
```

## 8. Optional examples and clear skips

The v0.9 C examples are useful, but they may need external tools.

If the tools are unavailable, the project should not pretend that everything succeeded.

A clear skip means:

```text
This optional artifact was not built because a required host tool was missing.
```

For example, this may skip:

```sh
make examples/v0_9/fib.elf
```

If it skips, this later command cannot work because the file does not exist:

```sh
./emulator run examples/v0_9/fib.elf
```

That is not an emulator execution failure.

It is an optional-build availability issue.

v1.0's job is to make that difference visible.

## 9. Beginner-friendly CLI errors

v1.0 also improves command-line error behavior.

Consider this command:

```sh
./emulator dump examples/v0_1/add.bin -1 16
```

The user probably means an invalid address.

But C programs often parse numbers into unsigned integer types. If a program is careless, `-1` can accidentally become a very large positive value.

That would produce a confusing later error.

v1.0 rejects signed-looking dump values early.

The lesson is:

```text
Bad CLI input should fail as CLI input.
It should not be allowed to become confusing emulator state.
```

Valid values still work:

```sh
./emulator dump examples/v0_1/add.bin 4096 16
./emulator dump examples/v0_1/add.bin 0x1000 16
```

Both `4096` and `0x1000` describe the same address.

The stable behavior is:

```text
accept clear valid input
reject confusing invalid input
explain the problem before execution starts
```

## 10. Error categories

A good v1.0 error helps you decide where to look next.

Here are useful categories:

| Category | Example | Where to investigate |
|---|---|---|
| CLI input error | bad `dump` address | command arguments |
| file/path error | missing example file | build command or path |
| loader error | unsupported ELF type | executable format |
| memory error | guest address out of range | guest program or loader mapping |
| CPU decode error | unsupported instruction | instruction support |
| syscall error | unsupported fake syscall number | guest ABI/register setup |
| optional tool skip | missing cross compiler | host environment |

This separation is part of the v1.0 teaching standard.

Without clear categories, a learner might see one failure and wonder:

```text
Did I write bad assembly?
Did the loader map the file wrong?
Did the CPU decode the instruction wrong?
Did the host toolchain fail?
Did the docs lie?
```

v1.0 cannot prevent every mistake.

But it should make the next question clearer.

## 11. Known limitations are part of the release

v1.0 is stable, not complete.

It still intentionally does not support:

- dynamic linking,
- PIE / `ET_DYN`,
- relocations,
- normal hosted C startup,
- `printf`,
- `malloc`,
- real Linux syscall handling beyond the fake `write` and `exit` ABI,
- `argv`, `envp`, or auxiliary vectors,
- MMU/page-table behavior,
- memory protection enforcement,
- floating point, SIMD, or NEON,
- Mach-O loading.

This list is not embarrassing. It is a teaching boundary.

A good limitation list prevents a common beginner trap:

```text
I tried a normal program, it failed, therefore the emulator is broken.
```

The better interpretation is often:

```text
I tried a program outside the supported teaching profile.
```

v1.0 makes the supported profile easier to see.

## 12. What changed inside the project

The v1.0 code and repository changes are mostly about polish and release discipline.

Conceptually:

```text
main.c
    prints stable help text
    parses command-line modes consistently
    rejects confusing dump arguments
    routes supported commands to loader/emulator/debugger paths

Makefile
    keeps make test easy
    adds named release-check targets
    checks docs, hygiene, cleanup, and fresh archive behavior

README, lessons, examples, test plans
    describe the same supported surface as the code
```

The user-facing result is:

```text
./emulator help works as the front door
<program> means raw .bin or supported ELF64
make test is the normal deterministic test path
make release-check is the release gate
make release-archive packages committed HEAD with .git included
examples/v1_0/ documents a representative smoke path
known limitations are visible
```

No large CPU subsystem is added.

No new executable format is added in v1.0.

That restraint is the point.

## 14. Common mistakes

### Mistake 1: thinking v1.0 means “complete ARM64 emulator”

v1.0 means stable teaching release.

It does not mean complete architecture coverage.

### Mistake 2: thinking v1.0 adds a new CPU concept

v1.0 is not like v0.2, v0.3, or v0.7.

It stabilizes the already-supported path.

### Mistake 3: confusing optional examples with deterministic tests

Optional examples may depend on local tools.

Deterministic tests should not.

### Mistake 4: treating `<program>` as only raw `.bin`

At v1.0, `<program>` means raw `.bin` or supported ELF64 `ET_EXEC`.

### Mistake 5: expecting normal hosted C

v0.9 supports tiny freestanding C, not normal hosted C with libc startup, `printf`, `malloc`, `argv`, or `envp`.

### Mistake 6: forgetting zero-fill happens before execution

Zero-fill is loader behavior.

The CPU does not run instructions to create those zero bytes. The loader prepares memory before the first fetch.
