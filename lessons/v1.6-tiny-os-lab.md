# v1.6 Lesson — Tiny OS Lab and Guest-Managed Tasks

## 1. The learning ladder so far

The emulator has grown in deliberately small steps.

v0.1 started with the smallest possible CPU loop: fetch, decode, execute, and
stop on `HLT`.

v0.2 made `pc` programmable with branches, loops, comparisons, and flags.

v0.3 added memory and the stack, so programs could store values outside
registers.

v0.4 added function calls. `BL`, `RET`, and `x30` made temporary control-flow
transfers visible.

v0.5 added the debugger REPL, turning CPU and memory state into something a
learner can inspect interactively.

v0.6 made traces and disassembly readable.

v0.7 added toy syscalls. `SVC #0` became a controlled doorway from guest code to
host-provided services.

v0.8 added ELF64 loading.

v0.9 showed how tiny freestanding C programs can target the emulator.

v1.0 stabilized release checks and documentation.

v1.1 added a small Mach-O teaching loader.

v1.2 introduced mapped memory and permissions.

v1.3 added memory-mapped devices.

v1.4 added exception and interrupt-shaped control flow.

v1.5 introduced the first toy-kernel profile: the host could configure tasks,
the kernel could start them, and tasks could yield, sleep, exit, fault, or panic
through deterministic `BRK` traps.

v1.6 begins the next step: the guest kernel can now create and inspect tasks at
runtime.

## 2. Why v1.6 exists

v1.5 proves that the emulator can save a task, restore another task, and keep a
small scheduler state machine deterministic.

But v1.5 is still mostly host-managed:

```text
host CLI/API creates tasks
guest kernel starts scheduler
tasks yield/exit/sleep
```

That is useful for testing the scheduler, but it does not yet feel like an OS
lab. A tiny kernel should be able to ask for a task to be created, inspect the
task table, and pass small messages between tasks.

v1.6 changes the mental model to:

```text
host loads kernel
kernel reads boot info
kernel creates tasks
tasks call kernel services
scheduler runs tasks deterministically
kernel/debugger can inspect task state
```

It is still not a real operating system. It is a small, deterministic teaching
model for OS concepts.

## 3. What v1.6 intentionally does not emulate

v1.6 does not add real ARM exception levels, real EL0/EL1 isolation, a real MMU
per process, a filesystem, networking, dynamic linking, SMP, atomics, or host
threads.

The emulator remains single-threaded. The scheduler remains deterministic. Task
switches still happen only through documented teaching traps or instruction-count
timer events.

## 4. The new service trap

v1.5 already reserved these toy-kernel traps:

```text
BRK #0x150  yield
BRK #0x151  task exit
BRK #0x152  kernel panic
BRK #0x153  console write
BRK #0x154  start tasks
BRK #0x155  sleep
```

v1.6 keeps all of those stable and adds one new dispatcher:

```text
BRK #0x160  toy-kernel service dispatcher
```

The convention is intentionally simple:

```text
x8  service ID
x0  first argument, then return value
x1  second argument or secondary return value
x2  third argument
x3  fourth argument
x4  fifth argument
x5  optional name/buffer pointer
x6  optional name/buffer length
```

The result in `x0` is either a nonnegative success value or a negative service
error encoded in the 64-bit register.

## 5. Service IDs

The first-pass v1.6 service table is:

```text
1   TASK_CREATE
2   TASK_YIELD
3   TASK_EXIT
4   TASK_SLEEP
5   TASK_GET_ID
6   TASK_GET_INFO
7   TASK_SEND
8   TASK_RECV
9   CONSOLE_WRITE
10  KERNEL_PANIC
```

The service errors are:

```text
-1  UNKNOWN
-2  BAD_ARGUMENT
-3  NO_SLOT
-4  BAD_ENTRY
-5  BAD_STACK
-6  BAD_FLAGS
-7  NOT_FOUND
-8  WOULD_BLOCK
```

## 6. Creating a task from the guest

`TASK_CREATE` uses this register contract:

```text
x8 = 1
x0 = entry pc
x1 = stack base, or 0 for emulator-allocated default stack
x2 = stack size, or 0 for emulator-allocated default stack
x3 = initial task x0 argument
x4 = flags, currently must be 0
x5 = optional task label pointer
x6 = optional task label length
BRK #0x160
```

On success, `x0` becomes the deterministic task ID.

For the first pass, `x1 == 0` and `x2 == 0` mean: allocate the same style of
16 KiB task stack that v1.5 host-created tasks use.

If the kernel provides its own stack range, the emulator validates that it is
writable ordinary guest memory, does not wrap around the address space, and does
not overlap boot-info, the descriptor table, or another live task stack.

The task entry must be 4-byte aligned, executable guest RAM, and not a device
range.

## 7. Task IDs and labels

Task IDs are deterministic. They are assigned in creation order.

The first-pass policy is simple: IDs are not reused. This keeps traces and
debugger output easy to reason about.

Task labels are optional and truncated to 15 visible bytes plus a terminator.
They are teaching labels, not security boundaries.

## 8. Boot-info version 2

When `--kernel-boot-info` is enabled, the boot-info block now uses version `2`.

The original v1.5 fields remain first so old teaching code that reads memory
size, stack base, task count, or device bases still sees the same layout before
the extension.

v1.6 appends:

```text
descriptor_table_address
descriptor_count
descriptor_size
service_trap
mailbox_slots
mailbox_message_size
```

The descriptor table address is currently `0x00081000`.

## 9. Guest-visible task descriptors

Each task descriptor contains:

```text
magic
version
descriptor_size
task_id
state
entry_pc
saved_pc
saved_sp
stack_base
stack_size
exit_code
fault_cause
fault_address
wake_tick
switch_count
name[16]
```

The descriptor table is a teaching aid. The emulator still uses its internal C
state as the source of truth, then refreshes the guest-visible descriptors after
task creation, switches, exits, sleeps, wakeups, and faults.

## 10. Scheduler behavior

Guest-created tasks join the same scheduler as v1.5 host-created tasks.

That means:

```text
READY tasks can run
RUNNING is the current task
BLOCKED tasks wait for timer ticks
EXITED tasks are skipped
FAULTED tasks are skipped
```

The scheduler remains round-robin and deterministic.

The `TASK_YIELD`, `TASK_EXIT`, and `TASK_SLEEP` services intentionally mirror the
older v1.5 direct traps. This gives learners two ways to discuss the same idea:
direct trap numbers first, then a service table later.

## 11. Getting task information

`TASK_GET_ID` returns the current task ID when called from a running task.

`TASK_GET_INFO` takes a task ID in `x0` and returns the descriptor pointer in
`x0`, or `NOT_FOUND` if no task with that ID exists.

This makes small guest kernels easier to write because they do not need to know
the descriptor-array index ahead of time.

## 12. Tiny mailbox IPC

Each task has a small mailbox:

```text
slots:        4
message size: 32 bytes
ordering:     FIFO
blocking:     no, first pass is nonblocking
```

`TASK_SEND` uses:

```text
x8 = 7
x0 = destination task ID
x1 = source buffer
x2 = byte length, 0..32
BRK #0x160
```

`TASK_RECV` uses:

```text
x8 = 8
x0 = destination buffer
x1 = maximum bytes to copy
BRK #0x160
```

On receive success, `x0` is the number of copied bytes and `x1` is the sender
task ID. Empty receive and full send return `WOULD_BLOCK`.

## 13. Debugging v1.6

`emulator info` now prints the descriptor table, service trap, next task ID,
task labels, origin (`host` or `guest`), switch counts, and mailbox counts.

The debugger `kernel` command prints the same class of information while the
program is stopped.

Trace output includes task creation events:

```text
trace kernel-service create-task id=0 index=0 entry=0x0000000000001020
```

## 14. Common mistakes

Using an unmapped or non-executable task entry returns a task-create error.

Using a stack that overlaps another task stack, boot-info, or the descriptor
table returns a stack error.

Using unsupported flags returns `BAD_FLAGS`.

Receiving from an empty mailbox is not a crash; it returns `WOULD_BLOCK`.

Forgetting to start the scheduler after creating tasks means the kernel keeps
executing its own instructions instead of switching into tasks.

## 15. Exercises

1. Create one task from guest code, pass `x3 = 42`, and confirm the task sees
   `x0 == 42` at entry.
2. Create two tasks, have both call `TASK_YIELD`, and compare the trace with the
   v1.5 direct `BRK #0x150` yield trace.
3. Have task A send a short message to task B, then have task B receive it and
   print it through `CONSOLE_WRITE`.
4. Query `TASK_GET_INFO` for a task after it exits and inspect the descriptor's
   exit code.
5. Try to create a task with unsupported flags and observe the stable
   `BAD_FLAGS` result instead of a host crash.

## 16. Summary

v1.6 is the first step from "the host configured tasks" toward "the guest kernel
manages tasks." It adds a service dispatcher, guest task creation, task
descriptors, task IDs, and tiny mailbox IPC while preserving the deterministic
toy-kernel scheduler from v1.5.
