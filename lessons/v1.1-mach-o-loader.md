# v1.1 Lesson — Mach-O Loader

This guide is beginner-first. It continues the learning chain from v0.1 through v1.0 and explains how the emulator starts to understand Apple-style executable files without becoming a macOS or iOS emulator.

In v1.0, the project became a stable learning emulator:

```text
clone -> build -> test -> run examples -> debug examples -> understand the limits
```

v1.1 adds one new ability:

```text
The emulator can recognize and load a tiny, static, arm64 Mach-O executable profile.
```

That sentence is intentionally narrow. v1.1 does **not** mean normal macOS apps, iOS apps, Swift apps, Objective-C apps, shared libraries, `dyld`, or Darwin syscalls work. It means the loader can take one more step up the binary-format ladder:

```text
raw bytes -> ELF64 executable -> Mach-O executable profile
```

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
v1.0: the whole learning path becomes a stable release surface
v1.1: the loader can start reading simple arm64 Mach-O executables
```

The important pattern is that each version adds one concept without hiding the earlier concepts.

v1.1 is a loader milestone, like v0.8. The CPU is still the same teaching CPU. The memory model is still the same flat 1 MiB byte array. The fake syscall interface is still the same tiny interface:

```text
x8 = 64  -> fake write
x8 = 93  -> fake exit
svc #0   -> ask the emulator to handle the request
```

What changes is how the emulator decides where the program bytes go in memory and where execution begins.

---

## 2. Why Mach-O comes after ELF

v0.8 introduced ELF because ELF is a clear way to learn executable loading:

```text
ELF header -> program headers -> PT_LOAD segments -> entry point
```

Mach-O teaches the same broad idea with different names:

```text
Mach-O header -> load commands -> LC_SEGMENT_64 segments -> LC_MAIN entry offset
```

Both formats answer similar loader questions:

```text
What kind of file is this?
What CPU architecture is it for?
Which bytes should be copied into guest memory?
Where should those bytes go?
Where should pc start?
Which features require a real operating system loader?
```

The file format changes, but the learning goal stays the same. You are learning to separate three ideas:

```text
CPU execution:      fetch/decode/execute one instruction at pc
Memory mapping:     place program bytes at guest addresses
Runtime services:   provide syscalls, dynamic linking, process setup, and libraries
```

v1.1 only grows the second idea.

---

## 3. Raw `.bin`, ELF64, and Mach-O side by side

A raw `.bin` file has no metadata, so the emulator makes a fixed rule:

```text
copy the whole file to 0x1000
set pc = 0x1000
set sp = 0x100000
```

An ELF64 executable has a header and program headers, so the emulator follows `PT_LOAD` records:

```text
copy file bytes from p_offset to p_vaddr
zero-fill if p_memsz > p_filesz
set pc = e_entry
set sp = 0x100000
```

A Mach-O executable has a header and load commands, so the v1.1 emulator follows `LC_SEGMENT_64` records and `LC_MAIN`:

```text
copy file bytes from fileoff to vmaddr
zero-fill if vmsize > filesize
resolve LC_MAIN entryoff to a mapped address
set pc = resolved entry
set sp = 0x100000
```

The shared shape is:

```text
validate metadata
map segments
choose entry point
start CPU
```

That is the main v1.1 mental model.

---

## 4. What Mach-O is

Mach-O is the executable and object-file format used by Apple platforms. The name is short for **Mach object**.

For this project, you do not need to learn every Mach-O feature at once. Start with the pieces the loader now cares about:

```text
mach_header_64
    says this is a 64-bit Mach-O file
    says the CPU type is arm64
    says this is an executable
    says how many load commands follow

load_command
    a typed record after the header
    each command has a command id and command size

LC_SEGMENT_64
    describes bytes to map into memory

LC_MAIN
    describes where execution should begin
```

This is similar to how v0.8 did not teach every ELF feature. It taught the small executable profile the emulator could honestly support. v1.1 does the same for Mach-O.

---

## 5. The v1.1 supported Mach-O profile

The supported profile is deliberately small:

```text
64-bit Mach-O only
little-endian only
arm64 CPU type only
MH_EXECUTE files only
LC_SEGMENT_64 for memory mapping
LC_MAIN for the entry point
no dynamic linker
no shared libraries
no rebasing or binding metadata
no code signing behavior
no Apple process setup
```

This profile is good for tiny teaching fixtures. It is not enough for normal Apple programs because normal Apple programs expect many more services:

```text
dyld
libSystem
Darwin syscalls
argv/envp stack setup
thread-local storage
Objective-C or Swift runtime support
code-signing and platform metadata
shared libraries and frameworks
```

The emulator rejects the most important unsupported runtime features clearly. That is better than pretending to support them and then failing later with confusing CPU errors.

---

## 6. The new loader decision tree

The public loader still starts from one function:

```c
emulator_load_program(...)
```

The decision tree is now:

```text
read the file

if the first bytes are ELF magic:
    use the ELF64 loader

else if the first bytes are Mach-O magic:
    use the Mach-O loader

else:
    use the raw .bin loader
```

This keeps old examples working. A v0.1 raw binary is still just a raw binary. A v0.8 ELF fixture is still loaded by the ELF path. A v1.1 Mach-O fixture now gets a third path.

The CLI does not need a new command:

```sh
./emulator run <program>
./emulator trace <program>
./emulator regs <program>
./emulator dump <program> <address> <length>
./emulator info <program>
./emulator debug <program>
```

`<program>` is broader now:

```text
raw .bin
supported ELF64 ET_EXEC
supported Mach-O arm64 MH_EXECUTE
```

## 7. Header validation

The Mach-O loader first validates the `mach_header_64` fields. The most important checks are:

```text
file is large enough to contain the 64-bit header
magic is the little-endian 64-bit Mach-O magic
CPU type is arm64
file type is executable
load-command count is not zero
load-command table fits inside the file
```

Each check protects the loader from reading nonsense. For example, without the command-table bounds check, a corrupted file could say:

```text
sizeofcmds = enormous number
```

Then the loader might read past the end of the host file buffer. The loader should never trust guest-controlled file metadata until it has checked the relevant ranges.

That lesson was already true in v0.8 ELF loading. v1.1 repeats the lesson with Mach-O names.

---

## 8. Load commands

After the Mach-O header, the file contains load commands. Every load command begins with:

```text
cmd      = what kind of command this is
cmdsize  = how many bytes this command occupies
```

The loader walks from one command to the next by adding `cmdsize`. That means `cmdsize` must be checked carefully:

```text
cmdsize must be at least the basic load-command size
cmdsize must fit inside the file
cmdsize must stay inside the header's sizeofcmds table
the final command walk must end exactly where sizeofcmds says it should end
```

This protects edge cases like:

```text
cmdsize = 0
cmdsize smaller than the command's required structure
cmdsize points beyond the file
ncmds says there are more commands than sizeofcmds can contain
```

Those malformed cases belong in the v1.1 tests later.

---

## 9. Segment mapping with `LC_SEGMENT_64`

`LC_SEGMENT_64` is the Mach-O command that tells the loader how to map bytes. The fields that matter most are:

```text
vmaddr    guest memory address where the segment begins
vmsize    number of guest memory bytes reserved for the segment
fileoff   file offset where the segment's initialized bytes begin
filesize  number of initialized bytes to copy from the file
initprot  requested initial permissions, recorded but not enforced yet
```

The loader applies this rule:

```text
copy bytes[fileoff : fileoff + filesize] to memory[vmaddr : vmaddr + filesize]
if vmsize > filesize:
    set memory[vmaddr + filesize : vmaddr + vmsize] to zero
```

That second part is important. It is how a file can represent memory that should exist at runtime but does not need to occupy bytes in the file.

In ELF lessons, this was `.bss` zero-fill behavior. Mach-O has the same practical need: runtime memory can be larger than initialized file bytes.

v1.1 also validates the `nsects` value inside each `LC_SEGMENT_64` command. The loader does not need section records to copy bytes, but it checks that the section table claimed by the segment fits inside the command. This is inspection-oriented validation: even metadata that is not required for execution should not be allowed to point outside the command that owns it.

---

## 10. Entry point with `LC_MAIN`

ELF has a direct entry field:

```text
e_entry
```

Mach-O commonly uses `LC_MAIN`:

```text
entryoff
```

In v1.1, the loader resolves `entryoff` through the mapped segment file ranges:

```text
find a segment whose file range contains entryoff
entry = segment.vmaddr + (entryoff - segment.fileoff)
```

Then it checks:

```text
entry is 4-byte aligned
entry is inside a loaded segment
```

The 4-byte alignment check matters because AArch64 instructions are 4 bytes wide. Starting at an odd or halfword address would not make sense for this emulator's instruction fetch model.

---

## 11. Why unsupported dynamic features are rejected early

Normal Mach-O executables often contain commands like:

```text
LC_LOAD_DYLINKER
LC_LOAD_DYLIB
LC_DYLD_INFO
LC_DYLD_INFO_ONLY
```

Those commands are clues that the executable expects runtime work outside the tiny teaching loader.

For example:

```text
LC_LOAD_DYLINKER -> expects dyld
LC_LOAD_DYLIB    -> expects shared libraries
LC_DYLD_INFO     -> expects rebasing/binding support
```

The emulator does not provide those services, so v1.1 rejects them with loader errors.

This is an important design habit:

```text
Reject unsupported format/runtime features at load time.
Do not let them become mysterious CPU failures later.
```

---

## 12. What did not change in the CPU

The CPU still does the same core loop:

```text
fetch instruction at pc
decode instruction
execute instruction
advance or change pc
stop on HLT, guest exit, or error
```

Mach-O support does not automatically add new instructions. If a Mach-O fixture contains an unsupported AArch64 instruction, the loader may succeed and the CPU may still reject the instruction during execution.

That is okay. The error category matters:

```text
bad Mach-O metadata       -> loader error
unsupported CPU opcode    -> CPU decode/runtime error
unsupported fake syscall  -> syscall/runtime error
```

Keeping those categories separate helps learners debug the right layer.

---

## 13. What changed inside the project

The main code changes are in the loader and public data model.

The project now has Mach-O constants for:

```text
64-bit Mach-O magic
big-endian 64-bit Mach-O magic, for clear rejection
arm64 CPU type
MH_EXECUTE file type
LC_SEGMENT_64
LC_MAIN
LC_SYMTAB
LC_DYSYMTAB
unsupported dynamic-linking commands
fat/universal Mach-O archive magic values
```

The program format enum now has a Mach-O value:

```text
EMU_PROGRAM_RAW
EMU_PROGRAM_ELF64
EMU_PROGRAM_MACHO64
```

Loaded segments now also remember the source file offset, segment name, and section count. The file offset matters for Mach-O because `LC_MAIN` gives an entry offset that must be translated through the segment's file range.

The CLI also has an inspection command:

```sh
./emulator info <program>
```

For Mach-O files, this prints the detected format, entry point, stack pointer, mapped segments, load-command count, symbol count, indirect-symbol count, and a bounded list of symbol names/addresses when `LC_SYMTAB` is present. It loads and validates the file but does not execute guest instructions.

The memory model did not become more complex. Segment permissions are still recorded for future use, not enforced.

## 14. How to read v1.1 loader errors

A Mach-O loader error usually tells you which validation layer failed.

Examples:

```text
Mach-O loader error: big-endian Mach-O is unsupported in v1.1
```

This means the file is a Mach-O-like file, but not the little-endian profile this version supports.

```text
Mach-O loader error: unsupported CPU type ...; expected arm64
```

This means the file is for a different CPU architecture.

```text
Mach-O loader error: LC_LOAD_DYLIB/shared libraries are unsupported in v1.1
```

This means the file expects dynamic runtime behavior.

```text
Mach-O loader error: LC_MAIN is required in v1.1
```

This means the loader did not find the entry command it currently uses to set `pc`.

When reading an error, ask:

```text
Did the file fail format validation?
Did it request unsupported runtime behavior?
Did it map bytes outside emulator memory?
Did it choose an invalid entry point?
```

That question usually points to the next fix.

```sh
make
```

Print the command surface:

```sh

```

Old raw examples still work:

```sh

```

ELF examples still work:

```sh

./emulator run examples/v0_8/hello_elf.elf
```

Mach-O deterministic example fixtures are now generated from source, so you can try the loader without Xcode or Apple linker tools.

```sh

make examples/v1_1/hello.macho
make examples/v1_1/zero_fill.macho
```

Inspect one without running it:

```sh
./emulator info examples/v1_1/hello.macho
```

Run the hello example:

```sh
./emulator run examples/v1_1/hello.macho
```

The generated examples are deterministic. They are produced by `examples/v1_1/generate_macho_fixtures.py`, not by Xcode or Apple linker tools.

## 16. Common mistakes

Mistake:

```text
Assuming Mach-O support means normal macOS apps run.
```

Correction:

```text
v1.1 supports only a tiny static teaching profile.
```

Mistake:

```text
Confusing LC_MAIN entryoff with a direct pc value.
```

Correction:

```text
v1.1 resolves entryoff through a mapped segment's file range.
```

Mistake:

```text
Thinking segment permissions are enforced.
```

Correction:

```text
The loader records permission bits, but the flat memory model does not enforce page permissions yet.
```

Mistake:

```text
Expecting Apple syscalls to work.
```

Correction:

```text
Only the toy syscall ABI from v0.7 is supported.
```

## One-sentence summary

v1.1 teaches that Mach-O loading is another form of careful executable metadata validation: read the header, walk load commands, map simple segments, resolve the entry point, and reject anything that needs a real Apple runtime.
