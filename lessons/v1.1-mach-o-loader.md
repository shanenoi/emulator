# v1.1 Lesson — Mach-O Loader

This guide is beginner-first. It continues the learning chain from `lessons/v0.1-instruction-sandbox.md` through `lessons/v1.0-stable-learning-emulator.md`.

This lesson assumes you understand the earlier emulator ideas:

```text
pc fetches instructions
registers hold CPU state
memory stores bytes
branches change pc
functions use BL, RET, and x30
SVC #0 calls the toy syscall handler
ELF files use metadata to tell the loader where bytes go
v1.0 makes the supported path stable and tested
```

But this lesson does **not** assume you already know Mach-O.

Mach-O is treated as a new file format.

The short version:

```text
v1.1 teaches the loader to recognize, inspect, validate, and run a tiny
little-endian arm64 Mach-O MH_EXECUTE teaching profile.

It does not make the emulator a macOS or iOS emulator.
```

That boundary is the most important idea in this lesson.

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
v1.0: the whole learning path becomes a stable release surface
v1.1: the loader can load and inspect a tiny Mach-O executable profile
```

v1.1 is closest in spirit to v0.8.

v0.8 asked:

```text
How can an ELF file describe where its bytes should go in memory?
```

v1.1 asks:

```text
How can a Mach-O file describe where its bytes should go in memory?
```

The CPU loop is still the same:

```text
fetch instruction at pc
decode instruction
execute instruction
update pc/registers/memory
stop on HLT, fake exit, or error
```

Mach-O support changes the **loader**, not the basic CPU idea.

---

## 2. What Mach-O is

Mach-O is an executable/object file format used by Apple platforms.

That sentence can sound intimidating, so connect it to something you already know from v0.8.

An executable file format answers loader questions:

```text
What kind of file is this?
Which CPU architecture is it for?
Is it an executable, library, object file, or something else?
Which file bytes should be copied into memory?
Which guest addresses should receive those bytes?
Where should pc start?
Does the file require runtime features this emulator does not provide?
```

ELF answers those questions with:

```text
ELF header
program headers
PT_LOAD entries
e_entry
```

Mach-O answers similar questions with:

```text
Mach-O header
load commands
LC_SEGMENT_64 commands
LC_MAIN
```

The names are different, but the loader job is familiar.

---

## 3. What v1.1 supports

v1.1 supports a deliberately small Mach-O profile:

```text
thin file
64-bit Mach-O
little-endian
arm64 CPU type
MH_EXECUTE file type
LC_SEGMENT_64 for mapped segments
LC_MAIN for the entry point
optional LC_SYMTAB for info output
optional LC_DYSYMTAB if it does not advertise unsupported relocations
```

The phrase `MH_EXECUTE` means:

```text
This Mach-O file is an executable image.
```

The phrase `little-endian` means multi-byte integer fields are stored with the least significant byte first. This matches the format profile the loader is written to parse.

The phrase `arm64` means the file is meant for AArch64 instructions, the instruction family this teaching emulator supports.

v1.1 does **not** support normal dynamically linked macOS/iOS programs.

It rejects features such as:

```text
fat/universal Mach-O containers
32-bit Mach-O
big-endian Mach-O
non-arm64 Mach-O
non-MH_EXECUTE Mach-O files
dyld loading
shared libraries
rebasing
binding
relocations
Apple/Darwin syscalls
Objective-C or Swift runtime behavior
```

This is not a weakness in the lesson. It is the lesson boundary.

The emulator is learning one loader shape, not becoming macOS.

## 4. Raw `.bin`, ELF64, and Mach-O side by side

The easiest way to understand v1.1 is to compare all three supported program-loading paths.

### Raw `.bin`

A raw `.bin` file has no metadata.

The emulator uses a fixed rule:

```text
copy whole file to 0x1000
set pc = 0x1000
set sp = 0x100000
```

The loader invents the loading policy because the file does not contain one.

### ELF64

An ELF64 executable has metadata.

The loader follows `PT_LOAD` program headers:

```text
copy file bytes from p_offset to p_vaddr
zero-fill if p_memsz > p_filesz
set pc = e_entry
set sp = 0x100000
```

The file tells the loader how to map segments.

### Mach-O

A Mach-O executable also has metadata, but it uses different names.

The loader follows `LC_SEGMENT_64` load commands and `LC_MAIN`:

```text
copy file bytes from fileoff to vmaddr
zero-fill if vmsize > filesize
resolve LC_MAIN entryoff into a guest address
set pc to that resolved entry
set sp = 0x100000
```

The big comparison:

```text
raw .bin: file has no map, so emulator chooses 0x1000
ELF64:    PT_LOAD says where bytes go, e_entry says where pc starts
Mach-O:   LC_SEGMENT_64 says where bytes go, LC_MAIN helps pc start
```

Keep this table in mind:

| Idea | Raw `.bin` | ELF64 | Mach-O |
|---|---|---|---|
| file metadata? | no | yes | yes |
| segment description | fixed whole file | `PT_LOAD` | `LC_SEGMENT_64` |
| initialized file size | whole file | `p_filesz` | `filesize` |
| runtime memory size | whole file | `p_memsz` | `vmsize` |
| entry source | fixed `0x1000` | `e_entry` | `LC_MAIN entryoff` resolved through segment |

## 5. The loader decision tree

When you run:

```sh
./emulator run <program>
```

the loader needs to decide what kind of file `<program>` is.

Conceptually:

```text
read first bytes
    |
    +-- ELF magic? ------------------> parse as ELF64
    |
    +-- Mach-O magic? ---------------> parse as Mach-O
    |
    +-- unsupported Mach-O magic? ---> reject clearly
    |
    +-- otherwise -------------------> treat as raw .bin
```

This matters because a random raw `.bin` program should still work.

But a file that clearly begins with unsupported Mach-O magic should not silently run as raw bytes. If a file says, “I am a Mach-O file,” and the emulator cannot support that variant, v1.1 rejects it clearly.

Examples:

```text
little-endian 64-bit arm64 MH_EXECUTE -> try Mach-O loader
big-endian Mach-O                   -> reject as unsupported Mach-O variant
32-bit Mach-O                       -> reject as unsupported Mach-O variant
fat/universal Mach-O                -> reject as unsupported container
ordinary raw bytes                  -> raw .bin path
```

That is safer than guessing.

## 6. The Mach-O header

A v1.1 Mach-O file starts with a `mach_header_64`.

You do not need to memorize every field. The beginner mental model is:

```text
The header tells the loader what kind of Mach-O file this is
and how many load commands follow it.
```

Important fields include:

```text
magic      -> identifies 64-bit little-endian Mach-O
cputype    -> should be arm64
filetype   -> should be MH_EXECUTE
ncmds      -> number of load commands
sizeofcmds -> total byte size of the load-command table
```

The loader validates these before trusting the rest of the file.

Examples of bad files:

```text
file is too short to contain the full header
magic says 32-bit Mach-O
cputype says a different architecture
filetype says this is not an executable
ncmds says commands exist but sizeofcmds is too small
sizeofcmds points past the end of the file
```

The rule is:

```text
Do not walk metadata until the range containing that metadata is proven valid.
```

That rule was true for ELF too.

Mach-O just gives us a new place to practice it.

---

## 7. Load commands

After the header comes a table of load commands.

Think of load commands as instructions for the loader, not instructions for the CPU.

Every load command starts with two important fields:

```text
cmd      -> what kind of command this is
cmdsize  -> how many bytes this command occupies
```

The loader walks the table like this:

```text
current = start of load commands

for each command:
    read cmd and cmdsize
    validate cmdsize
    handle the command if v1.1 supports it
    current = current + cmdsize

after the loop:
    current should equal the end of the command table
```

The `cmdsize` field is dangerous if unchecked.

Bad values include:

```text
cmdsize = 0
cmdsize smaller than the basic load-command header
cmdsize smaller than the required structure for that command
cmdsize not aligned to 8 bytes
cmdsize points past sizeofcmds
cmdsize points past the end of the file
```

v1.1 rejects these.

This protects the host emulator from reading beyond the file buffer.

## 8. `LC_SEGMENT_64`: mapping file bytes into memory

`LC_SEGMENT_64` is the main command that maps bytes into guest memory.

If ELF `PT_LOAD` means:

```text
load this ELF segment
```

then Mach-O `LC_SEGMENT_64` means roughly:

```text
load this Mach-O segment
```

Important fields:

```text
segname   -> human-readable segment name, such as __TEXT or __DATA
vmaddr    -> guest memory address where the segment begins
vmsize    -> how many guest memory bytes the segment occupies
fileoff   -> file offset where initialized bytes begin
filesize  -> how many initialized bytes to copy from the file
initprot  -> requested memory permissions, recorded but not enforced yet
nsects    -> number of section records stored inside this command
```

The mapping rule is:

```text
copy file[fileoff : fileoff + filesize]
to   memory[vmaddr : vmaddr + filesize]

if vmsize > filesize:
    fill memory[vmaddr + filesize : vmaddr + vmsize] with zero
```

Visually:

```text
Mach-O file:

fileoff
0x180:  01 02 03 04

LC_SEGMENT_64 says:

vmaddr   = 0x4000
filesize = 4
vmsize   = 12

Guest memory after loading:

0x4000:  01 02 03 04 00 00 00 00 00 00 00 00
         ^^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^
         copied      zero-filled
```

This is the same zero-fill idea from ELF:

```text
ELF:    p_filesz < p_memsz
Mach-O: filesize < vmsize
```

The names changed. The idea did not.

The loader must reject unsafe segment descriptions:

```text
filesize > vmsize
fileoff + filesize goes past the end of the file
vmaddr + vmsize goes past emulator memory
segment overlaps a previously loaded segment
nsects claims section records that do not fit inside cmdsize
```

v1.1 records segment names and section counts for inspection, but it does not enforce memory permissions yet.

That means a segment may say it is read-only or executable, but the current flat memory model does not act like a real MMU.

## 9. `LC_MAIN`: choosing the starting `pc`

ELF has a direct entry address:

```text
e_entry = guest address where pc starts
```

Mach-O's `LC_MAIN` works differently.

It gives an `entryoff`:

```text
entryoff = file offset of the entry point
```

That is not directly a guest address.

The loader must translate it through the mapped segment file ranges.

The algorithm is:

```text
for each loaded segment:
    does entryoff fall inside [segment.fileoff, segment.fileoff + segment.filesize)?
        if yes:
            entry = segment.vmaddr + (entryoff - segment.fileoff)
```

Example:

```text
LC_SEGMENT_64:
    fileoff = 0x180
    vmaddr  = 0x4000

LC_MAIN:
    entryoff = 0x188

entry = 0x4000 + (0x188 - 0x180)
entry = 0x4008
```

Then the loader validates:

```text
entry was found in a loaded segment
entry is inside initialized file-backed bytes
entry is 4-byte aligned
LC_MAIN appears exactly once
```

Why 4-byte aligned?

AArch64 instructions are 4 bytes wide. The emulator fetches a 32-bit instruction at `pc`. Starting at an unaligned address would not match the instruction model from v0.1.

So the loader sets:

```text
pc = resolved entry address
```

only after the translation and validation succeed.

## 10. Unsupported dynamic features

Most real-world Mach-O executables are not tiny static teaching files.

They often expect Apple's runtime loader and operating-system environment.

Commands such as these are important signs:

```text
LC_LOAD_DYLINKER
LC_LOAD_DYLIB
LC_DYLD_INFO
LC_DYLD_INFO_ONLY
```

They mean the file may need:

```text
dyld
shared libraries
rebasing
binding
symbol resolution
relocations
Darwin process setup
real Apple syscalls
```

v1.1 does not provide those.

Therefore it rejects them during loading.

This is better than ignoring them.

If the loader silently ignored `dyld` or dynamic-library metadata, the CPU might later fail with a confusing unsupported instruction, bad memory access, or missing syscall.

The clear rule is:

```text
If the file asks for runtime services outside v1.1, reject it before execution.
```

That keeps the error in the correct layer.

## 11. `LC_SYMTAB` and `LC_DYSYMTAB`: inspection, not linking

v1.1 also understands a small amount of symbol-table metadata for inspection.

This does not mean the emulator links programs.

It means `./emulator info` can print more useful information when a deterministic fixture contains symbols.

`LC_SYMTAB` describes:

```text
where symbol records live in the file
how many symbol records exist
where the symbol string table lives
how large the string table is
```

v1.1 validates those ranges and can show bounded symbol names/addresses.

That is why the docs mention `symbol names/addresses`.

The purpose is learner visibility:

```text
Can I inspect this file and see what the loader learned?
```

not dynamic linking.

`LC_DYSYMTAB` is treated carefully. If it advertises indirect symbols without unsupported relocation work, the loader can report counts. If it advertises relocations the emulator cannot perform, v1.1 rejects the file.

The boundary is:

```text
small metadata inspection: yes
real dynamic linking/relocation: no
```

## 12. The `info` command

v1.1 adds an inspection command:

```sh
./emulator info <program>
```

`info` loads and validates the program, then prints loader metadata without executing guest instructions.

That makes it different from `run`:

```text
run  -> load, then execute from pc
info -> load, then describe what was loaded
```

For Mach-O files, `info` can show:

```text
format
entry point
stack pointer
mapped segments
segment names
file offsets
file sizes
memory sizes
load-command count
symbol count
indirect-symbol count
bounded symbol names/addresses when LC_SYMTAB is present
```

This is useful before debugging execution.

For example, if a Mach-O program starts at the wrong address, use `info` first:

```sh
./emulator info examples/v1_1/hello.macho
```

Ask:

```text
What format was detected?
What entry address was chosen?
Which segments were mapped?
Is the entry inside one of those segments?
Does a zero-fill segment show mem_size larger than file_size?
```

Only after the loader view makes sense should you move to:

```sh
./emulator trace <program>
./emulator debug <program>
```

## 13. CLI commands at v1.1

The public command surface is now:

```sh
./emulator run <program>
./emulator trace <program>
./emulator regs <program>
./emulator dump <program> <address> <length>
./emulator info <program>
./emulator debug <program>
./emulator help
```

At v1.1, `<program>` means:

```text
raw .bin
supported ELF64 ET_EXEC
supported little-endian arm64 Mach-O MH_EXECUTE
```

The execution modes still mean what they meant before:

```text
run   -> execute until halt/error/fake exit
trace -> print each instruction as it executes
regs  -> run and print final registers
dump  -> load/run enough to inspect memory as designed by the command
debug -> interactive stepping and breakpoints
info  -> inspect loader metadata without executing instructions
```

The new part is that these modes can use the Mach-O loader when the file is in the supported profile.

## 14. Deterministic Mach-O examples

v1.1 does not require Xcode or Apple's linker to learn the Mach-O loader.

The project includes a deterministic generator:

```text
examples/v1_1/generate_macho_fixtures.py
```

It can generate small `.macho` files with known behavior:

```text
minimal_exit.macho
    exits through the toy syscall ABI with a known status

hello.macho
    writes a small message through fake write

zero_fill.macho
    demonstrates memory size larger than file size
```

You can build them with:

```sh
make examples/v1_1/minimal_exit.macho
make examples/v1_1/hello.macho
make examples/v1_1/zero_fill.macho
```

Then inspect one:

```sh
./emulator info examples/v1_1/hello.macho
```

And run one:

```sh
./emulator run examples/v1_1/hello.macho
```

The generated `.macho` files are build artifacts. They are ignored by git and removed by:

```sh
make clean
```

That follows the v1.0 release discipline.

## 16. How to read v1.1 errors

When Mach-O loading fails, classify the error before trying to fix it.

### Format detection error

Example:

```text
unsupported 32-bit Mach-O
```

Meaning:

```text
The file is Mach-O-like, but not the 64-bit profile v1.1 supports.
```

### Header validation error

Example:

```text
unsupported CPU type; expected arm64
```

Meaning:

```text
The file is for a different architecture.
```

### Load-command error

Example:

```text
load command size is invalid
```

Meaning:

```text
The metadata table cannot be walked safely.
```

### Segment mapping error

Example:

```text
segment extends beyond guest memory
```

Meaning:

```text
The file asks the emulator to map bytes outside its 1 MiB memory.
```

### Entry-point error

Example:

```text
LC_MAIN entry is not mapped
```

Meaning:

```text
The entryoff could not be translated through a loaded segment.
```

### Unsupported runtime feature

Example:

```text
LC_LOAD_DYLIB/shared libraries are unsupported in v1.1
```

Meaning:

```text
The file wants dynamic linking, which is outside this teaching profile.
```

The important habit is:

```text
Loader errors are about file structure and mapping.
CPU errors are about instruction execution.
Syscall errors are about the toy runtime ABI.
```

Do not mix those layers unless you have evidence.

## 17. What did not change

v1.1 does not change the basic teaching CPU.

It does not add:

```text
a complete ARM64 instruction set
Apple syscall compatibility
dyld
shared-library loading
Objective-C or Swift runtime support
memory protection enforcement
page tables
threads
signals
normal macOS process startup
```

The fake syscall ABI is still the v0.7 ABI:

```text
x8 = 64  -> fake write
x8 = 93  -> fake exit
svc #0   -> ask the emulator to handle it
```

A Mach-O fixture can print only because it uses the emulator's toy syscall ABI, not because Darwin syscalls work.

That is a major scope distinction.

## 19. Common mistakes

### Mistake 1: thinking Mach-O support means macOS app support

v1.1 supports a tiny static teaching profile.

It does not run normal dynamically linked macOS apps.

### Mistake 2: treating `LC_MAIN entryoff` as a direct `pc`

`entryoff` is a file offset.

The loader must translate it through a segment:

```text
entry = segment.vmaddr + (entryoff - segment.fileoff)
```

### Mistake 3: forgetting zero-fill

If `vmsize > filesize`, the extra memory bytes should be zero.

Those bytes are not copied from the file.

They are created by the loader.

### Mistake 4: assuming segment permissions are enforced

v1.1 records permissions for inspection and future use.

The current flat memory model does not enforce page protections.

### Mistake 5: expecting Apple syscalls

The executable container is Mach-O, but the runtime ABI is still the toy syscall ABI.

### Mistake 6: ignoring unsupported commands

Unsupported dynamic commands should be rejected early.

Silently ignoring them would create harder-to-debug execution failures.

### Mistake 7: thinking symbols mean linking

`info` may show symbol names/addresses.

That is inspection only. It is not dynamic symbol resolution.
