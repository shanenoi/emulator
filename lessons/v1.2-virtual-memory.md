# v1.2 Lesson — Virtual Memory and Page Permissions

This guide is beginner-first. It continues the learning chain from
`lessons/v0.1-instruction-sandbox.md` through
`lessons/v1.1-mach-o-loader.md`.

The short version:

```text
v1.2 starts teaching memory as mapped pages with permissions.

Before v1.2, memory was mostly one big byte array.
After v1.2, loaded programs get a visible map:

code/data pages -> mapped with permissions
stack pages     -> mapped read/write
guard page      -> shown as --- below the stack
other pages     -> unmapped
```

v1.2 does **not** turn the emulator into a real operating-system virtual-memory
subsystem. It is a small teaching model that helps you answer:

```text
Is this address mapped?
Can this instruction read here?
Can this instruction write here?
Can the CPU fetch an instruction here?
Where is the stack allowed to grow?
Did the fault happen because of bounds, unmapped memory, read permission,
write permission, or execute permission?
```

---

## 1. The learning ladder so far

The project has been building one layer at a time:

```text
v0.1: fetch, decode, execute raw instructions
v0.2: pc can jump for branches and loops
v0.3: instructions can read/write memory and use the stack
v0.4: functions can call and return through x30
v0.5: debugger can pause, inspect, step, and continue
v0.6: trace output can show readable decoded instructions
v0.7: guest code can print and exit through SVC #0 toy syscalls
v0.8: the loader can load simple ELF64 executables
v0.9: tiny C programs can compile into those ELF64 executables
v1.0: the whole learning path becomes a stable release surface
v1.1: the loader can load and inspect a tiny Mach-O executable profile
v1.2: loaded programs now get mapped pages and memory permissions
```

v1.2 is closest in spirit to v0.3, v0.8, and v1.1 at the same time:

```text
v0.3 taught: instructions access memory addresses.
v0.8 taught: ELF segments describe where file bytes go in memory.
v1.1 taught: Mach-O segments describe where file bytes go in memory.
v1.2 teaches: not every address is valid, and not every mapped address allows every action.
```

The CPU loop is still the same:

```text
fetch instruction at pc
decode instruction
execute instruction
update pc/registers/memory
stop on HLT, fake exit, or error
```

The new question is what happens **before** a memory access succeeds.

## 2. The old mental model: one big byte array

Earlier lessons used a simple mental model:

```text
Memory is a 1 MiB array of bytes.

address 0x00000 -> first byte
address 0x01000 -> raw program load address
address 0xfffff -> last byte
```

That model was perfect for the early emulator because it let you focus on:

```text
pc
registers
instruction decoding
load/store instructions
stack movement
function calls
syscalls
ELF/Mach-O file parsing
```

But it hides an important systems-programming idea:

```text
A real program is not allowed to use every address for every purpose.
```

For example, these should feel different:

```text
read a string from a data page        -> probably okay
write into a read-only code page      -> suspicious
execute bytes from a data-only page   -> suspicious
read an address no loader ever mapped -> definitely wrong
push below the stack                  -> stack overflow/underflow style bug
```

v1.2 introduces enough structure to explain those differences.

---

## 3. The new mental model: pages

A page is a fixed-size chunk of memory.

In this project, the teaching page size is:

```text
4096 bytes
```

That means addresses are grouped like this:

```text
page 0: 0x00000 - 0x00fff
page 1: 0x01000 - 0x01fff
page 2: 0x02000 - 0x02fff
...
```

The exact byte address still matters. `STR X0, [SP, #-8]!` still writes exactly
8 bytes at the calculated address.

The page answers a bigger question first:

```text
Does this region of memory exist for the guest program?
```

Think of memory like a building:

```text
byte address = exact room number
page         = floor of the building
mapping      = permission to enter that floor
permission   = what you are allowed to do there
```

---

## 4. What a mapping is

A mapping says:

```text
from this address
for this many bytes/pages
allow these actions
with this readable name
```

In code, the important pieces are:

```text
start address
size
permissions
name
```

Example mapping:

```text
r-x 0x0000000000001000-0x0000000000002000 name=raw:program
```

Read it like this:

```text
The page from 0x1000 up to, but not including, 0x2000 is mapped.
The guest may read it.
The guest may execute instructions from it.
The guest may not write to it.
The debugger/info view calls it raw:program.
```

The end address is exclusive:

```text
0x1000 is inside
0x1fff is inside
0x2000 is not inside
```

---

## 5. Permission labels

v1.2 uses the same compact labels you often see in systems tools:

```text
r--  readable only
rw-  readable and writable
r-x  readable and executable
rwx  readable, writable, and executable
---  no permissions / effectively unmapped
```

Each letter answers one question:

```text
r: may an instruction or helper read bytes here?
w: may an instruction write bytes here?
x: may pc fetch an instruction here?
```

Examples:

```text
instruction fetch from r-x page -> okay
instruction fetch from rw- page -> execute permission fault
STR to rw- page                 -> okay
STR to r-x page                 -> write permission fault
LDR from unmapped page          -> unmapped access fault
```

---

## 6. Why instruction fetch is special

In v0.1, fetching an instruction meant:

```text
read 4 bytes from memory at pc
```

In v1.2, fetching means:

```text
1. pc must be 4-byte aligned.
2. the 4 bytes must be inside guest memory.
3. the page must be mapped.
4. the page must allow execute.
5. then the emulator reads the opcode bytes.
```

This distinction matters because reading bytes and executing bytes are not the
same idea.

A string in memory is readable:

```text
"hello"
```

But it should not automatically be executable code.

A code page is executable:

```text
MOVZ X0, #1
HLT #0
```

But it should not automatically be writable by guest code.

---

## 7. Data reads and writes

Load instructions such as `LDR` need read permission.

Store instructions such as `STR` need write permission.

```text
LDR X0, [X1]
    -> calculate address from X1
    -> check the touched pages are readable
    -> read bytes
    -> place value in X0

STR X0, [X1]
    -> calculate address from X1
    -> check the touched pages are writable
    -> write bytes from X0
```

The permission check happens before bytes are returned or changed.

That makes faults deterministic:

```text
bad access -> report error -> do not partially modify memory
```

---

## 8. Multi-byte accesses cross pages

A memory access can touch more than one page.

Example:

```text
read 8 bytes starting at 0x1ffc
```

The bytes are:

```text
0x1ffc
0x1ffd
0x1ffe
0x1fff
0x2000
0x2001
0x2002
0x2003
```

That access touches two pages:

```text
0x1000-0x1fff
0x2000-0x2fff
```

v1.2 checks the whole range, not just the first byte.

So this should fail:

```text
first page:  readable
second page: unmapped
read 8 bytes crossing into second page -> unmapped access fault
```

This is one of the most important edge cases in v1.2.

---

## 9. The stack and the guard page

Earlier lessons used this simple rule:

```text
sp starts at the top of memory: 0x100000
```

That is still true.

But now the stack has a mapped area below it:

```text
0x100000  <- initial sp, top of memory
   |
   | mapped stack pages, rw-
   v
0x0f0000  <- bottom of 64 KiB stack mapping
0x0ef000  <- guard page below stack, unmapped
```

The stack grows downward. A push usually subtracts from `sp` and then writes.

A normal push stays inside the mapped stack:

```text
sp = 0x100000
push 8 bytes
new sp = 0x0ffff8
write at 0x0ffff8 -> inside stack mapping -> okay
```

A broken program that pushes too far reaches the guard page:

```text
write below 0x0f0000 -> unmapped access fault
```

The guard page is useful because it turns a bad stack bug into a clear teaching
error instead of silently corrupting unrelated memory.

## 10. How loaders create mappings

Raw `.bin`, ELF, and Mach-O all end with the same goal:

```text
put bytes in guest memory
set pc
set sp
```

v1.2 adds one more job:

```text
create memory mappings before the program runs
```

### Raw `.bin`

Raw files still load at `0x1000`.

The loader maps the raw program as readable/executable. Raw mappings are rounded
up to a 4-byte instruction boundary so older partial-instruction examples still
produce decode errors instead of fetch-permission errors:

```text
raw file bytes -> 0x1000
mapping        -> r-x raw:program
pc             -> 0x1000
sp             -> 0x100000
stack mapping  -> rw- stack
```

Raw files do not contain permission metadata, so the emulator chooses a simple
teaching policy.

Two details are worth knowing:

```text
empty raw file       -> rejected before mapping
zero-length access   -> successful no-op after ordinary bounds validation
```

The first rule keeps the loader from creating a meaningless program. The second
rule lets helper code ask “is zero bytes readable?” without turning that into a
fault.

### ELF64

ELF already has segment flags:

```text
PF_R -> readable
PF_W -> writable
PF_X -> executable
```

v1.2 records those flags as page permissions.

Typical ELF layout:

```text
.text segment -> r-x
.data segment -> rw-
.bss memory   -> usually part of rw- data memory
stack         -> rw-
```

### Mach-O

Mach-O `LC_SEGMENT_64` has protection metadata too.

Typical Mach-O teaching layout:

```text
__TEXT -> r-x
__DATA -> rw-
stack  -> rw-
```

The CPU still does not know whether bytes came from raw, ELF, or Mach-O. After
loading, it only sees memory, registers, `pc`, and `sp`.

## 11. Zero-fill with pages

Zero-fill was introduced in the ELF lesson, but it becomes more important in
v1.2.

A file segment can say:

```text
file_size = 16 bytes
mem_size  = 64 bytes
```

That means:

```text
copy 16 bytes from the file
then create 48 more bytes in memory and set them to zero
```

A visual version:

```text
file:
[ real bytes: 16 ]

memory:
[ real bytes: 16 ][ zeros: 48 ]
```

The zero bytes are not stored in the file, but the program is allowed to use them
because the loader maps memory for the full `mem_size`.

In v1.2, zero-fill must still obey mappings:

```text
segment says mem_size covers the zero area
loader maps that area
loader writes zeros during setup
program later reads/writes according to segment permissions
```

This is how simple global storage works in the tiny C examples.

## 12. What `info` shows now

The `info` command was introduced in v1.1 for loader inspection.

v1.2 makes it more useful:

```sh
./emulator info examples/v1_1/hello.macho
```

You should see the normal loader metadata plus mappings:

```text
format: macho64
entry: 0x0000000000001000
stack_pointer: 0x0000000000100000
segments: 1
...
mappings: 2
  [0] r-x 0x0000000000001000-0x000000000000102c size=0x2c name=macho:__TEXT
  [1] rw- 0x00000000000f0000-0x0000000000100000 size=0x10000 name=stack
      guard --- 0x00000000000ef000-0x00000000000f0000 size=0x1000 name=stack-guard
```

This lets you ask, before running anything:

```text
Where is code?
Where is stack?
Is this page executable?
Is this page writable?
Where is the stack guard?
```

The exact end address of the program mapping depends on the fixture. Earlier
versions rounded mentally to pages; v1.2 records mapping ranges separately from
the broad page-coverage metadata so adjacent ELF/Mach-O lesson fixtures still
work without accidentally making nearby data writable/executable.

## 13. Debugger mapping commands

The debugger gets two memory-map inspection commands:

```text
maps
map <address>
```

Use `maps` to list mappings:

```sh
printf 'maps\nquit\n' | ./emulator debug examples/v1_1/hello.macho
```

Use `map <address>` to ask about one address:

```sh
printf 'map 0x1000\nmap 0xe0000\nquit\n' | ./emulator debug examples/v1_1/hello.macho
```

Possible answers:

```text
address 0x0000000000001000 is in r-x mapping ... name=macho:__TEXT
address 0x00000000000ef000 is in --- guard ... name=stack-guard
address 0x00000000000e0000 is unmapped
```

## 14. Mapping overlap policy

v1.2 rejects true overlapping mappings.

This is bad:

```text
mapping A: 0x1000-0x1100 r-x
mapping B: 0x1080-0x1180 rw-
```

If that were allowed, one piece of code might think the range is code while
another thinks it is data. That makes permission faults confusing.

This is allowed:

```text
mapping A: 0x1000-0x1100 r-x
mapping B: 0x1100-0x1200 rw-
```

The second mapping begins exactly where the first one ends. That is adjacency,
not overlap. Keeping adjacency valid preserves earlier ELF/Mach-O examples that
place tiny segments next to one another.

The permission check follows the mapping records, not just a broad page-level OR.
That means a writable mapping next to an executable mapping does not silently make
the executable bytes writable.

## 15. Compatibility rule for entry segments

ELF and Mach-O both carry permission metadata.

In a stricter loader, a program whose entry point sits in a non-executable segment
would load and then fail with an execute-permission fault, or be rejected before
execution.

This teaching emulator keeps one compatibility rule from earlier lessons:

```text
if the entry point is inside a loaded segment,
the loader makes that mapping executable
```

Why? Some older deterministic fixtures were written before v1.2 cared about
execute permissions. This rule lets old lessons continue to teach their original
topic while v1.2 introduces permissions gradually.

The important scope boundary is:

```text
v1.2 teaches permission checking,
but it does not try to become a strict production executable verifier.
```

Future tests can add stricter malformed fixtures if the project decides to remove
this compatibility behavior.

This is the debugger version of asking:

```text
What does the emulator believe about this address?
```

## 16. Out of scope: what is intentionally not implemented

v1.2 is not a real ARMv8 MMU.

It does not implement:

```text
real ARM page tables
translation registers
TLBs
kernel/user privilege levels
process isolation
signals
SIGSEGV delivery
demand paging
copy-on-write
mmap/brk heap management
shared memory
thread stacks
ASLR
PIE relocation
```

That is intentional.

The teaching goal is smaller:

```text
mapped or unmapped?
readable?
writable?
executable?
which mapping contains this address?
```

## 17. Common mistakes

### Mistake 1: Thinking `sp = 0x100000` means that address is writable

`sp` points just above the current stack contents.

A push subtracts first, then writes:

```text
sp starts at 0x100000
STP ..., [SP, #-16]!
new sp is 0x0ffff0
write happens at 0x0ffff0
```

So the initial `sp` itself is the top boundary, not a byte you write directly.

### Mistake 2: Thinking file size and memory size are always equal

They are not.

Zero-fill means memory can be larger than the bytes stored in the file.

### Mistake 3: Checking only the first byte of an access

A 64-bit access touches 8 bytes.

If those 8 bytes cross into an unmapped page, the whole access must fail.

### Mistake 4: Thinking readable means executable

A data string can be readable without being executable.

A code page can be executable without being writable.

Those are different permissions.

### Mistake 5: Treating this as a production VM

This is a learning model.

It is supposed to be simple enough to inspect and debug.
